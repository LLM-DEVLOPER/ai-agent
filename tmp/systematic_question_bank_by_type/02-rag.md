# RAG 全链路

重要性：

- 高频程度：最高
- 对应原始题库：[02-rag-pipeline.md](/Users/nako/WebstormProjects/github/thefoxfairy/interview/interview/tmp/raw_question_source_by_type/02-rag-pipeline.md)

## 这一类在考什么

这类问题本质上在考你能不能把 RAG 当成一个完整工程链路来讲清，而不是只会说“检索增强生成”。

## 这一类的核心主线

标准回答顺序建议固定为：

1. `RAG 想解决什么问题`
2. `完整链路有哪些模块`
3. `链路里每一层解决什么问题`
4. `出了问题怎么排障`
5. `RAG 和微调如何选`

## 必会主问题

1. RAG 解决的核心问题是什么？
2. 一个完整 RAG 链路包含哪些模块？
3. query rewrite、query expansion、router、rerank 各解决什么问题？
4. RAG 和微调怎么选？
5. 检索到了相关内容，但回答还是错，问题通常在哪？
6. 召回率低时优先排查哪些环节？
7. 什么时候要从单路检索升级到混合检索？
8. RAG 怎么处理知识时效性和版本漂移？
9. RAG 的 bad case 怎么分类？
10. 企业文档、社区内容、结构化数据接入方式有什么不同？

## 高频追问

1. 什么时候该改召回，什么时候该改重排，什么时候该改生成？
2. 用户问题很开放、检索目标不唯一时怎么办？
3. 检索链路里怎么加入权限、时间、用户画像过滤？
4. GraphRAG 适合什么场景？

## 标准回答抓手

- 永远按链路讲：`query -> 检索 -> 重排 -> 上下文构造 -> 生成 -> 约束`
- 排障时按模块倒查，不要一句“调 Prompt”
- 一定要说“相关内容到了但回答错”通常不是单一模块问题

## 常见失分点

- 只讲召回，不讲重排和答案约束
- 不知道 RAG 和微调的边界
- 不会从现象反推故障模块

## 最后该怎么复习

你至少要能白板讲出一条完整 RAG 链路，并能把每个环节的故障讲清。
