# 评测、幻觉治理与 bad case

重要性：

- 高频程度：高
- 对应原始题库：[07-eval-hallucination.md](/Users/nako/WebstormProjects/github/thefoxfairy/interview/interview/tmp/raw_question_source_by_type/07-eval-hallucination.md)

## 这一类在考什么

面试官真正想看的是：你会不会把“不靠谱”拆成可验证指标，并建立闭环。

## 这一类的核心主线

1. `分层评测`
2. `bad case 分类`
3. `幻觉来源`
4. `修复策略`
5. `线上回归`

## 必会主问题

1. RAG / Agent 项目怎么做分层评测？
2. 检索效果、回答效果、工具调用效果怎么分别评估？
3. 幻觉的主要来源有哪些？
4. bad case 怎么分类？
5. 离线评测和在线评测怎么联动？
6. 业务说系统不靠谱时，怎么拆成指标？
7. 线上效果退化时怎么定位是模型、知识库还是工具问题？

## 高频追问

1. 评测集怎么构造？
2. 自动评测和人工评测怎么结合？
3. 幻觉治理是加检索、加约束，还是换模型？
4. 评测指标怎么映射到业务指标？

## 标准回答抓手

- 评测分层：`召回层 -> 生成层 -> 工具层 -> 业务层`
- bad case 闭环：`采样 -> 标注 -> 分类 -> 修复 -> 回归`
- 业务问题一定要翻译成数据问题

## 常见失分点

- 只会说 Rouge、BLEU
- 不会定义 bad case 分类体系
- 线上问题只会说“模型不太行”
