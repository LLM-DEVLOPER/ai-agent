# 项目深挖

## 1. 项目介绍应该先讲什么

1. 项目介绍最先应该讲什么？
2. 项目介绍里最需要优先讲清哪些信息？
3. 为什么项目介绍不能一上来就堆技术名词？
4. 业务背景、核心问题、解决方案、结果一般应该按什么顺序展开？
5. 项目介绍的核心目标通常应该怎么定义？

## 2. 业务背景、项目来源与数据来源

1. 项目来源为什么经常会被追问？
2. 目标用户、核心问题、业务价值为什么一定要先讲清？
3. 数据集是公开数据、业务数据还是自建数据，为什么很重要？
4. 项目里的数据通常从哪里来？
5. 数据构建和数据治理为什么经常暴露项目是否真实做过？
6. 目标业务理解为什么也是项目深挖的一部分？

## 3. 个人职责与 ownership

1. 为什么项目里最需要先讲清自己的真实职责？
2. 为什么“我们团队做了什么”不是有效回答？
3. 为什么项目里的边界和 ownership 特别重要？
4. 多人合作项目里，个人贡献应该怎么讲清？
5. 哪些表述最容易暴露自己没有真正做过核心部分？

## 4. 架构、链路与模块拆解

1. 为什么项目里一定要讲清整体架构？
2. 为什么项目链路讲不清，通常说明没有真正做透？
3. 数据从哪里来、怎么流动、怎么落盘，为什么都是高频追问？
4. 为什么很多项目深挖最后会落到接口、状态和数据结构细节？
5. 项目全链路通常应该怎么从用户请求一路拆到结果返回？

## 5. 技术选型与方案取舍

1. 为什么项目介绍里必须讲清“为什么选这个方案”？
2. 为什么技术选型比“会不会用”更重要？
3. 为什么“因为业界都这么做”不是一个好答案？
4. 模型、框架、数据库、向量库、编排方式一般该怎么讲选型依据？
5. “为什么不用别的方案”这类问题通常该从哪些维度拆？

## 6. 效果、评测与结果证明

1. 为什么项目里一定要讲清“效果怎么量化”？
2. 为什么“效果还不错”基本等于没回答？
3. 指标通常应该怎么拆？
4. 业务指标和技术指标为什么都要准备？
5. 你怎么证明某次优化真的是有效的？
6. 如果效果没达到预期，通常应该怎么讲？

## 7. 问题、事故与 bad case

1. 为什么“最难的 bad case 是什么”是高价值问题？
2. 为什么真正踩过坑的人和只做过 demo 的人，一问就能分出来？
3. 线上出过什么事故，为什么是高价值问题？
4. 遇到问题时，定位过程和最终答案分别该讲什么？
5. 如果一个项目从来没有问题，为什么反而显得不可信？

## 8. 复盘、trade-off 与优化方向

1. “如果重做一遍你会改哪里”这类问题本质上在考什么？
2. 为什么这个问题本质上在考复盘能力？
3. 项目里做过哪些 trade-off，为什么特别值得讲？
4. 成本、时延、效果之间的取舍通常该怎么讲？
5. 为什么“我会把所有地方都优化”不是一个好答案？

## 9. 不同项目通常会从哪些方向被深挖

1. RAG 项目通常会被从哪些方向深挖？
2. Agent 项目通常会被从哪些方向深挖？
3. AI 应用项目通常会被从哪些方向深挖？
4. 后端工程项目通常会被从哪些方向深挖？
5. 为什么同一个项目会被技术面、业务面、交叉面从不同角度追问？

## 10. 项目深挖高频问题

1. 这个项目的业务目标是什么？
2. 你在项目里具体负责什么？
3. 为什么要这么选型？
4. 项目的整体架构是什么？
5. 项目的核心技术难点是什么？
6. 效果怎么量化？
7. 最难的 bad case 是什么？
8. 线上出过什么事故？
9. 如果重做一次你最想改哪里？
10. 这个项目里最能体现你能力的点是什么？
