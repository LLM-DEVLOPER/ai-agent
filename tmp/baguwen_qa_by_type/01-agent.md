# Agent

## 1. Agent 的定义与边界

### Q1. 什么是 Agent？
一句话回答：
Agent 本质上不是一个回答器，而是一个围绕目标持续做决策、执行动作并根据反馈推进任务的闭环系统。

回答主线：
1. 它的核心不是“会说话”，而是“能围绕目标持续推进”。
2. 它通常不是一次生成结束，而是多步决策闭环。
3. 它往往要和工具、状态、外部环境一起工作。

高频追问：
没有工具还算不算 Agent？

追问回答：
可以算，但只能算弱执行型 Agent。只要它还能基于状态做多步决策并持续推进任务，就具备 Agent 核心；只是没有工具时，它更像“会规划的智能体”，不像“能真正干活的智能体”。

### Q2. Agent 和普通 ChatBot、AI 助手、脚本增强型应用的本质区别是什么？
一句话回答：
本质区别不在有没有聊天窗口，而在系统是不是把关键决策留到了运行时。

回答主线：
1. ChatBot 主要回答问题，Agent 主要完成任务。
2. 脚本增强型应用靠预定义流程，Agent 靠运行时动态选动作。
3. AI 助手可能包含 Agent 能力，但并不等于所有 AI 助手都是 Agent。

高频追问：
怎么快速判断一个系统是不是“伪 Agent”？

追问回答：
看主链路是不是写死了。如果步骤、分支和顺序早就定好了，只是让模型补文案、做分类或挑一个分支，那它本质还是自动化系统，不是 Agent。

### Q3. 为什么 Agent 不是“能聊天的大模型”，而是“围绕目标持续决策和执行的系统”？
一句话回答：
因为聊天只是表现层，Agent 的本质是目标驱动的决策和执行。

回答主线：
1. 单轮生成只能给内容，不能保证任务推进。
2. Agent 要能根据当前状态决定下一步动作。
3. 它还要读取执行反馈，必要时修正路径。

高频追问：
为什么很多 demo 看起来很像 Agent？

追问回答：
因为 demo 往往把路径、工具、上下文都喂得很理想，模型只需要顺着演一次。真正上线后遇到长链路、异常工具和脏状态，才会暴露它到底是不是一个完整系统。

### Q4. Agent 的核心闭环为什么一定是感知、决策、执行、反馈，而不是单次生成？
一句话回答：
因为真实任务是动态过程，不是一次生成文本就能结束的静态问题。

回答主线：
1. 先感知当前环境、目标和状态。
2. 再基于这些输入决定动作。
3. 执行后要读取反馈，否则系统无法纠偏和收敛。

高频追问：
如果没有反馈环节，会退化成什么？

追问回答：
会退化成一次性生成或半静态编排。它可以看起来很聪明，但一旦外部环境变化或者中间执行失败，系统就没有自我修正能力。

### Q5. 怎么判断一个系统本质上是 Agent，而不是包装成 Agent 的 Workflow？
一句话回答：
看核心路径是不是运行时动态决定，而不是预先写死。

回答主线：
1. 如果主链路靠规则图谱和固定分支驱动，本质更接近 Workflow。
2. 如果系统会在约束下动态选动作、调整路径，更接近 Agent。
3. 实际落地里，很多系统是 Workflow 和 Agent 的混合体。

高频追问：
混合系统怎么归类？

追问回答：
看谁掌控关键决策点。如果核心路径还是固定编排，只在局部给了模型一点动态决策能力，我会把它归成“以 Workflow 为主、带 Agent 能力的系统”，而不是纯 Agent。

## 2. 为什么需要 Agent

### Q1. 为什么会出现 Agent 这种应用形态？
一句话回答：
因为业务真正需要的越来越不是“说得像”，而是“真的把事推进完”。

回答主线：
1. 很多任务需要多步决策，不是一次回复能解决。
2. 任务过程中还要接实时信息和外部系统。
3. 传统问答形态承载不了完整任务闭环。

高频追问：
最典型的任务类型是什么？

追问回答：
像 DeepResearch、Copilot、复杂客服处理、运营协作、数据分析助手这类任务都有一个共同点: 路径不是预先写死的，而且最终看的是任务有没有完成，不是回答写得漂不漂亮。

### Q2. Agent 想解决的核心问题到底是什么？
一句话回答：
它要解决的是如何把模型从内容生成器变成任务执行器。

回答主线：
1. 不是只会说，而是会查、会做、会判断。
2. 不是停留在对话层，而是能持续推进任务。
3. 不是单点能力，而是把推理、执行和反馈连成闭环。

高频追问：
为什么这个升级对企业应用特别重要？

追问回答：
因为企业场景真正值钱的不是“生成一段不错的话”，而是能把查询、审批、汇总、调用系统、形成结果这条链路打通，直接替代一部分流程劳动。

### Q3. 什么任务天然更适合 Agent，而不是普通问答？
一句话回答：
开放式、多步、依赖外部执行的任务天然更适合 Agent。

回答主线：
1. 路径不固定，事先难以写死。
2. 中间结果会显著影响后续动作。
3. 最终目标是完成任务，不只是生成内容。

高频追问：
举一个判断标准。

追问回答：
如果你很难提前把步骤写成稳定 if-else，而且执行过程中必须边看边调，那么这个任务通常就更偏 Agent，而不是普通问答。

### Q4. 什么任务其实不适合 Agent？
一句话回答：
流程固定、规则清晰、合规严格的任务通常不适合 Agent 主导。

回答主线：
1. 这类任务要的是确定性和可审计。
2. 用 Workflow 或传统系统通常更稳更便宜。
3. Agent 会带来额外的不确定性和治理成本。

高频追问：
是不是说明 Agent 没价值？

追问回答：
不是。它说明 Agent 不是默认最优解。好的工程判断不是到处上 Agent，而是只把它放在真正需要弹性决策的地方。

### Q5. 为什么不是所有 AI 应用都应该做成 Agent？
一句话回答：
因为 Agent 提供的是自由度，不是免费收益，自由度越高，治理成本越大。

回答主线：
1. 很多业务更在意稳定、时延和成本。
2. 只有高不确定性任务才值得交换更多自由度。
3. 所以先判断问题结构，再决定架构形态。

高频追问：
面试里怎么体现这种判断力？

追问回答：
不要一上来就说“我会用 Agent 解”。更好的答法是先把任务分成确定性部分和不确定性部分，再说明哪里适合 Workflow，哪里才值得给 Agent。

## 3. Agent 的能力边界与落地现实

### Q1. 当前 Agent 的能力边界主要体现在哪些方面？
一句话回答：
主要卡在长链路稳定性、复杂规划能力、工具协同可靠性和整体成本控制。

回答主线：
1. 短任务容易显得聪明，长任务容易漂。
2. 工具、状态、反馈一复杂，系统就会变脆。
3. 真正的瓶颈通常不是单点，而是整条链路的复合问题。

高频追问：
线上最先暴露的是哪层？

追问回答：
通常是工具调用和状态管理先出问题，因为它们最容易遇到真实世界里的脏数据、超时、权限和异常返回，而这些问题会立刻放大到轨迹层。

### Q2. 为什么很多 Agent 在复杂长任务、多约束任务上仍然不稳定？
一句话回答：
因为 Agent 不是在做一次判断，而是在做一串相互依赖的判断，前面一小步偏了，后面会被持续放大。

回答主线：
1. 前面一步偏一点，后面就会持续偏。
2. 约束之间可能冲突，模型很难稳定兼顾。
3. 工具异常、上下文污染和评估缺失会一起叠加。

高频追问：
这种问题更该改模型还是改系统？

追问回答：
对应用团队，我会优先改系统。先把动作空间、状态机、trace、guardrail 这些控住，再看是不是模型真的不够；不然直接换模型，通常只是把错误做得更贵。

### Q3. 当前 Agent 的理想预期和真实落地之间主要差距在哪里？
一句话回答：
理想预期是高自主通用执行，真实落地通常是受控、半自主、强约束的系统。

回答主线：
1. 生产系统不会让模型无限自由发挥。
2. 线上更看重可控、可追踪、可回放。
3. 所以真正落地的 Agent 往往像“带智能决策的工程系统”。

高频追问：
这是不是说明 Agent 还不成熟？

追问回答：
可以说还在快速迭代期，但不能说没价值。它已经能在很多高不确定性节点上创造收益，只是现阶段最稳的形态不是全自主，而是受控自主。

### Q4. 从模型侧和系统侧看，最限制 Agent 落地的短板分别是什么？
一句话回答：
模型侧主要卡在长程规划和约束遵循，系统侧主要卡在状态、工具治理和评测闭环。

回答主线：
1. 模型不稳定会导致动作选择波动。
2. 系统治理不够会让波动被放大。
3. 落地边界往往是两者共同决定的。

高频追问：
如果只能先投一个方向，你选哪边？

追问回答：
对大多数应用团队，我会先投系统侧，因为动作空间、状态设计、工具 schema、trace 和回放这些改动更确定、更立竿见影。

### Q5. 如果要让 Agent 真正变得更可靠，通常最值得优先改哪几层？
一句话回答：
我会优先改动作空间、状态管理、工具 schema、显式编排和评测闭环，而不是先追求“更自由的智能”。

回答主线：
1. 先减少自由度，避免系统乱跑。
2. 再提高执行层和状态层的确定性。
3. 最后用 trace、回放和评测形成迭代闭环。

高频追问：
为什么不是先换更强的模型？

追问回答：
因为系统自由度不受控时，换模型只是让错误变贵，未必让系统变稳。很多线上问题不是“模型不够强”，而是系统没把风险关住。

## 4. Agent、Workflow 与规则编排的关系

### Q1. Agent 和 Workflow 的区别是什么？
一句话回答：
Workflow 是预定义路径，Agent 是运行时路径选择。

回答主线：
1. Workflow 强在确定性和可审计。
2. Agent 强在应对不确定性。
3. 关键差异是控制权在代码还是在运行时决策。

高频追问：
一句话怎么区分这两个？

追问回答：
能不能提前把主链路写死，是我最常用的区分标准。能写死，偏 Workflow；写不死，偏 Agent。

### Q2. 什么场景更适合 Workflow，什么场景更适合 Agent？
一句话回答：
稳定流程适合 Workflow，高不确定性任务适合 Agent。

回答主线：
1. 规则清晰、分支有限、合规高，适合 Workflow。
2. 路径不确定、依赖中间反馈，适合 Agent。
3. 真实业务经常是两者混合。

高频追问：
客服、搜索、Copilot 怎么判断？

追问回答：
FAQ 和标准售后更偏 Workflow，复杂工单处理更偏 Agent；普通搜索更偏检索系统，DeepResearch 更偏 Agent；代码补全偏工具化，复杂编程协作更偏 Agent。

### Q3. 一个系统里为什么经常会同时存在 Workflow 和 Agent？
一句话回答：
因为真实落地最常见的最优解就是 Workflow 保底、Agent 提升弹性。

回答主线：
1. 固定主链路用 Workflow 托底。
2. 高不确定性节点交给 Agent 决策。
3. 这样能兼顾效果、稳定性和治理成本。

高频追问：
这是不是说明 Agent 只是辅助模块？

追问回答：
不一定。要看业务里高不确定性节点占比多大。有的系统里 Agent 只是一个环节，有的系统里 Agent 已经是核心控制层。

### Q4. Workflow 什么时候会自然演化成 Agent？
一句话回答：
当规则分支越来越难维护，运行时决策越来越多时，Workflow 就会向 Agent 演化。

回答主线：
1. if-else 和规则树开始爆炸。
2. 中间反馈对后续路径的影响越来越大。
3. 继续硬编码的维护成本高于引入动态决策。

高频追问：
有没有一个明显信号？

追问回答：
有。比如业务方开始频繁提“这一步要看上一步结果灵活处理”，或者你发现规则改动成本越来越高、线上异常分支越来越多，这就是很典型的演化信号。

### Q5. 为什么很多业务最后会落成“Workflow 为主，Agent 为辅”？
一句话回答：
因为这是效果、稳定性和治理成本之间最现实的平衡点。

回答主线：
1. 纯 Workflow 不够灵活。
2. 纯 Agent 太难控。
3. 混合形态最容易上线、回溯和放量。

高频追问：
什么时候会反过来变成 Agent 为主？

追问回答：
当业务的核心价值就在于动态决策本身，比如研究、复杂分析、跨系统协作，这时 Agent 可能是主线，Workflow 只是保底壳子。

## 5. Agent 的最小架构

### Q1. 如果基于 LangChain / LangGraph 搭一个最小可用 Agent，最小闭环至少需要哪些模块？
一句话回答：
至少要有模型、状态、动作选择、工具执行、Observation 写回和结束条件，这六个模块缺一个，闭环都不稳定。

回答主线：
1. 模型负责基于当前上下文决定下一步要做什么。
2. 状态负责把目标、历史轨迹和中间结果稳住，不然每一步都会丢现场。
3. 工具执行和 Observation 写回负责把“思考”变成“动作”，再把动作结果重新喂给系统。

高频追问：
Memory 是最小必需吗？

追问回答：
不是。最小闭环只靠 session 内 state 就能跑起来；memory 更像增强项，用来解决跨轮偏好、长期历史和用户画像，不是第一步必须品。

### Q2. 这些模块在 LangChain / LangGraph 里通常分别由什么抽象承载？
一句话回答：
通常会拆成模型对象、Prompt 或 state、Tool、Graph 的节点与边，以及显式 stop logic，而不是揉成一个黑盒 agent。

回答主线：
1. 模型一般由 chat model 或 runnable 承载，负责推理和动作选择。
2. 状态和流转更适合放在 LangGraph 的 state graph 里统一管理，这样轨迹和分支才清楚。
3. 工具调用、节点编排和 stop logic 最好显式落图，不要全靠 prompt 暗示，不然后面很难排障。

高频追问：
为什么现在更常讲 LangGraph，而不是只讲 AgentExecutor？

追问回答：
因为 LangGraph 把状态、分支和终止条件都显式化了。做 demo 时黑盒 executor 很省事，但一到线上，要 trace、回放、控路径，graph 这种显式结构会明显更稳。

### Q3. 在一个最小可运行 Agent 里，哪些是最小必需，哪些属于增强项？
一句话回答：
最小必需是模型、状态、工具、Observation 和 stop condition；规划器、记忆、评估器、反思器通常都是增强项。

回答主线：
1. 没有最小必需，系统连完整闭环都跑不起来。
2. 增强项的作用是把系统从“能跑”提升到“更稳、更长、更复杂任务也能跑”。
3. 我更认可先把最小闭环跑通，再按问题补能力，而不是一开始把所有模块都堆满。

高频追问：
线上第一批最值得补的增强项是什么？

追问回答：
通常是 guardrail、trace、回放和基础评估，这些先把系统控住；planning、reflection、长期 memory 这些更适合在主链路稳定后再补。

### Q4. Observation 在 Agent 体系里为什么是关键输入？
一句话回答：
因为 Agent 后续所有决策，本质上都建立在“上一步到底发生了什么”这个输入上。

回答主线：
1. Observation 让系统知道上一步动作的真实结果，而不是只看模型自己想象的结果。
2. 它决定下一步是继续、回退、重试还是结束。
3. 它同时也是回放轨迹、定位问题和做评估时最关键的一层证据。

高频追问：
Observation 和工具返回值是一回事吗？

追问回答：
不完全一样。工具返回值是原始结果，Observation 是进入下一轮决策的结构化输入，里面通常还会加状态摘要、错误码、业务语义，方便系统后续判断。

### Q5. 为什么真实 Agent 通常不能只有“模型 + 工具”这两层？
一句话回答：
因为那样只有能力，没有控制，系统看起来能跑，但很难稳定收敛。

回答主线：
1. 工具只能负责执行，不能保证整条路径是对的。
2. 模型虽然能做决策，但不擅长独立管理长链路过程。
3. 所以真实系统一定还要补状态层、编排层和 guardrail 层，把过程控住。

高频追问：
为什么很多 demo 只有这两层也能跑？

追问回答：
因为 demo 往往任务短、工具少、上下文干净。真正到生产场景，异常分支、脏状态、权限限制一上来，两层结构很快就撑不住。

### Q6. Tool 层、状态层、编排层、Guardrail 层的边界应该怎么划？
一句话回答：
我的划分方式是: Tool 负责做事，State 负责记账，Orchestration 负责控路，Guardrail 负责兜底。

回答主线：
1. Tool 只暴露能力，不承担全局决策，不让它夹带过多业务控制逻辑。
2. State 负责保存目标、上下文和中间产物，相当于整个系统的现场。
3. Orchestration 控制节点流转，Guardrail 负责权限、格式、风险和终止条件。

高频追问：
最常见的边界混乱是什么？

追问回答：
最常见的混乱有两种: 一种是把业务规则塞进 tool，让 tool 变成黑盒；另一种是把所有控制逻辑都丢给模型，让链路完全不可控。这两种后面都会非常难测、难改、难排障。

## 6. Agent 的运行链路

### Q1. 如果基于 LangGraph 把一个 Agent 真正跑起来，从接到请求到返回结果，完整链路通常是什么？
一句话回答：
通常是先建 state，再让模型选动作，执行工具，写回 Observation，循环推进，直到命中 stop condition，最后再整理结果输出。

回答主线：
1. 入口阶段先做任务解析、上下文拼装和 state 初始化，把任务现场搭起来。
2. 中间阶段是 decision -> act -> observe 的主循环，这部分才是 Agent 真正工作的地方。
3. 结束阶段做结果整理、结构化输出、日志落盘和必要状态更新，保证后面可追踪、可回看。

高频追问：
LangGraph 最重要的价值是什么？

追问回答：
它把节点、状态和边都显式化了，让你能把“模型在做判断”和“系统在跑流程”分开管理。对生产系统来说，这点非常重要，因为大多数问题最终都要落到状态和路径上看。

### Q2. 为什么 Agent 要把“生成答案”和“选择动作”拆开看？
一句话回答：
因为“说什么”和“做什么”本来就不是一个问题，混在一起最容易把系统做成黑盒。

回答主线：
1. 生成答案关注表达质量和结果组织。
2. 选择动作关注路径正确性、调用安全和链路收敛。
3. 如果两者混在一起，后面很难判断到底是内容生成错了，还是动作决策错了。

高频追问：
什么时候可以合在一起？

追问回答：
任务很短、工具很少时可以轻量合并；但只要涉及多工具、多分支或者高风险动作，我都会建议显式拆开，不然后面很难控。

### Q3. Agent 每一步决策时，通常依赖哪些输入？
一句话回答：
通常依赖目标、当前状态、历史轨迹、Observation、工具描述和系统约束这几类输入。

回答主线：
1. 目标决定方向，不然模型不知道自己这一步到底在服务什么。
2. 状态和轨迹决定当前上下文，不然它会失去对任务进度的感知。
3. 工具 schema 和系统约束决定当前能做什么、不能做什么，这是系统可控性的来源。

高频追问：
为什么只喂聊天历史不够？

追问回答：
因为聊天历史只是文本表层，它不包含显式状态、工具约束和业务规则。只喂对话，本质上是在让模型凭感觉猜，而不是让系统基于结构化现场做判断。

### Q4. Agent 如何判断当前应该继续思考、调用工具，还是直接结束？
一句话回答：
本质上就是判断“离目标还差什么”，以及“继续动作的收益，是否还大于它的成本和风险”。

回答主线：
1. 如果缺关键信息，应该查或调工具。
2. 如果已有足够证据，应该汇总并结束。
3. 如果继续动作只会增加噪声和成本，就该停。

高频追问：
这个判断能不能全交给模型？

追问回答：
不能完全交。模型可以给候选判断，但 stop rule、最大步数、关键节点校验最好还是由系统侧显式兜住，不然很容易过早结束或停不下来。

### Q5. stop condition 一般应该怎么设计？
一句话回答：
要同时有语义完成条件和工程兜底条件，两边缺一边都不稳。

回答主线：
1. 语义层面要判断任务目标是否真的完成了。
2. 工程层面要限制步数、时间、成本和错误次数。
3. 高风险任务还要准备人工接管或降级出口，避免系统一直硬跑。

高频追问：
为什么只靠“模型说完成了”不够？

追问回答：
因为模型很容易过早结束，也很容易“看起来像完成了”。真正可靠的 stop condition 一定要和可验证状态、工具结果或业务规则绑定。

### Q6. 为什么很多 Agent 最后都要落成显式状态机，而不能完全交给模型自由发挥？
一句话回答：
因为线上系统最终拼的不是自由度上限，而是可控性下限。

回答主线：
1. 状态机能把关键分支、终止条件和回退路径显式化。
2. 这样线上问题更容易回放、定位和兜底。
3. 真正落地时，系统可控性通常比理论自由度更重要。

高频追问：
显式状态机会不会削弱 Agent 能力？

追问回答：
会牺牲一部分自由探索，但换来的是可上线、可回放、可治理。对生产系统来说，这个 trade-off 通常是值得的。

## 7. Agent 为什么容易失控

### Q1. Agent 为什么容易任务漂移、重复调用、重复试错，甚至死循环？
一句话回答：
因为目标表达、状态表示、动作约束和停止条件只要有一层没收紧，系统就会开始乱跑。

回答主线：
1. 目标不清会导致决策漂移。
2. Observation 处理不好会导致重复尝试。
3. stop condition 弱和动作空间过大最容易引发死循环。

高频追问：
先排哪一层最划算？

追问回答：
先看轨迹和 stop rule，再看 tool schema 和状态写回。很多循环问题不是模型傻，而是系统没告诉它“什么时候该停”和“当前到底发生了什么”。

### Q2. 为什么 Agent 会出现“每一步看起来都对，但最终结果是错的”？
一句话回答：
因为局部最优不等于全局正确，多步链路很容易在中间逐步偏航。

回答主线：
1. 每一步都可能是合理但不必要的动作。
2. 中间状态一旦被错误更新，后面会越走越顺地走偏。
3. 所以光看单步成功率不够，要看整条轨迹是否朝目标收敛。

高频追问：
这类问题怎么评估？

追问回答：
要同时看终局成功率和轨迹质量。很多系统 tool success 很高，但 task success 很低，本质就是局部动作没问题、全局路径不对。

### Q3. 为什么很多 Agent demo 很聪明，线上却很脆？
一句话回答：
因为 demo 用的是理想输入和短链路，线上面对的是脏状态、异常工具和复杂约束。

回答主线：
1. demo 往往没有覆盖异常分支。
2. 线上工具返回值和用户输入会更脏。
3. 真实系统还要考虑权限、成本、时延和并发。

高频追问：
一句话讲根因是什么？

追问回答：
demo 考的是能力上限，线上考的是系统下限。

### Q4. Agent 不收敛时，通常是哪几层同时出了问题？
一句话回答：
通常是目标层、状态层、工具层和终止层一起出了问题。

回答主线：
1. 目标模糊会让方向不稳。
2. 状态脏会让决策输入失真。
3. 工具和 stop rule 不严会让错误不断放大。

高频追问：
为什么说这不是单点故障？

追问回答：
因为 Agent 是复合系统。模型、上下文、工具、编排任一层出问题都会往下游传递，最后表现成“不收敛”，但根因往往不止一个。

### Q5. 如果要系统性地解释 Agent 失控，应该从哪些根因层来拆？
一句话回答：
我一般从目标定义、上下文状态、动作空间、执行环境和评估闭环这五层来拆。

回答主线：
1. 目标是否清晰、可验证。
2. 状态是否真实、干净、可回放。
3. 动作空间是否过大、工具是否受约束。

高频追问：
哪层最容易被忽略？

追问回答：
评估闭环最容易被忽略。很多团队能搭系统、能跑 demo，但没有持续回放和 bad case 机制，所以问题会反复出现。

## 8. Agent 的常见模式

### Q1. ReAct、Plan-and-Execute、Reflexion、Self-Refine 这几类模式分别在解决什么问题？
一句话回答：
它们本质上是在不同位置补“规划能力”和“纠错能力”，区别不在名字，而在规划和反馈被放在哪一层。

回答主线：
1. ReAct 强在边想边做，适合在线探索和信息不完整的任务。
2. Plan-and-Execute 强在先分解再执行，适合长链路、任务结构相对清晰的场景。
3. Reflexion、Self-Refine 强在结果出来后做自检和修正，适合可验证、可迭代优化的任务。

高频追问：
它们最大的差别是什么？

追问回答：
最大差别是规划放在前面、过程中还是后面，以及反馈是实时介入还是结果出来后再修。说白了，就是它们把“先想”和“再改”放在了不同位置，所以适合的任务长度和稳定性要求也不一样。

### Q2. 这些模式分别适合什么任务，不适合什么任务？
一句话回答：
没有通用最优模式，关键看任务长度、路径不确定性和反馈成本。

回答主线：
1. 短任务、在线探索类任务适合 ReAct，因为它能边查边走。
2. 长链路、可拆子目标的任务适合 Plan-and-Execute，因为先拆再跑更稳。
3. 结果容易校验、允许多轮修正的任务适合 Reflexion 或 Self-Refine。

高频追问：
最常见的错配是什么？

追问回答：
最常见的错配就是所有任务都默认上 ReAct。结果长任务没有全局规划，短任务又被过度思考，最后既慢又不稳。

### Q3. 为什么不是所有任务都值得上显式规划？
一句话回答：
因为规划本身就是额外成本，而且很多任务根本不值得为规划再多走一层链路。

回答主线：
1. 短任务本来就能直接做，显式规划收益很低。
2. 规划一旦错了，会把整条链路带偏，而且错误会更隐蔽。
3. 所以显式规划适合复杂任务，不适合作为默认配置。

高频追问：
怎么判断该不该上规划？

追问回答：
看两个点就够了: 一是任务有没有稳定的中间目标，二是错误规划的代价是不是比不规划更低。如果这两个条件都不明显，就别急着上规划。

### Q4. 为什么有些任务用 ReAct 反而会变差？
一句话回答：
因为 ReAct 会引入额外思考和探索步骤，在简单任务上这些步骤往往不是收益，而是噪声。

回答主线：
1. 它会增加 token、决策次数和链路长度。
2. 简单任务本来就不需要多轮探索，强行加 ReAct 只会多出误判机会。
3. 任务越短，越要警惕过度 Agent 化。

高频追问：
那什么场景 ReAct 反而很值？

追问回答：
当外部环境未知、必须边查边做，而且单步 Observation 对下一步决策很关键时，ReAct 的收益会很明显，这时候它的探索成本是值得付的。

### Q5. 如果让你在 LangGraph 里落一个真实业务 Agent，通常应该怎么选模式，而不是把 ReAct 当默认答案？
一句话回答：
我不会先选模式，我会先拆任务结构，再决定是纯编排、轻 ReAct、显式规划，还是多阶段混合。

回答主线：
1. 先看任务是不是短、稳、可写死，能写死就别强上 Agent 模式。
2. 再看是否需要中间子目标、显式验证和长链路管理。
3. 最后再看为了更强能力，值不值得接受更高复杂度和更重治理成本。

高频追问：
你最常用的落地方式是什么？

追问回答：
最常见的落地方式是 Workflow 打主链路，关键节点上轻量 ReAct 或短规划。真正全程自由探索的纯 Agent，我只会放在极少数高价值、高容错的任务里，因为那种链路的治理成本太高。

## 9. 单 Agent 与多 Agent

### Q1. 单 Agent 和多 Agent 应该怎么选？
一句话回答：
默认先单 Agent，只有当上下文、职责或工具边界已经明显撑爆时，我才会考虑多 Agent。

回答主线：
1. 单 Agent 更简单，成本更低，可观测性也更好。
2. 多 Agent 的核心价值是职责解耦、上下文隔离和角色分工。
3. 但一旦拆成多 Agent，通信、协调、同步和失控风险都会明显上升。

高频追问：
一句话给选择原则。

追问回答：
能用单 Agent 解决，就别急着上多 Agent；多 Agent 应该是为了解决明确的边界问题，而不是为了显得更高级。

### Q2. 什么情况下值得上多 Agent？
一句话回答：
当任务天然可拆、角色差异大、上下文冲突明显时，才值得上多 Agent。

回答主线：
1. 不同子任务需要不同专长、不同工具集合或不同决策逻辑。
2. 单 Agent 的上下文已经过载，继续往里塞只会越来越乱。
3. 拆开后如果能明显提升可维护性、可观测性和结果质量，这时才值得上。

高频追问：
哪些拆法是伪拆分？

追问回答：
只是把同一件事机械分成几个 agent，但它们没有真实边界、还共享大量上下文，这种拆法通常只会增加 token、增加协调成本，不会提升效果。

### Q3. 多 Agent 相比单 Agent 的核心收益和核心代价分别是什么？
一句话回答：
核心收益是分工和隔离，核心代价是通信、同步和治理成本。

回答主线：
1. 收益在于职责清晰、上下文更短、角色能力更容易收敛。
2. 代价在于消息传递、状态同步和调度复杂度显著上升。
3. 很多多 Agent 问题最后拼的不是模型多聪明，而是系统协调能力够不够稳。

高频追问：
最容易被低估的代价是什么？

追问回答：
最容易被低估的是状态一致性和错误定位。单 Agent 出错，你至少还能盯一条轨迹；多 Agent 出错，经常是多条轨迹互相影响，很难第一时间找到真正根因。

### Q4. 多 Agent 为什么特别容易带来成本膨胀和链路失控？
一句话回答：
因为 token、工具调用、消息传递和重复思考会被链式放大。

回答主线：
1. 每多一个 agent，就多一层上下文拼接、消息传递和决策成本。
2. 代理之间很容易重复工作，甚至互相来回确认。
3. 一旦没有统一裁决和终止机制，链路就会越跑越长、越跑越散。

高频追问：
怎么压这种成本？

追问回答：
靠严格角色边界、共享中间产物而不是共享全文、限制调用次数、加统一 coordinator 和 stop rule。多 Agent 最怕的不是 agent 少，而是每个 agent 都太万能。

### Q5. 在什么情况下坚持单 Agent 反而是更强的工程判断，而不是“能力做不到多 Agent”？
一句话回答：
当核心问题根本不在“角色不够多”，而在工具质量、状态设计或评测治理时，坚持单 Agent 往往是更成熟的选择。

回答主线：
1. 多 Agent 不是解决一切复杂性的万能钥匙，它只解决“该拆的复杂性”。
2. 很多问题拆 agent 之前，先把单 Agent 做稳会更有效，也更便宜。
3. 这不是保守，而是工程优先级判断。

高频追问：
怎么在面试里体现这种判断？

追问回答：
直接把 trade-off 讲透。比如“我评估过多 Agent，但发现真正卡点不是职责拆不清，而是工具返回不稳定，所以先把单 Agent 做稳更划算”，这种回答会比“我会多 Agent”更像成熟工程师。

## 10. Agent 和 Tool、Memory、RAG 的关系

### Q1. Agent 和 Tool Calling、Memory、RAG 分别是什么关系？
一句话回答：
最短的理解就是: Agent 负责决策，Tool 负责执行，Memory 负责保留，RAG 负责补知识。

回答主线：
1. 它们不是替代关系，而是分工关系。
2. Agent 是控制层，其余三个更像能力层和支撑层。
3. 真正上线的 Agent 很少脱离这三者单独存在。

高频追问：
谁才是核心？

追问回答：
控制层是核心。没有 Agent 时，Tool、Memory、RAG 只是独立能力；有了 Agent，它们才被组织成一个可推进任务的系统。

### Q2. 为什么很多 Agent 项目本质上是 `RAG + Tool + State` 的组合系统？
一句话回答：
因为大部分业务任务需要先拿知识，再做动作，再持续维护过程状态。

回答主线：
1. RAG 解决“知道什么”。
2. Tool 解决“能做什么”。
3. State 解决“当前进行到哪一步”。

高频追问：
那 Agent 在里面扮演什么？

追问回答：
Agent 扮演的是决策和编排角色，决定什么时候查、什么时候做、什么时候停，不是简单把三块拼在一起。

### Q3. 什么情况下 Agent 必须接工具？
一句话回答：
只要任务需要和真实世界交互，Agent 基本就必须接工具。

回答主线：
1. 查实时信息要工具。
2. 执行业务动作要工具。
3. 获取可验证外部结果也要工具。

高频追问：
不接工具可以做什么？

追问回答：
可以做纯文本推理、总结、规划和对话，但执行力会明显受限，更像 reasoning assistant，而不是 action agent。

### Q4. 什么情况下 Agent 必须接记忆？
一句话回答：
当任务跨轮、跨会话，或者用户偏好和历史会显著影响后续决策时，就必须接记忆。

回答主线：
1. 记忆让系统避免每轮都从零开始。
2. 它适合存用户偏好、长期任务状态和关键历史。
3. 但记忆不是越多越好，脏记忆会直接带偏 Agent。

高频追问：
什么不该进记忆？

追问回答：
低价值噪声、未经验证的中间猜测、容易过期的信息都不该直接进长期记忆。记忆要像数据库，不要像聊天流水账。

### Q5. 什么情况下 Agent 必须接 RAG？
一句话回答：
当任务依赖私有知识、实时知识或超出模型训练边界的知识时，就必须接 RAG。

回答主线：
1. 私域知识不在模型参数里。
2. 实时变化的信息不适合只靠预训练知识。
3. RAG 还能提供一定的可追溯性和引用依据。

高频追问：
有了大模型长上下文，还需要 RAG 吗？

追问回答：
通常还需要。长上下文解决的是“能装更多”，RAG 解决的是“怎么找到对的”。没有检索和筛选，长上下文只会把噪声也一起装进去。

### Q6. Tool、Memory、RAG 三者分别在解决什么问题，边界又该怎么划？
一句话回答：
Tool 解决执行，Memory 解决保留，RAG 解决知识补充，三者的边界要按职责而不是按实现技术来划。

回答主线：
1. Tool 是动作能力，不负责长期状态。
2. Memory 是状态和偏好，不负责知识搜索。
3. RAG 是外部知识获取，不负责业务动作执行。

高频追问：
最常见的边界错误是什么？

追问回答：
把检索结果直接当 memory 存、把记忆系统当知识库用、或者让 tool 承担业务控制逻辑。这三种都会让系统越来越乱。

### Q7. 如果基于 LangChain / LangGraph 落地，这三层通常应该怎么接，而不是互相混在一起？
一句话回答：
我的做法是把 RAG、Tool、Memory 都做成显式能力层，由 Agent 通过 state 和 graph 去组织，而不是塞进一个大 prompt 里糊过去。

回答主线：
1. RAG 单独成检索链路，输入输出清晰。
2. Tool 单独成 action layer，schema 和权限独立治理。
3. Memory 单独成 state/memory layer，读写条件显式化。

高频追问：
为什么不能直接都塞进一个 agent prompt？

追问回答：
因为一旦混成黑盒，后面你几乎没法判断问题到底出在检索、记忆、工具还是决策层。线上一出问题，排障成本会非常高。

## 11. 动作空间与系统约束

### Q1. 什么叫 Agent 的动作空间？
一句话回答：
动作空间就是 Agent 在当前系统里被允许做的所有决策和执行动作的集合。

回答主线：
1. 包括能调用哪些工具。
2. 包括每个工具能怎么调。
3. 还包括路径分支、重试策略和终止方式。

高频追问：
为什么动作空间是 Agent 设计里的核心概念？

追问回答：
因为 Agent 的本质就是在动作空间里做选择。动作空间定义得不好，后面再强的模型也只是更快地做错事。

### Q2. 为什么 Agent 的动作空间必须受约束，而不能无限开放？
一句话回答：
因为无限开放会让错误路径、风险动作和成本膨胀同时出现。

回答主线：
1. 模型不是稳定的全局优化器。
2. 自由度越高，错误组合越多。
3. 生产系统必须优先可控，而不是无限自由。

高频追问：
是不是约束越多越好？

追问回答：
不是。约束太死会把系统做成低效 Workflow。关键是限制危险自由度，同时保留真正需要的弹性。

### Q3. 动作空间过大时，通常会带来哪些系统性问题？
一句话回答：
通常会带来误调、漏调、重复试错、死循环、成本膨胀和权限风险。

回答主线：
1. 模型更难在大量候选动作里稳定选择。
2. 工具调用会出现很多无效尝试。
3. 整体链路的可预测性会迅速下降。

高频追问：
最先要收哪部分？

追问回答：
先收高风险工具和高歧义工具，再收参数范围和调用次数。先把危险和最容易误用的地方关住，收益最大。

### Q4. 如何理解“Agent 的自由度越高，系统越不稳定”？
一句话回答：
因为每增加一层可选空间，就增加了一层不确定性传播的可能。

回答主线：
1. 更多动作意味着更多错误分支。
2. 多步系统会把局部偏差累积成全局偏差。
3. 所以系统设计里，自由度和稳定性永远在 trade-off。

高频追问：
那是不是高自由度一定不好？

追问回答：
不是。高自由度在探索型任务里很有价值，但必须配合强约束、强评估和强回放，否则收益很容易被系统风险吃掉。

### Q5. 参数约束、权限控制、工具白名单、显式编排分别在限制什么？
一句话回答：
它们分别限制的是输入范围、可执行边界、能力集合和路径自由度。

回答主线：
1. 参数约束防止工具被乱用。
2. 权限控制和白名单防止越权和误调。
3. 显式编排限制链路发散和无意义探索。

高频追问：
哪个最常被低估？

追问回答：
参数约束最容易被低估。很多系统工具列表看起来很安全，但参数一旦不收，实际风险一点都不小。

### Q6. 如果你用 LangGraph 或自定义编排框架实现，一个可控的动作空间通常该怎么收紧？
一句话回答：
我的做法是从工具集合、工具 schema、节点边、状态机和 stop rule 五层一起收紧。

回答主线：
1. 工具只暴露必要能力，不搞大而全。
2. schema 严格化，参数范围和格式明确。
3. 图上的边和终止条件显式控制，避免自由乱跳。

高频追问：
哪一层收益最大？

追问回答：
通常是工具 schema 和显式边控制收益最大，因为它们直接决定“能不能做错事”和“会不会跑偏”。

## 12. 线上问题、评测与工程化

### Q1. Agent 系统最常见的线上问题通常有哪些？
一句话回答：
最常见的是误调工具、轨迹漂移、状态污染、超时、成本失控和任务未收敛。

回答主线：
1. 这类问题很多都不是模型单点问题。
2. 往往是工具、状态和编排一起放大出来的。
3. 所以线上治理一定要系统化。

高频追问：
最难处理的是哪种？

追问回答：
最难的是“偶发且不可复现”的轨迹问题，因为它既影响用户体验，又很难用单点日志直接复盘出来。

### Q2. 如何判断问题出在模型、提示词、工具、状态管理还是编排逻辑？
一句话回答：
核心方法不是猜，而是顺着轨迹做分层归因。

回答主线：
1. 先回放完整链路，看错误从哪一步开始出现。
2. 再区分是决策错、执行错还是状态写错。
3. 最后结合输入输出和环境信息做归因。

高频追问：
最快的定位入口是什么？

追问回答：
先看 trace 里的“决策点、工具调用、Observation 写回”这三个位置。很多问题一旦把这三段串起来，归因方向就已经很清楚了。

### Q3. 为什么 Agent 系统比普通问答系统更需要 trace、回放、状态快照和可观测性？
一句话回答：
因为 Agent 的问题不是一句输出能解释的，而是一条轨迹才能解释的。

回答主线：
1. 单轮问答只看输入输出，Agent 还要看过程。
2. 没有状态快照，就很难复现问题。
3. 没有回放机制，就几乎没法系统调优。

高频追问：
最小可观测集应该包含什么？

追问回答：
至少要有用户输入、关键状态、模型决策、工具调用参数和结果、Observation、终止原因、耗时和成本。

### Q4. Agent 为什么比单轮问答更难评测，评测时通常要看哪些层？
一句话回答：
因为它不是只评一句输出，而是在评整条任务执行链。

回答主线：
1. 要看最终结果对不对。
2. 还要看路径合不合理、工具用得对不对。
3. 有时还要看是否稳定、是否收敛、是否满足业务约束。

高频追问：
最重要的指标是什么？

追问回答：
我会把任务成功率放在第一位，但不会只看它。Agent 评测一定要把终局指标和过程指标一起看，不然很容易误判。

### Q5. 工具调用成功率、轨迹质量、规划能力、任务成功率之间应该怎么一起看？
一句话回答：
我的原则是终局优先、过程辅助，不能只看局部指标。

回答主线：
1. 工具成功率高，不代表任务一定成功。
2. 轨迹漂亮，不代表结果一定有业务价值。
3. 最合理的是把任务成功率当主指标，其余指标用来解释问题在哪。

高频追问：
哪个指标最容易把团队带偏？

追问回答：
工具成功率最容易带偏，因为它很容易做高，但经常掩盖“工具都调对了，任务还是没完成”的真实问题。

### Q6. 如何在效果、成本、时延之间做 Agent 的工程折中？
一句话回答：
我会先确定业务底线，再决定预算是花在更强模型、更长链路，还是更严格的系统治理上。

回答主线：
1. 不同业务对效果、成本、时延的容忍度完全不同。
2. 很多优化不是追单点最优，而是找可接受区间。
3. 真正成熟的做法是有降级策略和分层路由。

高频追问：
最常见的降级手段是什么？

追问回答：
比如减少工具步数、关闭显式规划、切轻量模型、从 Agent 降到 Workflow、从全自动降到半自动人工确认，这些都很常见。

## 13. 平台化与框架选型

### Q1. Agent 为什么会走向平台化，而不是继续让业务各自手写？
一句话回答：
因为大家重复造轮子太多，而 Agent 的共性基础设施又很重。

回答主线：
1. 工具接入、状态管理、trace、评测、权限控制都高度通用。
2. 每个业务单独手写，成本高且质量不稳定。
3. 平台化的价值是复用、治理和统一演进。

高频追问：
什么阶段适合做平台？

追问回答：
当你已经有多个业务在重复做同类 Agent 能力，且这些能力开始成为上线瓶颈时，就值得平台化；太早做往往会过度设计。

### Q2. Agent 中台和普通 AI 应用后端的区别是什么？
一句话回答：
普通 AI 后端主要托模型调用，Agent 中台要托整条智能执行链路。

回答主线：
1. 它不仅服务模型，还要服务工具、状态、编排和评测。
2. 它更强调轨迹治理，而不是单次请求治理。
3. 它要支持业务方快速搭 agent，而不只是调一个接口。

高频追问：
一句话讲核心差别。

追问回答：
普通 AI 后端解决“怎么调模型”，Agent 中台解决“怎么让智能系统稳定地跑起来”。

### Q3. 一个 Agent 平台通常需要哪些核心能力？
一句话回答：
通常要有编排、工具接入、状态管理、观测评测、权限治理和发布运维能力。

回答主线：
1. 编排和状态是核心骨架。
2. 工具、记忆、知识接入是能力层。
3. trace、回放、评测、权限和发布是治理层。

高频追问：
最容易漏掉哪块？

追问回答：
评测和回放最容易漏。很多平台先把“能搭起来”做了，却没把“怎么稳定迭代”做出来，后面很快就会卡住。

### Q4. 为什么 Agent 平台往往同时需要工作流编排能力和可视化画布？
一句话回答：
因为一方面要控制复杂链路，另一方面要降低业务使用门槛和提升可理解性。

回答主线：
1. 工作流编排保证链路可控。
2. 可视化画布帮助业务和研发共同理解系统。
3. 两者结合更适合企业协作和快速迭代。

高频追问：
画布真的只是 UI 吗？

追问回答：
不是。画布本质上是平台抽象的外显，它决定了用户怎么理解节点、状态、边和权限，所以它会反过来影响整个平台能力边界。

### Q5. LangChain、LangGraph、AutoGen、CrewAI 这类 Agent 框架通常应该从哪些维度做比较？
一句话回答：
我会重点看状态控制、编排能力、可观测性、扩展性、接入成本和生产适配度。

回答主线：
1. 先看它更适合 demo 还是 production。
2. 再看状态、图编排和多 Agent 支持够不够强。
3. 最后看和现有技术栈是否容易融合。

高频追问：
为什么 LangGraph 近两年更常被提？

追问回答：
因为它把状态和图编排显式化了，更贴近真实生产系统。很多团队从黑盒 AgentExecutor 转向 graph，本质上是在换取更强的可控性和可维护性。

### Q6. 如果项目最后没有继续重度依赖框架，而是逐步下沉自研，通常是因为什么？
一句话回答：
通常是因为框架抽象开始和业务真实约束错位了。

回答主线：
1. 框架适合起步和验证。
2. 业务一旦复杂，团队会更关注控制权、性能和可观测性。
3. 这时就容易从“用框架搭”转向“借框架思路，自研关键层”。

高频追问：
这是不是说明框架不好？

追问回答：
不是。框架非常适合起步和验证，只是到了某个规模，团队更想要的是关键控制权，而不是继续被通用抽象牵着走。

### Q7. Modular Agent 这类设计通常在解决什么问题？
一句话回答：
它主要在解决职责耦合过重、系统难扩展和难替换的问题。

回答主线：
1. 把 planner、executor、memory、tooling 等拆成可替换模块。
2. 便于不同业务按需组合。
3. 也更方便做分层治理和独立演进。

高频追问：
什么时候不值得做模块化？

追问回答：
项目还很早、形态没稳定时不值得过早模块化。太早抽象只会把简单问题做复杂。

## 14. Agent 项目深挖

### Q1. 什么样的业务场景适合做 Agent？
一句话回答：
最适合做 Agent 的，是那种目标明确、过程不固定、又必须接外部能力才能完成的任务。

回答主线：
1. 它得是任务闭环，而不是单纯对话陪聊。
2. 中间决策必须真的会影响结果，不是走个形式。
3. 上 Agent 之后要能换来真实收益，而不是只换来更复杂的架构。

高频追问：
什么叫“收益足够大”？

追问回答：
至少要能明显提高效率、提升转化、降低人工成本，或者把原来做不了的任务做出来。否则很难 justify 这套更复杂的系统。

### Q2. 如果不用 Agent，改做 Workflow，会损失什么能力，又会换来什么好处？
一句话回答：
会损失动态决策和任务弹性，但会换来更强的稳定性、可控性和可审计性。

回答主线：
1. Workflow 的优势是可预测、好控、好审计。
2. Agent 的优势是能处理不确定性和非固定路径。
3. 所以本质上是在用治理成本换任务弹性。

高频追问：
面试里怎么答这类权衡题？

追问回答：
不要只说哪种更高级，而要先讲业务目标，再讲损失和收益。你只要把 trade-off 讲透，面试官通常就能判断你是不是做过真实系统。

### Q3. Agent 架构通常应该怎么设计？
一句话回答：
我会把它设计成控制层、能力层、状态层和治理层四层，而不是只画一张“模型接工具”的示意图。

回答主线：
1. 控制层负责决策和编排，决定系统怎么跑。
2. 能力层负责工具、RAG、memory 等能力接入，决定系统能做什么。
3. 状态层和治理层负责把现场记住、把链路看清、把风险控住。

高频追问：
架构图里最值得强调什么？

追问回答：
最值得强调的是状态流转和关键决策点，因为这两部分最能体现你是在讲系统如何运行，而不是只是在罗列几个组件名。

### Q4. 如果让你结合一个真实项目讲 Agent 架构，最值得展开讲的模块通常有哪些？
一句话回答：
我会优先讲状态设计、工具层、编排逻辑、终止机制和观测评测，这五层最容易拉开项目深度差距。

回答主线：
1. 这几层最能体现系统是不是可控，而不只是能跑。
2. 也是线上最容易出问题、最能体现工程深度的地方。
3. 比起只讲“我用了什么模型”，这几层更能证明你真的做过系统。

高频追问：
为什么不先讲 prompt？

追问回答：
因为 prompt 更像局部优化，真正拉开差距的是你怎么把系统搭稳，怎么让它可回放、可评估、可治理。只会讲 prompt，通常说明项目深度还没到系统层。

### Q5. 为什么会选成当前框架、当前模式、当前工具体系？
一句话回答：
成熟的答案一定不是“因为它流行”，而是“因为它和任务结构、团队能力、治理要求正好匹配”。

回答主线：
1. 框架看控制力、开发效率和后续维护成本。
2. 模式看任务长度、复杂度和是否需要显式规划。
3. 工具体系看外部依赖、风险、权限和复用性。

高频追问：
怎么避免答成技术栈背诵？

追问回答：
一定要讲“如果不用这个会怎样”。把替代方案和你为什么没选它说清楚，回答就会立刻从“背技术栈”变成“做技术决策”。

### Q6. 线上最难处理的 Agent bad case 通常是什么？
一句话回答：
最难的 bad case 往往不是明显报错，而是“每一步都像对的，最后整体却是错的，而且还很难稳定复现”。

回答主线：
1. 这类 case 往往不是一个点炸了，而是多层小问题叠加。
2. 单看模型输出通常看不出异常，甚至每一步都像合理选择。
3. 必须靠轨迹、状态快照和环境信息一起复盘，才能把根因拆出来。

高频追问：
怎么回答这类复盘题更好？

追问回答：
用“现象 -> 定位过程 -> 根因 -> 修复 -> 预防机制”这个结构讲，不要只停留在“我们优化了 prompt”，那样会显得太浅。

### Q7. 从零设计一个企业级 Agent 系统时，应该先从哪些模块开始？
一句话回答：
先从可控最小闭环开始，再补治理，不要一上来就追求全功能平台。

回答主线：
1. 第一阶段先把核心任务跑通，证明主链路成立。
2. 第二阶段补 trace、回放、权限和评测，把系统控住。
3. 第三阶段再考虑平台化、多 Agent 和复杂能力扩展，不要顺序反了。

高频追问：
最容易做错的顺序是什么？

追问回答：
最容易一开始就堆规划、多 Agent、记忆、平台化，结果主链路都没跑稳。企业级不等于一开始就做得很重，而是先把主链路做对，再把能力做大。

### Q8. 如果重做一次当前 Agent 系统，通常最值得优先改哪一层？
一句话回答：
如果只能先动一层，我大多数时候会先动状态层和编排层，而不是先换模型。

回答主线：
1. 很多真实问题都不是模型本身不够强，而是状态设计和路径控制不够稳。
2. 这两层一旦改好，后面的模型、工具和评测都会更稳定。
3. 先把系统骨架搭稳，后面的优化收益才不会反复被抵消。

高频追问：
什么时候会优先改模型？

追问回答：
当系统约束已经做得比较完整，问题仍然稳定出在推理能力、工具选择能力或长程理解能力上，这时才优先考虑换模型或做模型侧增强，不然太容易本末倒置。

## 意图识别与意图对齐（面试高频专题）

### Q：什么是意图识别？它在 Agent 系统里的作用是什么？

**一句话回答：**
意图识别是 Agent 系统的入口分流器，它决定用户的请求该走哪条处理路径——是直接回答、检索知识库、调用工具、还是拒绝。

**回答主线：**

意图识别本质上是一个**分类问题**：把用户的自然语言输入映射到系统预定义的意图类别，再根据意图路由到对应的处理逻辑。

在 Agent 体系里，意图识别的价值是**在复杂系统里给每条请求找到最合适的处理路径**，而不是把所有请求都扔进同一个"大模型回答"流程：

```
用户输入
   ↓
意图分类（快速、低成本）
   ├─ FAQ/闲聊类 → 直接生成回答（不检索）
   ├─ 知识查询类 → RAG 检索流程
   ├─ 操作执行类 → 工具调用流程
   ├─ 复杂推理类 → CoT / 多步 Agent 流程
   └─ 超出范围类 → 拒答 / 转人工
```

分流的意义在于：把简单请求用简单（便宜、快速）的方式处理，只把真正需要的请求路由到昂贵的大模型流程，降低成本和延迟。

---

### Q：意图识别有哪些实现方案？各自的优劣是什么？

**回答要点：**

| 方案 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| **规则/关键词匹配** | 正则、关键词白名单 | 速度极快，可解释 | 泛化能力极差，维护成本高 | 意图极少、极稳定的场景 |
| **传统分类模型**（BERT/FastText） | Fine-tune 分类头 | 速度快（<50ms），成本低 | 需要标注数据，难处理边界 case | 意图数量中等、流量大 |
| **Embedding + 相似度** | 把输入 embed，和意图示例做相似度匹配 | 零样本泛化，易扩展 | 阈值难调，相似意图容易混淆 | 意图动态扩展，低流量 |
| **LLM 直接分类** | 让大模型判断意图（zero-shot/few-shot） | 泛化强，边界 case 处理好 | 慢、贵，P99 延迟难控 | 低频高复杂度场景 |
| **小模型路由 + 大模型兜底** | 小模型先判断，低置信度才用大模型 | 兼顾效率和准确率 | 系统复杂度增加 | **生产系统首选** |

**生产系统通常的做法**：

1. **第一层：快速规则过滤**——把明显超范围的请求（如谩骂、完全无关领域）快速拦截，不消耗模型资源
2. **第二层：小分类模型**——BERT/RoBERTa 微调的意图分类器，< 50ms，处理大多数明确意图
3. **第三层：大模型兜底**——低置信度时升级到 LLM 做细粒度判断

### Q：什么是意图对齐（Intent Alignment）？和意图识别有什么区别？

**回答要点：**

- **意图识别（Intent Detection）**：识别用户说的是什么意图——把自然语言映射到意图类别，是分类问题。

- **意图对齐（Intent Alignment）**：确保系统的回答真正满足用户的**真实需求**，而不只是字面上的请求——是更深一层的问题。

两者的区别：

```
用户说："帮我改一下这段代码"
意图识别的结果：code_editing
意图对齐要问的：用户真正想要什么？
  - 只是修 bug？
  - 还是要重构、优化性能？
  - 还是要解释为什么要这么改？
```

意图对齐失败的典型表现：系统回答了用户问的问题，但没有回答用户**真正需要的**问题。比如用户问"这个功能怎么用"，系统给了 API 文档链接，但用户其实想要的是一个快速上手的示例代码。

**意图对齐的工程实现**：

1. **意图消歧（Intent Disambiguation）**：当识别到的意图置信度低、或者意图下有多个子意图时，主动向用户确认："你是想要 A 还是 B？"

2. **槽位填充（Slot Filling）**：识别出意图后，检查必需参数是否齐全。比如"查天气"意图，必须有地点和时间；如果缺了，主动追问，而不是乱猜。

3. **上下文意图推断（Contextual Intent）**：结合对话历史理解意图。"帮我改一下" 在不同上下文里意图完全不同——紧接在"这段代码有 bug"后面，和紧接在"这篇文章风格太正式"后面，指的是完全不同的任务。

### Q：意图识别的训练数据怎么构造？线上数据和标注数据有什么区别？

**回答要点：**

**训练数据构造原则**：

1. **覆盖意图多样性**：每个意图要覆盖不同的表达方式（"帮我查天气" / "今天天气怎么样" / "明天要不要带伞" ——同一意图的三种说法）

2. **负样本和边界 case**：意图分类器最容易在边界 case 上出错——和某个意图相似但不属于该意图的请求。要专门收集和标注这些样本。

3. **意图外（Out-of-Scope，OOS）类**：不属于任何已知意图的请求——这是实际部署中最容易出问题的地方，必须有专门的 OOS 类，而不是强行分配到最近的意图。

**线上数据 vs 标注数据**：

- **标注数据**：质量高，但分布可能和线上真实用户不一样（标注员的表达方式和用户不同）
- **线上数据**：分布准确，但质量参差不齐，需要过滤和抽样标注

生产系统通常是：先用标注数据训练冷启动模型 → 上线后收集真实用户日志 → 对低置信度 case 做人工标注 → 迭代更新模型。

### Q：意图识别系统的评测怎么做？

**回答要点：**

意图识别的评测要关注以下几点：

**指标选择**：

| 指标 | 关注点 | 何时重要 |
|------|--------|---------|
| **整体准确率（Accuracy）** | 所有意图的平均正确率 | 意图分布均匀时 |
| **各意图 F1** | 每个意图的 Precision + Recall | 意图分布不均匀时（关注长尾意图） |
| **OOS 召回率** | 超出范围的请求有没有被正确识别 | 避免把 OOS 请求乱分类 |
| **置信度校准** | 模型的置信度有没有反映真实准确率 | 用于决定何时升级到大模型兜底 |

**线上评测补充**：

- **混淆矩阵分析**：找出哪些意图对之间最容易混淆（通常是语义相近的意图），有针对性地补充区分性训练数据
- **A/B 测试**：新模型上线后，对比路由准确率变化和下游任务完成率变化

### Q：意图识别做得再好，还是会有失败的情况，怎么设计兜底？

**回答要点：**

意图识别没有 100% 准确率，兜底设计比追求准确率更重要：

1. **置信度阈值 + 升级路由**：分类模型给出低置信度时（如 < 0.7），不做强分类，升级到大模型做细粒度判断，或者直接走通用回答路径

2. **意图确认机制**：在高风险操作（写操作、外部 API 调用）之前，显式向用户确认意图："你是要执行 XX 操作吗？"

3. **优雅拒答**：对 OOS 类请求，明确告知用户当前系统能处理什么范围，引导到正确入口，而不是乱猜一个意图强行回答

4. **错误反馈闭环**：用户投诉或评分差的 case 中，有很大比例是意图识别错误导致的。建立 bad case 收集 → 分析意图路由 → 修复 → 回归的闭环机制
