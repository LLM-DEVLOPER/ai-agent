# AI 系统设计面试知识点（完全版）

这份文档从"设计题怎么开场"一路讲到"企业知识库、客服 Agent、DeepResearch、成本控制、评测闭环"，每个环节都有具体技术选型理由和工程判断可以展开。

---

## 1. AI 系统设计题到底在考什么

**面试官可能会问：** AI 系统设计题和传统系统设计题有什么本质区别？面试官在看什么？

**回答要点：**

传统系统设计题考的是分布式、存储、一致性、扩展性这些工程基础；AI 系统设计题在这些之上，还要考你对模型能力边界的判断、对效果/成本/时延三角的权衡，以及对"系统怎么迭代"的理解。

最大的区别在于 AI 系统有一个传统系统没有的变量：**模型本身是不确定的**。你不能假设模型每次都给出正确答案，所以系统设计里必须考虑兜底、降级、评测和 bad case 闭环。面试官想看的不是你能画出多复杂的架构图，而是你能不能在约束条件下做出合理的取舍，并且说清楚为什么。

**面试官评分的三个维度：**

1. **需求澄清能力**：能不能在动手之前把约束问清楚，而不是上来就画图
2. **架构取舍能力**：能不能说清楚为什么选这个方案，而不是其他方案
3. **故障预防意识**：能不能主动提到系统的薄弱点和兜底策略

**深入追问：** 为什么很多候选人的 AI 系统设计答案听起来都差不多？

**回答要点：**

因为大家都在背"RAG + 向量数据库 + LLM"这个模板，却没有真正回答"为什么这个业务需要这个方案"。好的系统设计答案一定是从业务约束出发的，不同的用户规模、不同的准确率要求、不同的成本预算，会导出完全不同的架构决策。模板只是起点，不是答案。

## 2. 设计题应该怎么开场

**面试官可能会问：** 拿到一道系统设计题，你会怎么开始？

**回答要点：**

不要一上来就画架构图。正确的开场是先问清楚约束和目标，因为同样是"设计一个企业知识库问答系统"，面向 100 人内部团队和面向 10 万用户的 SaaS 产品，架构会完全不同。

开场要问的核心问题：

| 维度 | 要问的问题 | 影响的设计决策 |
|------|----------|--------------|
| **用户规模** | 日活用户数、并发请求峰值 | 是否需要分布式、缓存策略 |
| **数据规模** | 文档量、更新频率 | 向量库选型、索引策略 |
| **准确率要求** | 可接受的错误率是多少 | 是否需要 Rerank、人工审核 |
| **时延要求** | 用户能接受等待多久 | 模型选型、是否流式输出 |
| **成本约束** | 每次查询的成本上限 | 模型分级、缓存 |
| **安全合规** | 数据能否出域、有无权限隔离 | 部署方式、权限架构 |
| **现有系统** | 从零开始还是扩展已有系统 | 集成复杂度 |

把这些问清楚之后，再开始讲架构。这个过程本身就是在向面试官展示你的工程判断力——你知道架构是由约束决定的，而不是由技术偏好决定的。

**深入追问：** 如果面试官说"你来定约束"，怎么处理？

**回答要点：**

这是面试官在考你能不能自己建立合理假设。要主动给出一套合理的假设，例如："我假设这是一个面向企业内部的知识库，文档量在 10 万级别，日活用户 1000 人，对准确率要求高（错误率 < 5%），对时延要求中等（3~5 秒可接受），数据不能出域，需要部门级权限隔离。"

然后基于这套假设展开设计，并在结尾说"如果规模放大到 100 万文档或 10 万用户，以下几点架构需要调整..."。假设要合理、要说出来、要在后续设计里保持一致。

## 3. 设计一个企业知识库问答系统

**面试官可能会问：** 请设计一个企业内部知识库问答系统，支持员工用自然语言查询公司文档。

**回答要点：**

### 3.1 整体架构

核心链路是 RAG 流程，但企业场景有多个关键约束使架构比标准 RAG 复杂：

```
文档处理管道（离线）：
原始文档 → 解析（PDF/Word/HTML）→ 分块 → Embedding → 向量库 + 权限标签

查询链路（在线）：
用户查询 → 意图识别 → 混合检索（向量 + BM25）→ 权限过滤 → Rerank → 生成 → 输出
```

### 3.2 文档处理管道

**解析层**：不同格式文档用不同解析器——PDF 用 PDFMiner/Unstructured，表格用专门的表格解析，图片内容走 OCR。解析质量直接决定检索上限，是最容易被低估的环节。

**分块策略**：核心问题是块大小和重叠的权衡：
- 块太大：语义噪声多，检索精度低
- 块太小：上下文丢失，生成质量差
- 经验值：512~1024 tokens/块，10~15% 的滑动窗口重叠
- 更好的做法：**语义分块**（按段落/章节边界分，而不是固定字符数），保持语义完整性

**Embedding 选型**：
- 中文场景：BGE-M3（字节跳动）、BCE（网易有道）效果较好
- 多语言：text-embedding-3-large（OpenAI）
- 关键：Embedding 模型要和实际数据分布匹配，最好在领域数据上 fine-tune

### 3.3 多租户权限隔离

企业知识库最关键的约束。**权限隔离必须在检索层就做，不能等到生成前再补**。具体做法：

1. 在向量数据库里给每个 chunk 打上 `tenant_id`、`department`、`access_level` 等元数据标签
2. 检索时先用 metadata filter 过滤权限，再做语义检索：

```python
results = vector_db.search(
    query_embedding=query_emb,
    filter={"tenant_id": user.tenant_id, "department": {"$in": user.accessible_depts}},
    top_k=20
)
```

3. 向量库选型：Weaviate、Qdrant、Milvus 都支持 metadata filter；Pinecone 的 filter 性能相对弱一些

**如果等到生成阶段再过滤，一旦模型把无权限内容"看到了"，即使最终没输出，也已经存在信息泄露风险。**

### 3.4 混合检索与 Rerank

纯向量检索的弱点：对精确关键词（产品型号、人名、专有名词）效果差。生产系统标配：

**混合检索（Hybrid Search）**：向量检索 + BM25 关键词检索，用 Reciprocal Rank Fusion（RRF）合并排名：

$$\text{RRF\_score}(d) = \sum_{i} \frac{1}{k + \text{rank}_i(d)}$$

其中 $k=60$ 是平滑常数，$\text{rank}_i(d)$ 是文档 $d$ 在第 $i$ 个检索系统中的排名。

**Rerank**：用 Cross-Encoder 模型对候选文档做精确的（query, document）相关性打分，比双塔 Embedding 精度高但速度慢。通常检索 top-20~50，Rerank 后取 top-3~5 送生成。

常用 Rerank 模型：bge-reranker-v2-m3、Cohere Rerank、cross-encoder/ms-marco 系列。

### 3.5 增量索引

文档更新时的关键设计：**必须先删旧版本的所有 chunk，再加新 chunk**，否则向量库里同时存在新旧版本，检索时可能召回过期内容。

实现方式：维护 `document_id → [chunk_id_1, chunk_id_2, ...]` 的映射，更新时：
1. 按 document_id 查出所有旧 chunk_id
2. 批量删除旧 chunk
3. 重新解析、分块、Embedding，写入新 chunk

**深入追问：** 如果文档量很大，索引构建和更新的延迟怎么处理？

**回答要点：**

索引构建要异步化，文档上传后先返回"处理中"状态，后台异步做解析、分块、Embedding 和入库。三个优化方向：

1. **Embedding 批处理**：单条 Embedding 请求 overhead 大，批量处理（batch_size=32~64）能显著提升吞吐
2. **优先级队列**：频繁访问的文档优先更新索引，长尾文档可以延迟处理
3. **增量而非全量**：只对变更的段落重新 Embedding，不是整文档重做

### 3.6 多模态扩展：图文混合知识库

**面试官可能会问：** 如果知识库里有大量图片、图表、截图，怎么处理？

**回答要点：**

企业知识库里的图片通常有三类：**产品图、架构图/流程图、截图/图表**。处理策略不同：

**方案1：伪多模态（OCR + 图像描述）**

```
图片 → OCR提取文字 → 视觉模型生成caption → 合并为文本 → 标准RAG流程
```

- **优点**：成本低、检索效果可控、工程简单
- **缺点**：丢失视觉信息（颜色、布局、图形语义）
- **适用**：图片主要是文字内容（截图、扫描文档）

**方案2：原生多模态（CLIP + 多模态LLM）**

```
图片 → CLIP编码 → 向量库（图文统一空间）
查询 → 文本编码 → 跨模态检索 → 召回图片 → 多模态LLM生成答案
```

- **优点**：保留完整视觉信息，支持"找相似图"
- **缺点**：成本高（图片token消耗大）、时延高
- **适用**：图片本身是答案（产品图、设计稿）

**方案3：混合方案（推荐）**

```
- 文字类图片（截图、文档）→ OCR转文本 → 文本检索
- 图形类图片（架构图、流程图）→ 保留原图 + caption → 图文联合检索
- 检索时：文本检索召回文档，图片作为附件一起返回
- 生成时：把原图送给多模态LLM理解
```

**关键设计点**：

1. **图片元数据管理**：每张图片记录来源文档、页码、类型（截图/图表/产品图）
2. **成本控制**：图片按需加载，不是所有图都送LLM；高分辨率图压缩后再送模型
3. **缓存策略**：相同图片的OCR结果和caption缓存复用

### 3.7 复杂表格理解与跨表格推理

**面试官可能会问：** 如果知识库里有大量财务报表、数据表格，用户问"过去三年营收增长率"这类跨表格问题，怎么处理？

**回答要点：**

表格理解是RAG系统的难点，因为**表格的语义在结构里，而不只是文字里**。

**核心挑战**：

1. **结构丢失**：直接把表格转成文本，行列关系会丢失
2. **跨表推理**：答案需要从多个表格提取数据并计算
3. **精确数值**：视觉模型对小字和精确数字识别不如结构化解析

**解决方案（分层处理）**：

**第1层：表格解析与结构化存储**

```
PDF/Excel → 表格检测 → 结构化解析 → 存储为JSON/DataFrame
```

- 工具：Azure Document Intelligence、Camelot（PDF表格提取）、Unstructured
- 关键：保留表头、行列关系、单元格合并信息

**第2层：表格表示（三路并行）**

```
1. 结构化数据 → 存入关系数据库（支持SQL查询）
2. 表格摘要 → "这是2023年Q1财报，包含营收、成本、利润三列" → 文本检索
3. 表格Embedding → 用专门的表格Embedding模型（如TableBERT）→ 向量检索
```

**第3层：查询路由**

```
用户问题 → 意图识别
  ├─ 简单查询（"2023年营收是多少"）→ SQL查询
  ├─ 跨表计算（"过去三年增长率"）→ 检索相关表 → 用代码解释器计算
  └─ 开放问答（"财务状况如何"）→ RAG生成
```

**第4层：代码解释器（关键）**

对于需要计算的问题，让LLM生成Python代码，在沙箱里执行：

```python
# LLM生成的代码
revenue_2021 = df_2021['营收'].sum()
revenue_2023 = df_2023['营收'].sum()
growth_rate = (revenue_2023 - revenue_2021) / revenue_2021 * 100
```

- 工具：E2B、Modal、自建沙箱
- 安全：代码沙箱隔离、禁止网络访问、超时限制

**工程难点**：

1. **表格切分**：大表格怎么分块？按行切会破坏表头，按列切会破坏行关系
   - 解决：保留表头 + 滑动窗口切分，每个chunk都带完整表头
2. **跨表关联**：怎么知道哪些表需要一起看？
   - 解决：表格元数据（时间、主题、来源文档）+ 语义检索
3. **数值精度**：LLM算数不可靠
   - 解决：所有计算都用代码解释器，不让LLM直接算

**面试一句话总结**：表格理解要结构化存储 + SQL查询 + 代码解释器，不能只靠RAG；跨表推理的关键是先检索相关表，再用代码计算，而不是让LLM直接推理数字。

## 4. 设计一个智能客服 Agent

**面试官可能会问：** 请设计一个智能客服系统，能处理用户的售后咨询、退款申请和投诉。

**回答要点：**

### 4.1 整体架构

```
用户输入 → 意图分类 → 分流路由
              ├─ FAQ 类 → 直接检索返回
              ├─ 查询类 → 工具调用（查订单/物流）→ 生成回答
              ├─ 操作类 → 工具调用 + 二次确认 + 执行
              └─ 投诉类 → 情绪检测 → 人机转接判断
```

### 4.2 意图识别与分流

意图识别是第一层分流，质量直接决定后续链路效率。要用分类模型而不是纯 LLM 来做，原因是速度（< 100ms）和稳定性。

意图类型设计：
- **信息查询类**：订单状态、物流信息、商品规格 → 工具调用
- **FAQ 类**：退货政策、会员规则、通用问题 → RAG 检索
- **操作类**：申请退款、取消订单、修改地址 → 工具调用 + 审核
- **投诉类**：强烈不满、纠纷 → 快速识别，优先处理或转人工
- **闲聊类**：无需工具，直接回复或引导回主题

### 4.3 高风险操作的安全设计

退款、取消订单这类有副作用的操作，不能让模型直接执行。**分三层防护**：

1. **参数约束层**：退款金额从订单系统查询，不由模型计算；工具接口做参数范围校验（金额不能超过订单金额）

2. **二次确认层**：所有涉及金额的操作，执行前向用户展示确认："将为您退款 ¥99.00 到原支付渠道，确认吗？"

3. **人工审核层**：超过金额阈值（如 ¥1000）、特殊退款原因（质量纠纷）→ 自动转人工审核队列

```python
def process_refund(order_id, reason):
    order = get_order(order_id)  # 从系统查，不用模型算
    if order.refund_amount > THRESHOLD:
        return escalate_to_human(order, reason)  # 超阈值转人工
    # 向用户展示确认，等待确认后执行
    return await_user_confirmation(order.refund_amount)
```

### 4.4 人机转接

触发条件：
- 用户明确要求转人工
- 情绪识别到强烈不满（负面词汇、感叹号密集、重复投诉）
- Agent 连续 2 次无法解决同一问题
- 涉及法律纠纷或高金额纠纷
- 工单类型超出 Agent 授权范围

**转接时要把完整的对话上下文传给人工客服**（包括用户信息、对话历史、已尝试的解决方案、失败原因），不能让用户重复描述问题，这是客服系统用户体验的核心。

**深入追问：** 如何衡量客服 Agent 的效果？

**回答要点：**

评测要分两层：

**任务完成率（Task Completion Rate）**：用户的问题最终有没有被解决。这是最核心的指标，但也最难测——需要用户主动反馈或人工抽检。

**分层指标**：

| 指标 | 定义 | 目标 |
|------|------|------|
| 意图识别准确率 | 意图分类是否正确 | > 95% |
| 工具调用准确率 | 工具参数是否正确 | > 98% |
| 一次解决率（FCR） | 单次对话完成解决 | > 70% |
| 转人工率 | 需要人工介入的比例 | < 20% |
| 平均处理时长 | 从用户发起到问题关闭 | 基线对比 |

## 5. 设计一个 DeepResearch 类系统

**面试官可能会问：** 请设计一个能够自主搜索、阅读文献、生成研究报告的 AI 系统（类似 Perplexity DeepResearch 或 OpenAI DeepResearch）。

**回答要点：**

### 5.1 核心链路

DeepResearch 类系统的本质是一个**多轮信息收集 + 综合生成**的 Agent，链路比普通 RAG 复杂得多：

```
用户问题 → 问题分解（子问题列表）
    ↓
并行搜索子问题（每个子问题独立 ReAct 循环）
    ↓
多源信息汇聚与去重
    ↓
分层摘要（长文档 → 段落摘要 → 全文综合）
    ↓
报告生成（结构化输出 + 引用标注）
```

### 5.2 问题分解（Query Decomposition）

用户的研究问题通常是高层次的（"分析大模型在医疗领域的应用现状"），需要先分解成可搜索的子问题：

```
原始问题: 大模型在医疗领域的应用现状
分解为:
  1. 目前有哪些大模型专门针对医疗领域做了微调？
  2. 大模型在医学影像诊断中的表现如何？
  3. 大模型在临床决策支持中有哪些实际落地案例？
  4. 医疗大模型面临哪些监管和伦理挑战？
```

分解的好坏直接决定最终报告的覆盖面和深度。

### 5.3 并行搜索与信息汇聚

**并行搜索**：每个子问题独立启动搜索 + 阅读循环（ReAct），多个子问题并行执行，显著缩短总耗时。

**信息源管理**：
- Web 搜索（Bing/Google API）
- 学术搜索（arXiv、PubMed）
- 专业数据库（按领域配置）

**去重与聚合**：多个子问题可能检索到相同的来源，需要：
1. URL 去重（完全相同的来源只处理一次）
2. 语义去重（内容高度相似的段落合并）
3. 来源可信度评分（官方网站 > 学术论文 > 新闻 > 博客）

### 5.4 长文档处理：Map-Reduce 摘要

检索到的文档通常很长，直接放入上下文会超出 Token 限制。解决方案是**分层摘要（Hierarchical Summarization）**：

```
Map 阶段: 对每个文档段落独立生成摘要（并行）
Reduce 阶段: 对所有段落摘要再次综合，生成文档级摘要
最终: 对所有文档摘要生成最终报告
```

这是 LangChain 的 `MapReduceDocumentsChain` 的核心思路，也可以用 Refine（逐步精炼）策略：先处理第一个文档生成初稿，再用后续文档不断修正。

### 5.5 引用标注与溯源

研究报告的可信度来自引用。每个事实性陈述都要能追溯到来源：

- 在生成时要求模型输出 `[来源 ID]` 格式的引用标记
- 维护一个来源列表，报告末尾附参考文献
- 支持用户点击引用直接跳转到原文

**深入追问：** DeepResearch 系统最容易出现什么问题？

**回答要点：**

三个主要问题：

1. **信息污染（Hallucination from Sources）**：检索到的内容本身就是错的（错误信息、过期数据），模型把它当成事实引用。缓解：多源交叉验证，对关键事实要求多个独立来源都支持

2. **信息老化（Staleness）**：动态领域（如 AI 技术进展）的研究，搜索到的可能已经是 6 个月前的内容。缓解：在搜索时加时间过滤，优先最近 N 个月的内容

3. **覆盖偏差（Coverage Bias）**：分解的子问题没有覆盖某个重要角度，导致报告有盲区。缓解：生成报告后用另一个 LLM 对报告进行"批判性审查"，指出可能缺失的维度

## 6. 设计一个通用任务型 Agent

**面试官可能会问：** 请设计一个通用任务型 Agent，能接受用户的自然语言任务并自主完成。

**回答要点：**

### 6.1 核心模块

通用任务型 Agent 的五层模块：

```
Planner（规划）: 任务分解 → 执行计划 → 步骤依赖图
Executor（执行）: ReAct 循环，工具调用，Observation 处理
Memory（记忆）: 短期（当前任务状态）+ 长期（用户偏好、历史）
State（状态）: 任务目标、当前步骤、已完成步骤、工具调用历史
Evaluator（评估）: 判断任务是否完成，触发 Reflection
```

### 6.2 安全边界设计

通用 Agent 最危险的点：**动作空间无边界**。要从设计时就加入约束：

1. **工具白名单**：不同任务类型只暴露对应的工具子集，不要把所有工具都暴露
2. **操作分级**：读操作可以直接执行，写操作需要确认，不可逆操作（删除、发送）必须二次确认
3. **成本上限**：设置 API 调用次数上限、Token 消耗上限、时间上限
4. **最大步数**：硬性限制最大执行步数（如 20 步），超过就强制终止并汇报进度

### 6.3 State 设计

State 是整个系统的"现场记录"，需要支持：
- **回放（Replay）**：出问题时能还原执行过程
- **断点续跑（Resume）**：长任务中途中断可以从上次状态继续
- **并发保护**：多步操作时防止状态冲突

```python
class AgentState:
    task_id: str
    goal: str
    plan: List[Step]         # 规划的步骤列表
    current_step: int
    completed_steps: List[StepResult]
    tool_call_history: List[ToolCall]
    context_window: List[Message]  # 当前轮次的上下文
    metadata: Dict            # 成本计数、步数计数等
```

**深入追问：** 通用 Agent 最容易失控的地方是哪里？

**回答要点：**

最容易失控的有两类：

**目标漂移（Goal Drift）**：执行多步后，Agent 逐渐偏离了原始目标，去做了"看起来相关但不是用户要求的"事情。解决：在每个 ReAct 循环开始时，把原始 Goal 重新注入 Prompt，作为锚点。

**工具滥用（Tool Overuse）**：模型把能用工具的事情都用工具做，即使不必要。解决：在 Prompt 里明确"只有在必须获取外部信息时才使用工具，能直接回答的就直接回答"。

### 6.4 实例：设计一个"帮我订机票"的Agent

**面试官可能会问：** 用一个具体例子说明通用Agent的五层模块是怎么协作的。

**回答要点（用"帮我订3月15日北京到上海的机票"为例）**：

**Step 1: Planner（规划）**

用户输入："帮我订3月15日北京到上海的机票"

Planner分解任务：
```
1. 查询3月15日北京到上海的航班
2. 根据用户偏好（时间、价格、航司）筛选候选航班
3. 向用户展示候选航班并确认
4. 调用订票API完成预订
5. 发送确认信息给用户
```

依赖关系：1 → 2 → 3 → 4 → 5（严格串行）

**Step 2: Executor（执行第1步）**

ReAct循环：
```
Thought: 需要查询航班信息，调用航班查询工具
Action: search_flights(departure="北京", arrival="上海", date="2026-03-15")
Observation: 返回12个航班，包含时间、价格、航司信息
```

**Step 3: Memory（记忆调用）**

从长期记忆中读取用户偏好：
```
- 偏好时间段：早上8-10点或下午2-4点
- 价格敏感度：中等（不选最便宜也不选最贵）
- 偏好航司：国航、东航
- 历史：上次订过早上9点的航班，体验好
```

**Step 4: Executor（执行第2步）**

```
Thought: 根据用户偏好筛选
Action: filter_flights(flights=观察到的12个航班, preferences=用户偏好)
Observation: 筛选出3个候选航班
  - CA1234 08:30-10:45 ¥850 国航
  - MU5678 09:15-11:30 ¥780 东航
  - CA9012 14:00-16:15 ¥920 国航
```

**Step 5: State（状态更新）**

```python
state = {
    "task_id": "book_flight_001",
    "goal": "订3月15日北京到上海的机票",
    "plan": [查询航班, 筛选, 确认, 订票, 通知],
    "current_step": 2,  # 正在执行"向用户确认"
    "completed_steps": [
        {"step": "查询航班", "result": "12个航班"},
        {"step": "筛选", "result": "3个候选"}
    ],
    "tool_call_history": [
        {"tool": "search_flights", "params": {...}, "result": {...}},
        {"tool": "filter_flights", "params": {...}, "result": {...}}
    ],
    "context": {
        "candidate_flights": [CA1234, MU5678, CA9012],
        "user_preferences": {...}
    }
}
```

**Step 6: Executor（执行第3步 - 用户确认）**

```
Thought: 需要用户选择航班
Action: ask_user_confirmation(
    message="为您找到3个航班，推荐CA1234（早上8:30，国航，¥850），是否预订？",
    options=["预订CA1234", "看其他选项", "取消"]
)
Observation: 用户选择"预订CA1234"
```

**Step 7: Evaluator（评估是否继续）**

```
判断：用户已确认，可以执行订票
风险检查：
  - 价格是否在预算内？✓
  - 是否需要二次确认（高价票）？✗（¥850不算高价）
  - 用户账户余额是否足够？✓
决策：继续执行订票
```

**Step 8: Executor（执行第4步 - 订票）**

```
Thought: 调用订票API
Action: book_flight(
    flight_id="CA1234",
    passenger=从用户信息获取,
    payment_method=从用户偏好获取
)
Observation: 订票成功，订单号 ORDER123456
```

**Step 9: Evaluator（任务完成判断）**

```
检查：
  - 原始目标"订机票"是否完成？✓
  - 所有计划步骤是否执行完？还差最后一步"通知用户"
决策：执行最后一步
```

**Step 10: Executor（执行第5步 - 通知）**

```
Action: send_confirmation(
    order_id="ORDER123456",
    message="已为您预订3月15日CA1234航班（08:30-10:45），订单号ORDER123456"
)
Observation: 通知发送成功
```

**Step 11: Evaluator（最终评估）**

```
任务完成 ✓
总耗时：8秒
工具调用：5次
成本：$0.03
用户满意度：待反馈
```

**面试一句话总结**：通用Agent的五层模块通过State串联——Planner定计划，Executor执行并更新State，Memory提供上下文，Evaluator判断进度，整个过程State记录完整现场，支持回放和断点续跑。

## 7. 成本、时延和效果的三角权衡

**面试官可能会问：** 效果、成本、时延这三个目标经常冲突，你怎么做权衡？

**回答要点：**

效果、成本、时延是 AI 系统的三角约束，不可能同时最优。

**按业务场景定优先级**：

| 场景 | 优先级排序 | 典型配置 |
|------|----------|---------|
| 实时客服 | 时延 > 效果 > 成本 | 小模型 + 流式输出 |
| 企业知识库 | 效果 > 时延 > 成本 | 大模型 + Rerank |
| 批量数据处理 | 成本 > 效果 > 时延 | 量化模型 + 批处理 |
| 医疗/法律辅助 | 效果 >> 时延 ≈ 成本 | 最强模型 + 人工审核 |

**常见优化手段**：

1. **模型分级路由（Model Routing）**：用小模型（如 GPT-4o-mini）判断问题复杂度，简单问题用小模型直接回答，复杂问题才路由到大模型。可以节省 60%~80% 的 API 成本

2. **语义缓存（Semantic Cache）**：不只缓存完全相同的 query，而是缓存语义相似的 query 的结果。用向量相似度判断是否命中缓存（相似度 > 0.95 则命中）。GPTCache 是这类场景的开源实现

3. **流式输出（Streaming）**：首 token 延迟（TTFT）通常远小于完整响应时间，流式输出能显著改善用户感知时延

4. **降级策略**：大模型超时（> 10s）→ 降级到小模型；RAG 检索失败 → 降级到关键词搜索；整个 AI 链路故障 → 降级到静态 FAQ

**深入追问：** 如果老板说"效果要最好，成本要最低，时延要最短"，怎么回应？

**回答要点：**

这三个目标不能同时最优。我会先问清楚业务场景的核心指标：
1. 什么是"效果足够好"的底线（准确率达到 X%？用户满意度 > Y？）
2. 成本预算上限是多少（每次请求不超过 N 分钱？）
3. 时延的 P99 要求是多少（99% 的请求在 T 秒内响应）

有了这三个约束，才能给出"在这些约束下的最优方案"。这样的回答比"我会尽量平衡"更有价值，也更能体现工程判断力。

## 8. 评测体系与稳定性保障

**面试官可能会问：** AI 系统上线后怎么保证稳定性？评测、日志、限流、降级分别怎么做？

**回答要点：**

AI 系统的稳定性保障比传统系统多一层挑战：模型输出是不确定的，所以不能只靠单元测试，必须有持续的评测闭环。

### 8.1 全链路 Trace

每次请求必须记录完整链路：

```
trace = {
    "request_id": ...,
    "user_input": ...,
    "retrieved_docs": [...],    # 检索到的文档列表
    "reranked_docs": [...],     # Rerank 后的文档
    "prompt": ...,              # 送入模型的完整 Prompt
    "model_output": ...,        # 模型原始输出
    "final_response": ...,      # 最终展示给用户的内容
    "tool_calls": [...],        # 工具调用记录
    "latency_breakdown": {...}, # 各阶段耗时
    "cost": {...}               # Token 用量和费用
}
```

这些数据是后续排障、评测和优化的基础。缺了任何一层，出问题后根本没法定位。

### 8.2 评测体系

**离线评测**：维护 Golden Set（标注好的测试集），每次模型或 Prompt 变更后都跑一遍，对比指标变化。

RAG 系统的评测指标（RAGAS 框架）：

| 指标 | 定义 | 衡量的是什么 |
|------|------|------------|
| **Faithfulness（忠实度）** | 答案是否完全基于检索到的上下文 | 是否有幻觉 |
| **Answer Relevancy（相关性）** | 答案是否回答了用户的问题 | 有没有跑题 |
| **Context Precision（上下文精度）** | 检索到的文档有多少是真正有用的 | 检索精确率 |
| **Context Recall（上下文召回）** | 回答问题所需的信息有没有被检索到 | 检索召回率 |

**在线评测**：
- 用户显式反馈（点赞/踩）
- 隐式反馈（对话是否继续、是否重复提问）
- 人工抽检（每天抽 N% 样本做人工评分）

### 8.3 Bad Case 闭环

```
线上收集 → 分类归因 → 修复 → 回归测试 → 上线验证 → 加入 Golden Set
```

关键：**把 bad case 变成测试用例**，而不是修完就忘。otherwise，同类问题会反复出现。

分类归因要按链路分层：
- 检索层问题（召回的文档不相关）→ 优化 Embedding/Rerank
- 生成层问题（给了正确文档但答案错）→ 优化 Prompt 或换模型
- 数据层问题（知识库里本来就没有这个信息）→ 补充知识库
- 分块问题（关键信息被切断了）→ 调整分块策略

### 8.4 限流与降级

```
限流: 按用户/IP/租户限制请求频率，防止滥用打垮服务
     关键接口（如大模型调用）单独限流

降级策略（按优先级）:
  Level 1: 大模型超时 → 降级到小模型（保效果牺牲质量）
  Level 2: RAG 检索失败 → 降级到关键词搜索
  Level 3: 向量库故障 → 降级到纯生成（无检索）
  Level 4: AI 链路全故障 → 降级到静态 FAQ（保可用性）
```

**深入追问：** 如果系统上线后发现准确率比预期低很多，怎么排查？

**回答要点：**

按链路分层排查，**不要一上来就调 Prompt**：

1. **数据层**：知识库里有没有这个问题的答案？文档质量怎么样？
2. **检索层**：召回的文档是不是相关的？召回率和精确率分别是多少？可以用人工标注的相关文档集来测
3. **Rerank 层**：Rerank 后 top-k 里有没有正确答案？如果有，说明问题在生成层
4. **生成层**：给定正确的检索结果，模型能不能生成正确答案？可以人工注入正确文档来隔离测试
5. **评测层**：测试集是不是有代表性？评测指标是不是真的反映了用户体验？

## 9. 系统设计高频追问 Q&A

### Q1: 向量数据库怎么选？

**A:**

| 数据库 | 适用场景 | 核心优势 | 局限 |
|--------|---------|---------|------|
| **Qdrant** | 生产级部署、复杂 filter | Rust 实现，高性能；filter 灵活 | 社区相对小 |
| **Weaviate** | 多模态、混合搜索 | 内置 BM25 混合搜索；GraphQL 接口 | 资源占用较高 |
| **Milvus** | 超大规模（亿级向量） | 分布式，支持超大规模 | 运维复杂 |
| **Chroma** | 本地开发、快速原型 | 极简部署，Python 友好 | 不适合生产 |
| **Pinecone** | 快速上线、托管服务 | 零运维，易用 | 贵，filter 性能弱 |

选型原则：先看规模（千万向量以下大多数方案都够），再看 filter 需求（企业场景通常需要复杂权限 filter），再看运维成本（有 DevOps 能力用自托管，否则考虑托管服务）。

---

### Q2: 什么时候用流式输出，什么时候不用？

**A:**

**用流式（Streaming）**：
- 用户直接看到生成过程（聊天界面、报告生成）
- 响应较长（> 500 tokens），全部等完再展示用户体验差
- 前端可以实时渲染内容

**不用流式**：
- 需要对完整输出做后处理（JSON 解析、结构化提取）
- 输出很短（< 100 tokens），流式收益不大
- 需要对完整输出做校验再展示（安全过滤）

---

### Q3: 如何控制 AI 系统的成本？

**A:**

成本控制的四个维度：

1. **Token 层**：精简 System Prompt，去掉冗余的工具描述；检索结果做摘要再喂入，不要原文全量放入；用 Prompt Caching（固定前缀命中缓存）
2. **模型层**：简单任务用小模型；批量任务选 Batch API（通常有 50% 折扣）
3. **缓存层**：语义缓存高频相似问题；对相同文档的多次检索缓存 Embedding
4. **架构层**：能用规则做的不用模型；意图分类用小模型，只有复杂推理用大模型

成本监控：按接口、按用户、按功能模块分维度统计 Token 消耗，发现异常快速定位。

---

### Q4: 如果要支持 100 万文档的知识库，架构需要哪些调整？

**A:**

规模从 10 万到 100 万，以下几个点需要调整：

1. **向量库从单机到分布式**：Milvus 集群，按 collection 或 shard 做水平扩展
2. **Embedding 计算**：异步批量 Embedding 管道，用 GPU 加速，不能依赖实时计算
3. **检索层加缓存**：高频查询的检索结果做 Redis 缓存，降低向量库压力
4. **分层存储**：热数据（最近 30 天更新或高频访问）放高性能向量库，冷数据放低成本存储
5. **索引更新队列**：大规模增量更新用消息队列（Kafka）做削峰，保证更新吞吐

---

### Q5: 系统设计结尾应该说什么？

**A:**

好的设计结尾要主动展示"我知道这个方案的局限性"：

1. **当前方案的权衡**：我们选择 X 换取了 Y，代价是 Z
2. **规模扩展路径**：如果用户量 / 数据量增长 10 倍，以下哪些地方需要调整
3. **未解决的问题**：哪些技术挑战还没有完美答案（比如长期记忆的相关性衰减）
4. **监控和迭代计划**：上线后如何通过数据驱动持续优化

这种"我看到了更远的地方"的表达，往往比"这个方案完美无缺"更能打动面试官。
