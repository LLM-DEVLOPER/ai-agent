# 系统设计

## 1. AI 系统设计题到底在考什么

**面试官可能会问：** AI 系统设计题和传统系统设计题有什么本质区别？面试官在看什么？

**回答要点：**

传统系统设计题考的是分布式、存储、一致性、扩展性这些工程基础；AI 系统设计题在这些之上，还要考你对模型能力边界的判断、对效果/成本/时延三角的权衡，以及对"系统怎么迭代"的理解。

最大的区别在于 AI 系统有一个传统系统没有的变量：模型本身是不确定的。你不能假设模型每次都给出正确答案，所以系统设计里必须考虑兜底、降级、评测和 bad case 闭环。面试官想看的不是你能画出多复杂的架构图，而是你能不能在约束条件下做出合理的取舍，并且说清楚为什么。

**深入追问：** 为什么很多候选人的 AI 系统设计答案听起来都差不多？

**回答要点：**

因为大家都在背"RAG + 向量数据库 + LLM"这个模板，却没有真正回答"为什么这个业务需要这个方案"。好的系统设计答案一定是从业务约束出发的，不同的用户规模、不同的准确率要求、不同的成本预算，会导出完全不同的架构决策。模板只是起点，不是答案。

## 2. 设计题应该怎么开场

**面试官可能会问：** 拿到一道系统设计题，你会怎么开始？

**回答要点：**

不要一上来就画架构图。正确的开场是先问清楚约束和目标，因为同样是"设计一个企业知识库问答系统"，面向 100 人内部团队和面向 10 万用户的 SaaS 产品，架构会完全不同。

开场要问的核心问题包括：目标用户是谁（内部员工还是外部客户）、规模预估（日活、文档量、并发请求数）、成功指标是什么（准确率、召回率、用户满意度、任务完成率）、约束条件（成本预算、时延要求、数据安全合规要求）、现有系统是什么（是从零开始还是在已有系统上扩展）。

把这些问清楚之后，再开始讲架构。这个过程本身就是在向面试官展示你的工程判断力——你知道架构是由约束决定的，而不是由技术偏好决定的。

**深入追问：** 如果面试官说"你来定约束"，怎么处理？

**回答要点：**

这是面试官在考你能不能自己建立合理假设。这时候要主动给出一套合理的假设，比如"我假设这是一个面向企业内部的知识库，文档量在 10 万级别，日活用户 1000 人，对准确率要求高，对时延要求中等，数据不能出域"，然后基于这套假设展开设计。假设要合理、要说出来、要在后续设计里保持一致。

## 3. 设计一个企业知识库问答系统

**面试官可能会问：** 请设计一个企业内部知识库问答系统，支持员工用自然语言查询公司文档。

**回答要点：**

核心架构是 RAG 流程：文档解析 → 分块 → Embedding → 向量存储 → 检索 → 生成。但企业场景有几个关键约束，会让架构比标准 RAG 复杂很多。

第一个关键约束是多租户权限隔离。企业里不同部门的文档不能互相访问，权限隔离必须在检索层就做，不能等到生成前再补。具体做法是在向量数据库里给每个文档块打上租户 ID 和权限标签，检索时先过滤权限再做语义检索。如果等到生成阶段再过滤，一旦模型把无权限内容"看到了"，即使最终没输出，也已经存在信息泄露风险。

第二个关键约束是增量索引。企业文档会持续更新，文档更新时不能只加新内容，必须先删掉旧文档对应的所有 chunk，再加入新 chunk。否则向量库里会同时存在新旧版本，检索时可能召回过期内容。增量索引需要维护文档 ID 到 chunk ID 的映射关系，更新时按文档 ID 批量删除。

第三个关键约束是混合检索。纯向量检索对精确关键词（如产品型号、人名、专有名词）效果差，生产系统通常要做向量检索 + BM25 关键词检索的混合，再用 Rerank 模型做二次排序。

**深入追问：** 如果文档量很大，索引构建和更新的延迟怎么处理？

**回答要点：**

索引构建要做异步化，文档上传后先返回"处理中"状态，后台异步做解析、分块、Embedding 和入库。对于更新频繁的文档，可以做增量更新队列，按优先级处理。对于查询时文档还没索引完的情况，要有明确的提示，不能让系统静默地返回过期结果。

## 4. 设计一个客服 Agent

**面试官可能会问：** 请设计一个智能客服系统，能处理用户的售后咨询、退款申请和投诉。

**回答要点：**

客服 Agent 的核心链路是：意图识别 → FAQ 匹配 → RAG 检索 → 工具调用 → 人机转接。

意图识别是第一层分流。用户的问题先分类：是标准 FAQ（直接返回）、是需要查询订单状态（调工具）、是需要处理退款（调工具 + 审核）、还是复杂投诉（转人工）。这层分流做好，后面的链路才能高效。

工具调用层需要接入订单系统、退款系统、物流系统等后端服务。关键设计点是高风险操作的兜底：退款、取消订单这类有副作用的操作，不能让模型直接执行，必须加二次确认（向用户确认）或人工审核（超过金额阈值的退款走人工）。模型只负责理解意图和准备参数，最终执行要有明确的授权机制。

人机转接是客服系统的必要兜底。触发条件包括：用户明确要求转人工、情绪识别到用户强烈不满、Agent 连续两次无法解决问题、涉及法律纠纷或高金额投诉。转接时要把完整的对话上下文传给人工客服，不能让用户重复描述问题。

**深入追问：** 如果 Agent 给出了错误的退款金额，怎么防止这类事故？

**回答要点：**

防止这类事故要从两层入手。第一层是工具层的参数校验：退款金额必须从订单系统查询，不能由模型自己计算，工具接口要做金额范围校验。第二层是执行前的确认机制：所有涉及金额的操作，执行前必须向用户展示"将退款 X 元，确认吗"，用户确认后才执行。这两层加起来，即使模型理解出错，也有机会在执行前被发现和纠正。

## 5. 设计一个通用任务型 Agent

**面试官可能会问：** 请设计一个通用任务型 Agent，能接受用户的自然语言任务并自主完成。

**回答要点：**

通用任务型 Agent 的核心模块是 Planner、Tool、Memory、State 和 Evaluator，这五层缺一不可。

Planner 负责把用户的任务分解成可执行的子步骤。对于简单任务可以直接执行，对于复杂任务需要先做显式规划，生成一个步骤列表，再逐步执行。规划结果要存入 State，后续每步执行后更新进度。

Tool 层是 Agent 的执行能力来源。工具要做好 schema 设计（参数类型、范围、必填项），工具调用要有超时和重试机制，高风险工具（写操作、外部 API 调用）要有权限控制和调用频率限制。

Memory 分两层：短期记忆是当前任务的上下文和中间结果，存在 State 里；长期记忆是用户偏好和历史任务摘要，存在持久化存储里，按需检索。

State 是整个系统的现场，记录任务目标、当前步骤、已完成步骤、工具调用历史和 Observation。State 设计要支持回放和断点续跑。

Evaluator 负责判断任务是否完成。不能只靠模型自己说"完成了"，要有可验证的完成条件，比如工具返回了预期结果、用户确认了输出。

**深入追问：** 通用 Agent 最容易失控的地方是哪里？

**回答要点：**

最容易失控的是动作空间过大和 stop condition 不严。工具越多，模型越容易选错；stop condition 不严，Agent 会一直跑下去或者过早结束。解决方案是：按任务类型动态限制可用工具集合，不要把所有工具都暴露给每个任务；显式设置最大步数和最大成本限制，超过就强制终止并告知用户。

## 6. 成本、时延和效果怎么权衡

**面试官可能会问：** 效果、成本、时延这三个目标经常冲突，你怎么做权衡？

**回答要点：**

效果、成本、时延是 AI 系统的三角约束，不可能同时最优，必须根据业务优先级做取舍。

不同业务的优先级完全不同。实时客服场景时延优先，用户等待超过 3 秒就会流失，这时可以接受效果略差，但必须控制时延；企业知识库问答效果优先，用户可以等 5-10 秒，但答案必须准确；批量数据处理成本优先，时延不敏感，但要控制每次调用的费用。

常见的权衡手段包括：模型分级路由（简单问题用小模型，复杂问题用大模型）；缓存（高频相似问题缓存结果）；异步处理（不需要实时返回的任务走异步队列）；降级策略（大模型超时时降级到规则系统或小模型）。

关键是要有明确的业务底线：什么指标不能低于什么值。有了底线，才能在底线之上做优化，而不是盲目追求单点最优。

**深入追问：** 如果老板说"效果要最好，成本要最低，时延要最短"，怎么回应？

**回答要点：**

这三个目标不能同时最优，必须明确优先级。我会先问清楚业务场景的核心指标是什么，再给出在约束条件下的最优方案。比如"在时延 3 秒以内、成本每次不超过 X 分钱的约束下，效果能做到 Y"，这样的回答比"我会尽量平衡"更有价值，也更能体现工程判断力。

## 7. 评测、稳定性与可落地性

**面试官可能会问：** AI 系统上线后怎么保证稳定性？评测、日志、限流、降级分别怎么做？

**回答要点：**

AI 系统的稳定性保障比传统系统多一层挑战：模型输出是不确定的，所以不能只靠单元测试，必须有持续的评测闭环。

日志和 trace 是基础。每次请求要记录：用户输入、检索结果、模型输入（完整 prompt）、模型输出、工具调用参数和结果、最终返回内容、耗时和成本。这些数据是后续排障和优化的基础，缺了任何一层都会让问题定位变得很难。

评测要做成持续流程，不是上线前做一次就完。要维护一个 golden set（标注好的测试集），每次模型或 prompt 变更后都跑一遍，对比指标变化。同时要有 bad case 收集机制，线上用户反馈的差评要能快速进入评测集。

限流和降级是稳定性的工程兜底。限流防止突发流量打垮模型服务；降级策略要提前设计好，比如大模型超时时降级到小模型、RAG 检索失败时降级到关键词搜索、整个 AI 链路故障时降级到静态 FAQ。

**深入追问：** bad case 闭环具体怎么做？

**回答要点：**

bad case 闭环的完整链路是：线上收集（用户差评、人工抽检、自动异常检测）→ 分类归因（是检索问题、生成问题还是工具问题）→ 修复（调整 prompt、优化检索、补充知识库）→ 回归测试（确认修复没有引入新问题）→ 上线验证。关键是要把 bad case 变成测试用例，而不是修完就忘，否则同类问题会反复出现。

## 8. 系统设计高频追问

**面试官可能会问：** 为什么不用更简单的方案？你最怕什么事故？

**回答要点：**

"为什么不用更简单的方案"是系统设计里最高频的追问，本质是在考你有没有真正评估过替代方案。好的回答结构是：先承认更简单的方案存在，再说清楚它在哪个约束上满足不了需求，最后说明当前方案的额外复杂度是值得的。比如"我们评估过只用关键词搜索，但在语义相似问题上召回率只有 40%，达不到业务要求，所以才引入向量检索"。

"最怕什么事故"考的是你对系统风险的认知。AI 系统最怕的事故通常有几类：权限隔离失效导致数据泄露（这是最严重的，可能有法律风险）；模型输出错误信息被用户当作事实（尤其是医疗、法律、金融场景）；成本失控（模型被滥用或出现循环调用）；关键工具调用失败导致任务卡死。

**深入追问：** 如果系统上线后发现准确率比预期低很多，怎么排查？

**回答要点：**

按链路分层排查。先看检索层：召回的文档是不是相关的，召回率和精确率分别是多少。再看生成层：给定正确的检索结果，模型能不能生成正确答案。再看数据层：知识库里有没有这个问题的答案，文档分块是不是合理。最后看评测层：测试集是不是有代表性，评测指标是不是真的反映了用户体验。很多准确率问题根因在检索层，不要一上来就调 prompt。
