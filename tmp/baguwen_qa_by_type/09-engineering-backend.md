# 工程化与后端

---

## 1. AI 应用后端的分层与服务边界

**面试官可能会问：** AI 应用后端应该怎么分层？各层的职责边界是什么？

**回答要点：**

一个完整的 AI 应用后端通常分五层。网关层负责鉴权、限流、路由和协议转换，是所有流量的入口，不应该包含任何业务逻辑。编排层是 AI 应用的核心，负责 Agent 的 Planner/Router/Executor 逻辑，包括意图识别、工具选择、多步任务编排和状态管理，这一层不应该和具体业务逻辑混在一起，否则会导致编排逻辑和业务逻辑耦合，难以复用和测试。能力层封装各类工具和服务调用，比如搜索、数据库查询、外部 API，对编排层暴露统一接口。数据层负责向量数据库、关系数据库、缓存等存储，以及 RAG 的检索逻辑。模型层是推理服务本身（vLLM、OpenAI API 等），只负责接收 prompt 返回 completion，不感知业务语义。

分层的核心价值是：每层可以独立扩展、独立替换、独立测试。比如换一个推理服务只需要改模型层，不影响编排层；编排逻辑变了不影响网关层的限流策略。

**深入追问：** 编排层和业务层混在一起会有什么问题？

**回答要点：**

最直接的问题是复用性差：同一个 Agent 编排逻辑（比如 ReAct 循环）如果和业务代码耦合，换一个业务场景就要重写。其次是可测试性差：编排逻辑依赖业务状态，单元测试很难 mock。另外是可观测性差：混在一起的代码很难单独追踪 Agent 的决策链路，出问题时排查困难。

## 2. 长任务、异步任务与任务编排

**面试官可能会问：** 为什么 AI 任务不能走同步接口？应该怎么设计？

**回答要点：**

AI 任务（尤其是多步 Agent、批量处理）不能走同步接口，原因有三：第一，执行时间不可预测，一个多步 Agent 可能需要几十秒甚至几分钟，HTTP 同步接口通常有 30-60 秒超时限制，直接超时；第二，资源占用问题，同步接口会占用一个线程/连接直到任务完成，高并发下服务器资源被耗尽；第三，失败重试困难，同步接口失败后客户端不知道任务执行到哪一步，重试可能导致重复执行。

正确的设计是任务队列 + 状态机 + 轮询/回调。客户端提交任务后立刻拿到一个 task_id，服务端把任务放入队列（Celery + Redis、BullMQ、或云服务的消息队列）异步执行。任务状态用状态机管理（pending → running → succeeded/failed），每个状态转换都持久化到数据库。客户端通过轮询接口查询状态，或者服务端任务完成后通过 webhook 回调通知客户端。

**深入追问：** 任务状态机应该设计哪些状态，失败了怎么处理？

**回答要点：**

基本状态包括：pending（已提交等待执行）、running（执行中）、succeeded（成功）、failed（失败）、cancelled（已取消）。对于多步 Agent，还可以加 paused（等待人工确认）状态。失败处理要区分可重试失败（网络超时、限流）和不可重试失败（输入违规、逻辑错误），可重试失败应该自动重试并记录重试次数，超过最大重试次数后转为 failed 并告警。重试时要注意幂等性，确保重试不会重复执行有副作用的操作。

**深入追问：** 为什么不能只靠 LLM 自己说"完成了"来判断任务结束？应该怎么验证？

**回答要点：**

LLM 容易过度自信，即使任务实际失败也可能输出"已完成"。可靠的任务完成验证需要多层机制：一是可验证的完成条件（工具返回了预期结果、输出通过格式校验、数据库写入成功），而不是只看模型的文本输出；二是用另一个 LLM 做独立验证（Judge 模式），对任务结果进行质量评估；三是关键任务加人工确认环节，高风险操作（退款、删除数据）必须有人工审核；四是结果一致性检查，对同一任务多次采样，如果结果差异很大说明任务不稳定。工程实践中，状态机的 succeeded 状态应该基于可验证的条件触发，而不是模型说"完成"就标记完成。

## 3. SSE、WebSocket 与流式交互链路

**面试官可能会问：** AI 流式输出为什么优先用 SSE 而不是 WebSocket？

**回答要点：**

SSE（Server-Sent Events）是单向推送，服务端 → 客户端，基于 HTTP 协议，天然支持断线重连（浏览器自动重连），实现简单，对基础设施友好（CDN、负载均衡、防火墙都支持 HTTP）。AI 流式输出的场景是服务端持续推送生成的 token，客户端只需要接收，不需要在流式过程中向服务端发消息，SSE 完全够用。

WebSocket 是双向通信，适合需要客户端主动发消息的场景，比如实时协作编辑、多人聊天、需要在生成过程中打断或修改的交互。WebSocket 的代价是：需要维护长连接状态，对负载均衡有要求（需要 sticky session 或共享状态），基础设施配置更复杂。

选型原则：如果只是"服务端推流给客户端"，用 SSE；如果需要"客户端在流式过程中也要发消息给服务端"，用 WebSocket。

**深入追问：** SSE 断线重连怎么处理，如何保证不丢消息？

**回答要点：**

SSE 协议本身支持 `id` 字段，每条消息带一个递增的 event id。客户端断线重连时，浏览器会在请求头里带上 `Last-Event-ID`，服务端从这个 id 之后继续推送，不丢消息。服务端需要把已发送的消息缓存一段时间（比如 Redis），以便断线重连时补发。对于 AI 生成场景，如果生成已经完成，重连时直接返回完整结果即可。

## 4. 幂等、重试与超时

**面试官可能会问：** 工具调用场景为什么特别需要幂等？怎么实现？

**回答要点：**

工具调用场景特别需要幂等，原因是工具往往有副作用：发邮件、扣款、写数据库、调用外部 API。如果工具调用因为网络超时或服务重启而重试，没有幂等保护就会重复执行副作用，比如发两封邮件、扣两次款。这在普通 API 场景里也存在，但 AI Agent 的多步执行和自动重试机制让这个问题更突出。

幂等实现的核心是幂等键（idempotency key）：每次工具调用生成一个唯一的 request_id，服务端在执行前先查这个 id 是否已经执行过，如果已执行直接返回上次的结果，不重复执行。幂等键可以存在 Redis（设置 TTL）或数据库里。对于 Agent 框架，工具调用的幂等键应该和 Agent 的执行步骤绑定，确保同一步骤的重试不会重复执行。

**深入追问：** 超时策略应该怎么设计，AI 调用链路上有哪些超时需要考虑？

**回答要点：**

AI 调用链路上有多层超时需要分别设置：客户端到网关的超时（通常 30-60 秒，对应用户等待上限）、网关到编排层的超时、编排层到推理服务的超时（要比模型实际生成时间长，否则会误超时）、工具调用的超时（每个工具单独设置，数据库查询和外部 API 超时不同）。超时要从外到内逐层收紧，确保内层超时先触发，外层有时间做降级处理，而不是外层先超时导致内层请求变成"孤儿请求"继续消耗资源。

## 5. 日志、Trace 与可观测性

**面试官可能会问：** AI 系统为什么特别需要 trace？普通日志不够用吗？

**回答要点：**

普通日志记录的是"发生了什么"，但 AI 系统出问题时需要回答的是"模型为什么做了这个决策"。一个 Agent 的执行链路可能是：用户输入 → 意图识别 → 工具选择 → 工具参数生成 → 工具执行 → 结果解析 → 下一步决策 → 最终回答。如果只有日志，你能看到每一步的输入输出，但无法把它们串联成一条完整的决策链路，也无法回放某次失败的执行过程。

AI 系统的 trace 需要记录的内容比普通系统更多：每次 LLM 调用的完整 prompt 和 completion（不能只记摘要）、工具选择的理由（模型输出的 reasoning）、工具调用的参数和返回值、每一步的耗时和 token 消耗、最终回答和中间状态的关联关系。这些信息串联起来才能回放决策过程，定位"模型在哪一步选错了工具"或"哪个工具返回了脏数据导致后续出错"。LangSmith、Langfuse、Arize Phoenix 等工具专门为 LLM 应用提供这种全链路 trace 能力，比自己用 OpenTelemetry 从头搭要省很多力气。

**深入追问：** trace 数据量很大，怎么控制存储成本？

**回答要点：**

几个策略：一是采样，对正常请求只采样一部分（比如 10%），对失败请求、慢请求、用户反馈差的请求全量保留；二是分级存储，热数据（最近 7 天）存快速存储，冷数据压缩归档；三是只存关键字段，完整 prompt 可以只存 hash，需要回放时再从原始日志重建；四是设置 TTL，大多数 trace 数据 30 天后价值很低，自动清理。

**深入追问：** 大模型应用里怎么做 A/B 测试？

**回答要点：**

AI 应用的 A/B 测试比传统应用更复杂，因为输出不确定性高，需要更大样本量才能得出统计显著结论。标准流程：流量分桶（用户 ID 或请求 ID 做 hash 分组，确保同一用户始终进同一组）→ 对照组/实验组各自的 Prompt/模型/检索策略 → 收集同维度指标（任务完成率、用户满意度、TTFT、token 消耗）→ 统计显著性检验（T-test 或 Mann-Whitney U test）。

关键注意点：样本量要足够（通常千级以上），否则结论不可靠；指标要多维度（不能只看一个指标，可能顾此失彼）；要控制变量（一次只改一个因素，否则无法归因）；长期效果和短期效果可能不同（新 Prompt 短期效果好但长期用户留存差）。实践中，对于高风险变更（换模型、大改 Prompt），先小流量灰度（5% 流量），观察 24 小时无异常再逐步放量。

## 6. 并发控制与资源治理

**面试官可能会问：** AI 后端的并发控制和普通后端有什么不同？

**回答要点：**

普通后端的并发控制主要是线程池和连接池，资源消耗相对均匀，QPS 限流基本够用。AI 后端的特殊性在于：单个请求的资源消耗差异极大（短 prompt 快速完成，长 prompt + 多工具调用可能占用 GPU 几十秒），而且资源消耗不只是 CPU/内存，还有 GPU 显存和推理服务的并发槽位。

资源治理需要多维度：并发请求数限制（防止推理服务 OOM）、token 预算控制（限制单请求的 max_tokens，防止一个请求占用过多 GPU 时间）、工具调用并发限制（防止 Agent 同时发起大量外部 API 调用打爆下游）、用户级配额（防止单用户占用过多资源影响其他用户）。背压（backpressure）机制也很重要：当下游推理服务处理不过来时，上游应该主动拒绝新请求而不是无限排队，避免队列无限增长导致所有请求都超时。

**深入追问：** 怎么防止 Agent 的工具调用失控，比如无限循环调用工具？

**回答要点：**

几个防护层：一是最大步骤数限制，Agent 每次执行设置最大工具调用次数（比如 20 步），超过后强制终止并返回当前状态；二是工具调用频率限制，同一个工具在一次 Agent 执行中调用次数不超过阈值；三是循环检测，如果连续几步的工具调用参数完全相同，说明 Agent 陷入循环，主动中断；四是总 token 预算，整个 Agent 执行过程的 token 消耗超过上限后终止。这些限制要在编排层实现，不能依赖模型自己停下来。

## 7. 框架选型与工程化能力

**面试官可能会问：** LangChain、LangGraph、FastAPI 这些框架怎么选，各自适合什么场景？

**回答要点：**

LangChain 适合快速原型和标准化流程，它封装了大量常见模式（RAG pipeline、工具调用、memory 管理），上手快，生态丰富。但它的抽象层较厚，调试困难，生产环境中过度封装会导致排查问题时很难看清楚底层发生了什么，对性能敏感的场景也不够灵活。

LangGraph 适合需要复杂状态管理和条件分支的 Agent，它把 Agent 的执行流程建模为有向图，每个节点是一个处理步骤，边是条件转换。相比 LangChain 的链式调用，LangGraph 更适合多步骤、有循环、需要人工介入（human-in-the-loop）的场景。

FastAPI 适合自己从头搭推理服务或工具服务的 API 层，性能好，类型安全，异步支持完善。Spring Boot 适合 Java 技术栈的团队，在企业级场景（权限管理、审计、与现有系统集成）有优势。选型原则：优先考虑团队熟悉度和可维护性，其次考虑场景匹配度。简单场景直接用 OpenAI SDK + FastAPI 往往比套 LangChain 更清晰。

**深入追问：** 生产环境中 LangChain 最常见的坑是什么？

**回答要点：**

最常见的几个坑：一是版本兼容性，LangChain 迭代很快，API 经常 breaking change，锁定版本很重要；二是调试困难，链式调用层层封装，出错时堆栈很深，很难定位是哪一层的问题，需要配合 LangSmith 的 trace 才能看清楚；三是过度依赖框架的 memory 和 chain 抽象，导致业务逻辑和框架逻辑耦合，后续迁移成本高。

**深入追问：** 什么是 Structured Output？和 Function Calling 的关系是什么？

**回答要点：**

Structured Output 是强制模型按固定 JSON Schema 输出的能力，确保输出格式严格符合预定义结构。Function Calling 是模型决定调用哪个工具并生成参数的能力。两者底层机制相似（都是约束模型输出为结构化 JSON），但用途不同。

Structured Output 适合不需要模型做工具选择的场景：数据抽取（从文本中提取结构化信息）、分类输出（返回固定字段的分类结果）、表单填充（生成符合特定格式的数据）。这些场景下，你已经知道要什么格式，只需要模型填充内容，不需要模型决策"用哪个工具"。

Function Calling 适合需要模型动态选择工具的场景：Agent 系统（模型根据任务选择调用哪个工具）、多工具协作（模型决定工具调用顺序和参数）。这些场景下，工具选择本身是模型的决策任务。

选型原则：如果任务是"把这段文本转成这个格式"，用 Structured Output；如果任务是"根据用户需求决定调用哪些工具"，用 Function Calling。Structured Output 更简单稳定，Function Calling 更灵活但也更容易出错。

## 8. 工程化与后端项目深挖

**面试官可能会问：** 讲一个你做过的 AI 后端项目，架构是怎么设计的，遇到了什么工程化挑战，如果重做会优先改哪里？

**回答要点：**

架构设计层面，一个典型的 AI 后端项目架构是：前端/客户端 → API 网关（限流、鉴权）→ 编排服务（Agent 逻辑、工具调用编排）→ 工具服务层（搜索、数据库、外部 API）→ 推理服务（vLLM/OpenAI API）→ 数据层（向量库、关系库、缓存）。编排服务和工具服务分开部署，编排服务无状态，工具服务各自独立扩展。

常见工程化挑战：长任务超时问题，最初用同步接口，Agent 执行超过 60 秒就超时，改成异步任务队列后解决；工具调用幂等问题，Agent 重试时重复发邮件，加了幂等键后解决；可观测性不足，出问题时只有日志，无法回放 Agent 决策过程，接入 Langfuse 全链路 trace 后排查效率大幅提升；并发打爆推理服务，没有做 token 级限流，高峰期 KV Cache OOM，加了并发槽位控制和 token 预算后稳定。

如果重做，优先改的是可观测性和异步任务架构。可观测性要从第一天就建好，出问题时没有 trace 数据，排查成本极高。异步任务架构要在设计阶段就确定，后期从同步改异步的重构成本很大。限流和幂等是稳定性底线，也要早做。
