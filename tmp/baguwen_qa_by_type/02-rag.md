# RAG 面试知识点

这份文档以真实面试追问链路为主线整理，目标是让你从"为什么需要 RAG"一路讲到"项目里怎么落地、怎么排障、怎么做取舍"，每个环节都有具体技术细节和落地经验可以展开。

---

## 1. 核心概念与原理

**面试官可能会问：** 什么是 RAG？用一句话概括它的核心思想。为什么需要 RAG？它解决了传统 LLM 的哪些痛点？

**回答要点：**

- **定义：** RAG（Retrieval-Augmented Generation，检索增强生成）是将检索和生成相结合的 AI 框架。核心思想是利用外部知识库的信息来增强 LLM 的生成能力，让模型能生成更准确、更可信的回答。
- **核心思想：** 可以概括为"开卷考试"或"先看书，再回答"。先根据用户问题从可更新的外部知识库检索相关参考资料，再将资料与问题一同交给 LLM，让 LLM 基于事实依据生成最终回答。
- **解决的痛点：**
  1. **知识滞后性：** LLM 的知识冻结在训练截止时刻。RAG 通过连接实时更新的外部知识库赋予模型访问最新信息的能力。
  2. **幻觉问题：** LLM 在知识缺失时倾向于"编造"事实。RAG 通过提供确切上下文，强制模型"言之有据"，显著减少幻觉。
  3. **缺乏可追溯性：** 传统 LLM 的回答是黑盒。RAG 的回答可以明确追溯到引用的原始文档，在金融、法律、医疗等领域至关重要。
  4. **私有数据利用难题：** 企业无法用内部私有数据公开训练大模型。RAG 允许 LLM 在不接触和记忆私有数据的前提下安全利用这些数据。

**深入追问：** RAG 能不能真正解决幻觉问题？它的边界在哪里？

**回答要点：** RAG 不能 100% 消灭幻觉，但能显著降低其发生率。以下情况仍可能产生幻觉：

1. **检索质量差（Garbage In, Garbage Out）：** 检索到的文档本身错误、过时或与问题不相关，LLM 可能基于这些"垃圾"信息推理出看似合理但实际错误的答案。
2. **LLM 自身局限性：** 即使上下文正确，部分模型也可能在生成长答案时"脱离"上下文，重新依赖其内部的错误知识。通常发生在 Prompt 指令不够强力或模型指令遵循能力较弱时。
3. **上下文冲突或信息不足：** 检索到多个相互矛盾的文档，或信息不足以回答复杂问题时，LLM 可能被迫猜测和编造。

**深入追问：** 什么问题适合 RAG，什么问题不适合只靠 RAG？

**回答要点：**
- 依赖外部知识的问题最适合 RAG：企业知识库问答、制度查询、客服 FAQ、私域文档检索。
- 开放创作、复杂规划、强推理、复杂决策，这类问题往往不是"缺知识"而是"缺能力"，只靠 RAG 不一定够，需要结合 Fine-tuning 或 Agentic 架构。

## 2. RAG 核心工作流程

**面试官可能会问：** 请描述一个完整的 RAG 工作流程，并说明每个阶段的作用。

**回答要点：**

RAG 的工作流程分为两大阶段：**离线索引阶段**和**在线检索与生成阶段**。

**阶段一：离线索引（Indexing）**

目标是将非结构化数据（PDF、文档、网页）转换为可高效检索的结构化格式。

1. **数据清洗与加载：** 从多源加载原始数据，深度清洗，去除广告、导航栏等噪声，提取纯净文本。
2. **分块（Chunking）：** 将长文档分割成大小适中、语义完整的文本片段。这是保证检索质量的基石。
3. **嵌入（Embedding）：** 使用嵌入模型将每个文本块转换为代表其语义的高维向量。
4. **索引与存储：** 将向量及其对应的原始文本块、元数据存入向量数据库，建立高效向量索引（如 HNSW）以实现快速搜索。

**核心思想：** "垃圾进，垃圾出"。离线索引是整个 RAG 系统的基础，分块策略和嵌入模型的选择直接决定了检索效果的上限。

**阶段二：在线检索与生成（Retrieval & Generation）**

目标是根据实时用户查询生成高质量、有依据的回答。

1. **Query 理解或改写：** 对用户问题进行意图识别或改写，提升检索覆盖度。
2. **查询向量化：** 使用与索引阶段相同的嵌入模型，将用户问题转换为向量。
3. **候选召回：** 在向量数据库中使用相似度搜索找出 Top-K 候选文本块。
4. **过滤：** 根据元数据（权限、时间、文档类型等）对候选池进行过滤。
5. **重排序（Re-ranking）：** 使用更精确但更慢的模型对 Top-K 进行精排，选出最相关的 Top-N（N<K）。
6. **上下文构造：** 对召回内容进行排序、压缩、去重、截断，组装成最终 Prompt。
7. **生成答案：** 将用户问题和上下文发给 LLM，生成最终答案并附上引用来源。
8. **引用或拒答：** 在证据不足时拒答，而不是乱答。

**深入追问：** 为什么 RAG 一般要拆成离线和在线两段？

**回答要点：** 因为职责天然不同。离线阶段负责"把知识准备好"，偏数据工程和知识组织；在线阶段负责"把知识用起来"，偏检索、生成和服务质量。拆开之后边界清楚，优化和排障也更容易落层。

**深入追问：** 在整条 RAG 链路里，哪几个环节最容易决定最终效果的上限？

**回答要点：** 通常是三层：① Chunk 质量（知识单元切得好不好）；② Embedding 质量（语义表征是否准确）；③ 检索链路（召回率和精排效果）。这三层在模型出场前就决定了系统上限，很多 RAG 问题在这里已经注定了，而不是模型的问题。

## 3. 分块（Chunking）

**面试官可能会问：** 分块有哪些策略？为什么分块是 RAG 最关键的基础问题之一？如何平衡语义完整性和块大小？

**回答要点：**

- **重要性：** Chunk 决定知识以什么粒度被检索、排序和送进模型，直接影响召回效果和最终答案质量。Chunk 太小会把语义切断，拿到半句话、半个逻辑；Chunk 太大会把噪声一起带进来，稀释相关性，污染上下文。
- **常见策略：**
  - **固定长度分块：** 按 Token 数切分，简单但容易破坏语义。
  - **基于分隔符分块：** 按句子、段落等自然分隔符切割，保留自然边界。
  - **递归分块（最推荐）：** 优先使用大型分隔符（`\n\n`），如果块仍过大则递归退回到更小的分隔符（`.`），确保语义连贯性。
  - **重叠分块（Chunk Overlap）：** 相邻块之间有部分内容重叠（约 10% 的块大小），避免在块边界处切断关键信息。
  - **Markdown/Code-Aware 分割：** 根据 Markdown 标题层级或代码的函数/类边界分割，保留文档逻辑结构。
  - **Small-to-Big / 父子块：** 索引更小的子块，检索时返回其所在的更大父块。兼顾检索精确性和上下文完整性。
  - **语义分块：** 用 Embedding 模型检测语义边界，在语义转折处切分。
- **实用经验：**
  - 通用文本，先从递归分块加适度 Overlap 起步。
  - 文档结构明显时，优先保结构。
  - 既想精准命中又想保完整上下文时，优先考虑父子块或 Small-to-Big。
  - 块大小没有通用准则，通常从 200-500 Token 开始实验并根据效果调整。

**深入追问：** 如果系统总是"能查到关键词，但上下文不完整"，通常先看哪一层？

**回答要点：** 先回头看 Chunk，不要先怪模型。很多"答不完整"的问题，本质上不是模型没能力，而是知识单元一开始就被切坏了。

## 4. 嵌入（Embedding）

**面试官可能会问：** Embedding 在 RAG 里扮演什么角色？为什么它决定召回上限？如何选择 Embedding 模型？

**回答要点：**

- **角色：** Embedding 模型是 RAG 的"语义翻译官"。它将人类语言转换为机器可以计算相似度的数学向量，把 Query 和文档映射到同一套语义空间。如果 Embedding 把"相关"表达错了，后面的索引、ANN、Rerank 都只能在错误表征上补救，所以它直接决定召回上限。
- **选择标准：**
  - **领域适应性：** MTEB 榜单是起点，但必须在自己的业务数据上构建评测集进行验证。优先选择在目标领域（金融、医疗）表现好的模型。
  - **语言支持：** 中文场景可以考虑 `BGE-large-zh`、`M3E-large` 等开源模型，需要平衡性能、速度和部署成本。
  - **维度与吞吐成本：** 维度越高精度越好，但存储和计算成本也越高。
  - **多语言能力：** 跨语言场景需要支持多语言的 Embedding 模型。
  - **私有化部署要求：** 数据安全敏感时，优先考虑可本地部署的开源模型。

**深入追问：** 通用 Embedding 和领域 Embedding 怎么取舍？什么时候值得做 Embedding 微调？

**回答要点：**
- 通用 Embedding 适合冷启动，泛化好、维护成本低。
- 领域 Embedding 更适合专业术语密集、细粒度语义区分要求高的场景（如"高血压 I 期"和"高血压 II 期"需要精确区分）。
- Embedding 微调是后手，不是第一反应。只有确认问题长期稳定地卡在表征层，而且靠 Chunk 优化、混合检索、Rerank 都补不动时，才值得做。

## 5. 召回：BM25、向量检索与混合检索

**面试官可能会问：** BM25、向量检索、混合检索分别适合什么场景？为什么很多生产系统最终都会走到混合检索？

**回答要点：**

- **BM25（稀疏检索）：** 基于词频统计，对精确关键词、产品型号、人名、错误码等专有名词匹配非常敏感，但无法理解语义，换个说法就可能召不回来。
- **向量检索（密集检索）：** 基于语义相似度，能理解同义词和概念，对同义改写、口语表达、非标准写法更友好，但对精确关键词不敏感。
- **混合检索（生产级系统的标配）：** 结合 BM25 和向量检索，能同时兼顾关键词精确匹配和语义模糊匹配，显著提升召回鲁棒性。通常使用 **Reciprocal Rank Fusion（RRF）** 算法融合两者的排名结果，效果稳定且无需大量调参。
- 真实业务 Query 往往同时包含关键词信号和语义信号，混合检索最大的价值是"稳"——不是理论最优，而是对不同 Query 类型都能给出更好的下限。

**深入追问：** 混合检索最难的点是什么？

**回答要点：** 难点不是"把两路结果拼起来"，而是结果融合、权重调校和不同 Query 类型下的稳定性。RRF 是一个很好的起点，但在某些场景下两路权重仍需根据业务数据调整。混合检索考验的是系统设计，不是简单叠加两套检索器。

## 6. 向量数据库、过滤与索引结构

**面试官可能会问：** 向量数据库在 RAG 中到底负责什么？`pgvector`、`FAISS`、`Milvus`、`Qdrant` 这类方案通常怎么选？元数据过滤为什么重要？

**回答要点：**

- **职责：** 向量数据库负责存储向量、原文引用和元数据，并提供高维相似度搜索能力。在企业场景里还要承担过滤、分区、多租户隔离、权限控制等能力。
- **常见方案取舍：**
  - **pgvector：** 轻量、一体化，适合小规模起步，减少基础设施复杂度。
  - **FAISS：** 更像底层库而非完整服务，适合自己掌控服务层，但缺乏生产级功能（持久化、动态更新、过滤）。
  - **Milvus：** 云原生设计，读写分离、计算存储分离，可扩展到千亿级向量规模，支持多种索引类型（HNSW、IVF_PQ）、强大的标量字段过滤和分区能力，是生产级首选。最新稳定版（2.4.x）引入了 Growing Index，支持增量构建索引，大幅提升数据实时更新效率。
  - **Qdrant / Weaviate：** 同样适合复杂过滤、多租户、服务化部署场景，API 设计更现代。
- **元数据过滤：** 解决"不是所有召回到的内容都应该进入候选池"这个问题。企业知识场景里权限过滤和多租户隔离不是可选项，而是底线能力，必须尽量前置到检索链路而不是等生成前再补。

**深入追问：** IVF 和 HNSW 的区别怎么理解？为什么很多人一上来就选重型向量库不一定划算？

**回答要点：**
- **IVF（Inverted File Index）：** 更像先把向量空间分桶（聚类），查询时只搜相关桶内的向量。适合超大规模数据，内存占用相对可控，但精度依赖聚类质量。
- **HNSW（Hierarchical Navigable Small World）：** 构建层次化的近邻图，查询时沿图逐步逼近目标。查询速度快、精度高，但内存占用更大。
- 小规模阶段最重要的是先把数据治理、Chunk、召回和 Rerank 跑顺，不一定要先引入最重的向量数据库。向量库选型的核心不是追新，而是让当前系统复杂度和业务复杂度匹配。

## 7. 重排序（Re-ranking）

**面试官可能会问：** 为什么召回之后通常还要做 Rerank？它和召回的职责边界是什么？

**回答要点：**

- **边界：** 召回追求**快和全（高 Recall）**，可能引入一些不那么相关的结果；Rerank 在召回的小候选集上追求**准和精（高 Precision）**。
- **模型：** Rerank 通常使用 **Cross-Encoder** 模型。它将 Query 和每个候选文档拼接后一起输入模型，能捕捉更深层次的交互信息，给出比向量相似度更精准的相关性评分。`bge-reranker-large` 是目前效果较好的开源模型之一。
- **代价：** Cross-Encoder 更准但更慢、更贵，适合小候选池精排（如从 Top-50 精排到 Top-5），不适合全量暴力重排。

**深入追问：** topk、topn、阈值这类参数怎么定？

**回答要点：**
- topk 偏候选池大小（召回多少候选），topn 偏最终进模型的材料数量（精排后留几条）。
- 先看召回层别漏多少，再看 Rerank 成本和上下文窗口预算。没有固定答案，一定要和业务时延、成本、效果一起调。

## 8. 生成阶段：上下文构造、Prompt、引用与拒答

**面试官可能会问：** 为什么生成阶段不是"把检索结果直接拼进去"这么简单？一个好的 RAG Prompt 通常包含什么？为什么需要引用和拒答？

**回答要点：**

- **上下文构造的必要性：** 直接把检索结果硬拼进去，常见问题是顺序不对、噪声太多、重复太多、重点不突出。上下文构造通常需要：
  - **排序：** 把最相关的材料放在 Prompt 的开头或结尾（避免"中段遗忘"问题）。
  - **压缩：** 对检索文档进行摘要，减少噪音，节省 Token 预算。
  - **去重：** 去掉语义重复的内容，避免稀释关键信息。
  - **截断：** 在上下文窗口限制内合理截断。
- **Prompt 结构：** 一个完整的 RAG Prompt 至少包含：
  1. **角色与任务指令：** "你是一个 AI 助手，请根据以下上下文回答问题"
  2. **检索到的上下文（Context）**
  3. **用户问题（Question）**
  4. **回答格式约束**
  5. **引用规则：** 要求模型标注答案来自哪条文档
  6. **拒答规则：** "如果上下文中没有足够信息，请回答不知道"
- **引用和拒答的意义：** 引用是为了可追溯性；拒答是为了在证据不足时别乱答；答案约束是为了减少模型脱离材料自由发挥。

**深入追问：** 为什么会出现"明明查到了，但还是答不好"？

**回答要点：** 常见原因三类：
1. 正确材料没排到前面（Rerank 或上下文排序问题）
2. 上下文太脏、太长、太乱（没做压缩和去重）
3. Prompt 没把"必须基于材料回答"约束住（指令遵循问题）

## 9. 评测、排障与优化闭环

**面试官可能会问：** 如何评估一个 RAG 系统的效果？为什么不能只看最终答案？真实项目里怎么做优化闭环？

**回答要点：**

- **为什么要分层评测：** RAG 不能只看最终答案，因为只看结果根本分不清问题在召回、排序、上下文构造还是生成。成熟的 RAG 评测一定是分层评测，至少把召回层、排序层、生成层拆开来看。
- **常见指标：**
  - **召回率（Recall）：** 候选池覆盖能力，正确答案有没有被找到。
  - **MRR（Mean Reciprocal Rank）：** 第一个正确结果排得靠不靠前。
  - **NDCG：** 整体排序质量。
  - **Relevance：** 答案是否与问题直接相关。
  - **Faithfulness（忠实性）：** 答案是否完全基于提供的上下文，没有捏造。
  - **Context Utilization：** 检索到的上下文有没有被真正利用。
- **自动化评估框架：** 可以使用 **RAGAS** 或 **TruLens** 等框架，通过 LLM 自动对上述指标打分，实现高效的自动化评估。
- **实用优化闭环：**
  - 先做一个小而可靠的标注集。
  - 每次改 Chunk、Embedding、召回、Rerank、Prompt，都回归同一套评测集。
  - 线上 Bad Case 一定要持续回流到离线评测里。

**深入追问：** 线上效果不好，你会优先怎么排？

**回答要点：** 按链路分层排查：
- **没查到：** 先看 Query 改写、Chunk 质量、Embedding 表征、过滤条件、召回参数。
- **查到了但排不前：** 看 Rerank 模型、融合策略、阈值设置。
- **排到了但没答好：** 看上下文构造、Prompt 约束、拒答规则。
- **答出来了但不可信：** 看引用机制、忠实性评估和 Bad Case 回流。

最怕一上来就归因给模型，很多 RAG 问题在模型出场前就已经注定了。

## 10. 主流高级方案：多路召回、Agentic RAG、GraphRAG

**面试官可能会问：** 除了基础 RAG，你还了解哪些高级方案？什么时候值得上这些方案？

**回答要点：**

高级方案不是为了"显得复杂"，而是在基础 RAG 已经跑通之后，针对特定问题继续补短板。

- **多查询改写 / Multi-Query Retrieval：**
  - **思想：** 用 LLM 将原始问题重写成多个语义相近或更具体的问题，从多个角度捕捉用户意图，再合并检索结果。
  - **目的：** 解决用户问题表述太短、太模糊、单次 Query 容易漏召的问题，显著提高召回率。

- **Agentic RAG：**
  - **思想：** 将 RAG 作为 LLM Agent 的核心工具。Agent 可以自主分析问题，决定何时检索、何时推理、甚至在信息不足时主动追问或重复检索。
  - **目的：** 解决问题本身需要拆解、追问、多步执行、动态决策的场景，使 RAG 从"被动问答机"进化为"主动问题解决者"。

- **GraphRAG（知识图谱增强 RAG）：**
  - **思想：** 结合向量数据库和知识图谱。对关系性强的问题在知识图谱中做精确路径搜索；对开放性问题用向量检索。
  - **目的：** 解决知识之间关系密集、多跳关联明显的场景，结合知识图谱的精确性和向量检索的语义泛化能力。

**深入追问：** GraphRAG 为什么不是默认方案？

**回答要点：** 建设和维护成本更高，对数据结构要求也更高（需要先构建知识图谱）。很多业务场景普通 RAG 已经能解决，没必要一上来就把复杂度拉满。这些方案的共同点是：效果上限更高，但系统复杂度、治理成本和评测难度也会一起上升。

## 11. 系统工程与实践考量

**面试官可能会问：** 在实际部署 RAG 系统时，你会考虑哪些工程问题？

**回答要点：**

真正把 RAG 做到线上，工程问题通常比"选哪个模型"更容易把系统做崩。通常优先考虑五件事：

**① 索引怎么更新**

索引更新是 RAG 真正的工程难点之一：
- 新增文档：直接增量索引，对新文档分块、嵌入并添加到向量库。
- 文档更新：找到旧文档的向量并删除，再对新版本重新分块、嵌入和添加。不能只补新 Chunk，不清旧 Chunk。
- 文档删除：原文、Chunk、向量、元数据和索引都要一起清理。
- Chunk 策略、Embedding 模型或索引结构大改时，通常需要全量重建。

**② 检索和生成链路怎么观测**

RAG 系统一定要能回放：用户问了什么、Query 被怎么改写、召回了什么、Rerank 后保留了什么、最终喂给模型什么、模型又输出了什么。没有这条链路的可观测性，排障基本靠猜。

**③ 成本和时延怎么控**

成本和时延通常集中在：Embedding 生成、检索和 Rerank、大模型生成、长上下文 Token 成本。常见取舍：
- Rerank 只精排小候选池（如 Top-50 → Top-5）。
- 上下文先压缩和去重，再送模型。
- 对高频 Query 做语义缓存。
- 对低频复杂 Query 允许更高成本。

**④ 权限和隐私怎么管**

企业知识场景里，多租户和权限隔离必须尽量前置到检索链路，而不是等生成前再补。在整个 RAG 管道中确保私有文档得到加密和权限控制，LLM 的 API 调用不将敏感数据暴露给第三方。

**⑤ Bad Case 怎么回流**

线上 Bad Case 一定要持续回流到离线评测里，形成闭环。否则优化靠感觉，系统很难收敛。

**编排框架：** 实践中通常使用 **LangChain** 或 **LlamaIndex** 等框架构建 RAG 管道。这些框架封装了数据加载、分块、嵌入、检索、Prompting 等模块，大大简化了开发流程。

## 12. RAG 与 Fine-tuning 的取舍

**面试官可能会问：** RAG 和 Fine-tuning 的主要区别是什么？如何选择？

**回答要点：**

这是一个高频经典面试题。

| **特性** | **RAG（检索增强生成）** | **Fine-tuning（微调）** |
| --- | --- | --- |
| **主要目的** | 增强模型**知识广度**，获取最新或私有数据 | 增强模型**特定能力**，如遵循指令、改变输出风格 |
| **工作原理** | "先检索后生成"，将外部知识作为上下文提供 | "更新模型参数"，使用特定任务数据继续训练 |
| **知识更新** | **快速、实时**，只需更新外部知识库 | **耗时、昂贵**，需要重新训练或微调模型 |
| **风险** | 可能因检索质量差影响回答 | 可能导致模型**灾难性遗忘**原有知识 |
| **典型场景** | 企业内部知识库问答、实时新闻摘要 | 改变客服机器人语气、让模型输出特定 JSON 格式 |

**选择原则：**
- 当问题的核心是**"知识"**（模型不知道某事）→ 选 **RAG**。
- 当问题的核心是**"技能"**（模型不知道如何做某事）→ 选 **Fine-tuning**。
- 在复杂生产实践中，通常采用 **"RAG + Fine-tuning"** 混合策略：用 Fine-tuning 让模型更"听话"、更懂"规矩"，再用 RAG 为它提供事实弹药。

**深入追问：** 什么情况下更适合只做 RAG，什么情况下需要 RAG + Fine-tuning？

**回答要点：**
- 知识变化快、私域数据多、需要可追溯时，通常优先只做 RAG。
- 如果系统已经能查到知识，但输出风格、结构化格式（如稳定输出 JSON）、指令遵循要求很高，单靠 RAG 不够，才考虑和微调结合。
- 如果问题根因是"模型没知识"，先补知识；如果根因是"模型不会按要求做"，再补能力。

## 13. 项目面试高频追问

**面试官可能会问：** 如果让你介绍一个做过的 RAG 项目，你通常会怎么讲？面试官最想听什么？

**回答要点：**

最稳的讲法按这条链路展开：

> **业务背景 → 数据来源 → 离线索引（分块/Embedding 选择原因）→ 在线检索（召回策略/Rerank 选择原因）→ 生成约束（Prompt 设计/拒答机制）→ 评测与优化（指标/闭环）→ 上线效果（Bad Case 和解决过程）**

面试官最想听的通常不是你用了什么框架，而是：
1. 你为什么需要 RAG，而不是只靠 Prompt 或微调
2. 你的数据是怎么治理的，Chunk 为什么这么切
3. 检索、Rerank、向量库为什么这么选
4. 你怎么证明系统真的变好了
5. 遇到过哪些 Bad Case，最后怎么定位和修

**深入追问：** 如果重做一次，你最想先改哪一层？

**回答要点：** 通常优先看离线知识组织和检索链路，因为这两层最容易决定系统上限。最怕的是一上来就归因给模型，因为很多 RAG 问题在模型出场前就已经注定了。
