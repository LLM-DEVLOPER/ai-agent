# 工具调用、Function Calling 与 MCP 面试知识点

这份文档按面试真实追问链路整理，从"为什么需要工具调用"一路讲到"怎么设计一个稳定的工具调用系统"，每个环节都有具体判断依据和工程经验可以展开。

---

## 1. 为什么 Agent 一定会走到工具调用

**面试官可能会问：** 为什么很多 Agent 最后一定要接工具？工具调用在 Agent 体系里到底解决什么问题？

**回答要点：**

- 模型的本质是"会说"，而工具调用把模型从"会说"变成"会做"。只靠模型内部知识，无法获取实时信息、无法操作外部系统、无法执行写操作。知识截止日期、私有数据访问、副作用执行——这三类需求都必须靠工具来解决。
- 工具调用在 Agent 体系里解决的核心问题是**行动能力**：让模型不只是生成文本，而是能真正与外部世界交互。没有工具的 Agent 只是一个会规划的智能体，有了工具才是能真正干活的智能体。
- 什么场景必须接工具：需要实时数据（天气、股价、新闻）、需要操作外部系统（发邮件、写数据库、调 API）、需要执行计算（代码运行、数学计算）、需要访问私域知识（企业内部系统）。
- 什么场景不一定需要工具：开放式创作、纯推理分析、知识类问答（模型参数里已有的）——这类任务强行接工具反而增加复杂度和失败风险。

**深入追问：** 工具调用和普通 API 调用的本质区别是什么？

**回答要点：**

普通 API 调用是由代码逻辑硬编码触发的，调用时机、参数、目标都是预先确定的。工具调用是由模型在运行时**动态决策**的——模型自己判断当前是否需要调用、调用哪个工具、用什么参数。这让系统具备了应对未预见情况的能力，但也引入了不确定性，这正是工具调用系统最难治理的地方。

## 2. Function Calling 的定义与运行链路

**面试官可能会问：** 什么是 Function Calling？它的完整链路是什么？

**回答要点：**

- **Function Calling** 是一种让 LLM 在生成过程中，输出结构化的工具调用请求（而不是自然语言文本）的能力。模型不直接执行工具，而是生成"我要调用哪个工具、用什么参数"的结构化决策，由外部系统执行后再把结果返回给模型。
- **本质是结构化动作决策**：和普通文本生成的区别在于，Function Calling 的输出是机器可直接执行的结构化 JSON，而不是给人看的自然语言。这是模型从"生成内容"到"执行动作"的关键桥梁。
- **完整链路：**
  1. 用户输入 + 工具列表（含 schema）送入模型
  2. 模型判断是否需要调用工具，选择哪个工具，生成参数
  3. 外部系统执行工具调用，拿到结果
  4. 工具结果作为新消息送回模型
  5. 模型基于工具结果生成最终回答
  6. 如果任务未完成，回到第 2 步继续循环

**深入追问：** 模型是怎么判断该不该调工具、调哪个工具的？

**回答要点：**

模型的判断依据来自三个地方：当前用户的意图、工具的描述信息（name + description）、以及上下文状态。工具描述写得好不好，直接决定模型能不能选对工具——这是工具调用系统里最容易被低估但影响最大的因素之一。

模型选工具的过程本质上是一次语义匹配：用户意图和工具描述之间的语义对齐程度，决定了选择正确率。所以工具描述不是随便写的注释，而是影响调用质量的核心设计。

**深入追问：** 工具返回结果后为什么还要再喂回模型，而不是直接给用户？

**回答要点：**

因为工具返回的是原始数据，不是用户能直接消费的答案。模型需要基于工具结果做进一步的理解、整合和表达——把机器返回的结构化数据转化成对用户有价值的回答。而且很多任务需要多轮工具调用，一次工具调用往往只是链路的中间步骤，不是终点。

## 3. MCP：定义、定位与协议详解

**面试官可能会问：** MCP 到底解决了什么问题？协议里标准化了哪些东西？MCP 为什么不只是一套 Tools 协议？和 Function Calling、普通 HTTP API 有什么区别？

**回答要点：**

- **MCP（Model Context Protocol）** 是 Anthropic 提出的开放协议，目标是标准化 AI 模型与外部工具、数据源之间的交互方式。在 MCP 之前，每个 Agent 框架（LangChain、LlamaIndex 等）都有自己的工具接入方式，工具无法跨框架复用，生态高度碎片化。MCP 让工具开发者只需实现一次，就能被所有支持 MCP 的 Agent 使用。
- **和 Function Calling 的区别：** Function Calling 是模型侧的能力——模型如何生成结构化工具调用请求；MCP 是协议层的标准——工具如何被描述、发现、调用和返回结果。Function Calling 是模型端的接口规范，MCP 是工具端的接入协议，两者解决不同层次的问题，可以同时存在。
- **和普通 HTTP API 的区别：** HTTP API 是为开发者设计的，需要提前知道文档和参数。MCP 内置了**能力发现**机制——Agent 可以在运行时动态查询 MCP Server 暴露了哪些工具，不需要提前硬编码，工具实现和 Agent 编排因此解耦。

MCP 标准化了 AI 模型与外部能力之间交互的完整接口，包含以下几个核心能力层：

- **Tools（工具）**：最核心的部分，定义了工具的名称、描述、参数 schema、调用方式和结果返回格式。这是 MCP 被讨论最多的部分，但不是全部。
- **Resources（资源）**：标准化了模型如何访问外部数据源（文件、数据库、API 返回内容），解决"模型如何读取结构化或非结构化外部内容"的问题。和 Tools 的区别是：Tools 是"执行动作"，Resources 是"读取数据"。
- **Prompts（提示模板）**：允许 MCP Server 向 Agent 暴露预定义的 Prompt 模板，让工具服务方可以提供标准化的任务引导，不只是提供执行能力。适合把复杂工具的使用方式封装成可复用的 Prompt。
- **Sampling（采样）**：允许 MCP Server 反向请求模型做推理，实现服务端驱动的模型调用，而不只是客户端（Agent）驱动。这让工具本身也可以在内部调用 LLM 能力，实现更复杂的工具逻辑。
- **能力发现（Capability Discovery）**：MCP Server 可以动态暴露自己支持哪些 tools/resources/prompts，Agent 在运行时查询，不需要提前硬编码。这是 MCP 和普通 REST API 的本质区别之一——传统接口是静态注册，MCP 是动态发现。

**MCP Server 的角色：** 是工具能力的封装和暴露层，把底层 API、数据库、服务以 MCP 协议的方式对外提供。Agent（MCP Client）通过协议与 Server 交互，不需要知道底层实现细节，实现了 Agent 与工具生态的解耦。

**深入追问：** 什么情况下项目里真的需要用到 MCP 的 prompts、resources、sampling？

**回答要点：**

- **Resources**：当工具需要给 Agent 提供大量上下文数据（如文件内容、数据库查询结果）而不是只返回简单字段时，用 Resources 比在 Tool 返回值里塞大段文本更合理。
- **Prompts**：当工具有复杂的使用模式，需要引导 Agent 按特定方式调用时，把使用指南封装成 Prompt 模板比写在文档里更直接有效。
- **Sampling**：当工具本身需要 LLM 做中间推理时（如工具内部需要理解用户意图、做内容分类），才用到 Sampling。大多数简单工具用不到。

多数项目只用到 Tools 这一层就够了，Prompts/Resources/Sampling 是在工具复杂度上升之后才需要考虑的。

## 4. Skill 的抽象与作用

**面试官可能会问：** 什么是 Skill？Skill 和 Tool 有什么区别？为什么很多系统会把能力先封装成 Skill？

**回答要点：**

- **Tool（工具）** 是原子能力——执行单一操作，如"查天气""发邮件""查数据库"。它是执行层的最小单元，通常是对某个外部 API 或系统的直接封装。
- **Skill（技能）** 是组合能力——本质上是**业务 SOP 的 AI 化封装**，把一个固定的多步业务流程打包成一个可调用的单元，模型只需要决定"要不要用这个 Skill"，不需要自己去组合底层工具。比如"帮我预订会议室"内部可能要查日历、发邀请、更新系统，但对模型来说只是一次 Skill 调用。Skill 是比 Tool 更高一层的抽象，面向业务任务而不是技术操作。
- **为什么要封装成 Skill：**
  1. **复用性**：同一个业务能力可以被不同 Agent 或不同场景调用，不用每次重新组合底层工具。
  2. **稳定性**：Skill 内部的编排逻辑是固定的，不依赖模型每次动态决策，比让模型自己组合 Tool 更稳定可控。
  3. **简化模型决策**：模型面对的是 Skill 这个高层接口，不需要理解底层多个 Tool 的细节，降低了决策复杂度和出错概率。
- **什么场景更适合设计成 Skill：** 业务流程固定、需要多步工具组合、对稳定性要求高的场景。如果流程完全固定，Skill 内部可以用规则编排而不依赖模型；如果流程有一定灵活性，Skill 可以在内部局部引入 Function Calling。

**深入追问：** Skill 和 MCP 暴露的能力是什么关系？

**回答要点：**

MCP 是协议层，定义的是工具/能力如何被描述和调用。Skill 是设计层，是对业务能力的组合封装。一个 Skill 可以通过 MCP 协议对外暴露，让其他 Agent 或系统通过标准协议调用它；也可以作为 MCP Server 内部封装的一个高层 Tool。两者不冲突，MCP 解决的是"怎么接入"，Skill 解决的是"封装什么"。

## 5. Schema 设计与调用质量

**面试官可能会问：** 工具 schema 为什么会直接影响调用正确率？什么样的 schema 更容易让模型调用对？

**回答要点：**

- 模型选工具和生成参数，完全依赖 schema 里的信息——工具名、工具描述、参数名、参数描述、类型约束。Schema 写得差，模型就算能力强也会调错。这是工具调用里最容易被低估但影响最大的工程决策之一。
- **好的 schema 通常具备：**
  - **工具名语义清晰**：`get_weather` 比 `tool_001` 好得多，模型靠名字做第一层判断。
  - **description 准确且无歧义**：描述要说清楚工具做什么、适合什么场景、不适合什么场景。有歧义的描述是误调的主要来源。
  - **参数名语义化**：`city_name` 比 `param1` 更容易让模型填对值。
  - **required 字段明确**：必填参数一定要标 required，否则模型可能漏填。
  - **枚举约束**：对于有限取值的参数，用 enum 列出所有合法值，减少模型自由发挥。
  - **默认值**：合理的默认值可以减少参数错误率。

**深入追问：** `tool_choice`、强制调用、禁止调用分别适合什么场景？

**回答要点：**

- **`tool_choice: auto`（默认）**：让模型自己决定是否调用，适合大多数场景。
- **强制调用（`tool_choice: required` 或指定工具）**：适合你确定当前任务一定需要调用某个工具的场景，比如数据提取类任务，强制走 Structured Output 输出。
- **禁止调用**：适合最终回答阶段，已经拿到了所有工具结果，不需要再调用，只需要模型做整合和表达。

**深入追问：** Structured Output 和 Function Calling 是什么关系？

**回答要点：**

两者底层机制相似，都是让模型输出符合特定 JSON Schema 的结构化内容。区别在于用途：Function Calling 更偏"模型决定调用哪个工具并生成参数"，Structured Output 更偏"强制模型按固定格式输出结果"。很多场景可以用 Structured Output 替代 Function Calling——当你不需要模型做工具选择，只需要它按固定格式输出时，Structured Output 更简单、更稳定。

## 6. 误调、漏调与参数错误

**面试官可能会问：** 模型为什么会误调、漏调、重复调工具？参数为什么容易出错？这些问题通常该从哪几层治理？

**回答要点：**

- **误调**（不该调却调了）：通常是 schema description 写得太宽，覆盖了不该覆盖的场景，或者多个工具描述有重叠，模型分不清边界。
- **漏调**（该调却没调）：通常是工具描述没有覆盖当前场景的表述方式，或者用户 query 和工具描述之间语义距离太大，模型认为不需要工具就能直接回答。
- **重复调**：通常是缺少状态追踪，模型不知道这个工具已经调过了，或者上下文没有保留上一次的工具结果。
- **参数错误**（缺失、类型错、字段错填）：最常见的原因是 schema 没有做足够的约束（缺少 required、类型不明确、没有 enum）、参数描述不清楚，或者模型对参数含义理解有偏差。

**这些问题通常出在哪几层：**

- Schema 层（最高频）：描述不清、约束不足、工具边界模糊
- Prompt 层：没有明确告诉模型什么时候该调、什么时候不该调
- 编排逻辑层：缺少状态追踪、没有处理工具调用失败的兜底
- 模型能力层（最少见）：只有在前三层都治理好之后，才考虑是不是模型本身的问题

**深入追问：** 如果工具选择和参数生成都不稳定，优先该从哪几层治理？

**回答要点：**

优先顺序是：**Schema 和工具描述 → Prompt 约束 → 编排兜底逻辑 → 考虑微调**。

最常见的误区是一上来就怪模型能力不行，但 80% 的工具调用问题在 schema 和描述层就能解决。先把工具描述写清楚、加足约束、消除歧义，再看 Prompt 里有没有明确的调用规则，最后才考虑是否需要针对工具调用做专门的对齐训练。

## 7. 多工具编排与上下文处理

**面试官可能会问：** 多工具调度为什么比单工具调用复杂很多？工具结果应该怎么处理？

**回答要点：**

- **复杂度来源：** 单工具调用只有"调不调、怎么调"两个问题。多工具调度还要解决：工具之间的依赖关系、执行顺序、结果传递、失败处理、上下文膨胀。工具一多，链路就容易失控。
- **串行 vs 并行：**
  - **串行调用**：下一个工具需要上一个工具的结果时必须串行，比如先搜索、再用搜索结果去查详情。
  - **并行调用**：工具之间没有依赖关系时可以并行，比如同时查天气和查日历，显著减少总耗时。
- **Router / Planner / Executor 的分工：**
  - **Planner**：负责把复杂任务拆解成子任务，决定工具调用的整体策略。
  - **Router**：负责根据当前状态选择下一个该调的工具。
  - **Executor**：负责执行具体的工具调用，处理返回结果。

**深入追问：** 工具结果为什么不能原样塞回上下文？

**回答要点：**

工具返回的是原始数据，往往又长、又杂、充满对当前任务无用的字段。原样塞进上下文会：① 浪费 Token 预算；② 引入噪声，稀释关键信息；③ 让模型注意力分散，降低最终回答质量。

正确做法是在工具结果进入上下文前做处理：提取关键字段、压缩冗余内容、只保留当前任务真正需要的信息。多工具串行时尤其重要——每一步都做整理，不要等最后一起清，否则上下文很快变成日志堆。

## 8. 失败处理与稳定性

**面试官可能会问：** 工具超时、失败、返回脏数据时应该怎么处理？如何提升 Function Calling 的整体稳定性？

**回答要点：**

- **工具失败不能让模型"自由发挥"**：工具失败后如果不做任何处理，直接让模型继续，模型会用内部知识填充，产生幻觉。失败必须有明确的处理策略，不能让模型自己决定怎么办。
- **四种处理策略：**
  - **重试**：适合瞬时性失败（网络超时、限流），要设置最大重试次数和退避策略，避免无限重试。
  - **回退**：适合工具返回结果不可信（脏数据、格式异常），换一种方式获取信息或降级处理。
  - **换工具**：适合当前工具不可用但有备选方案的场景，提前设计好工具的替代路径。
  - **拒答**：适合没有任何可靠信息来源时，明确告知用户无法完成，而不是编造答案。

**提升稳定性的方向：**
  - Schema 层：描述清晰、约束充分、边界明确——这是最高性价比的优化。
  - Prompt 层：明确调用规则、拒答规则、失败处理指令。
  - 编排层：状态追踪、超时控制、重试逻辑、兜底路径。
  - 只有在以上三层都做好之后，才考虑针对工具调用做专门的模型对齐训练。

**深入追问：** 什么时候值得为工具调用专门做训练或对齐？

**回答要点：**

当以下条件同时成立时才值得做：工具调用是核心链路、调用量足够大、通过 schema/prompt/编排优化已经到了瓶颈、有足够的高质量标注数据。大多数情况下，工程侧优化比模型训练性价比高得多——先把 schema 写好、prompt 写清楚、编排做稳，再考虑训练。

## 9. 项目深挖

**面试官可能会问：** 如果让你设计一个工具调用系统，你会怎么做？线上最难处理的 bad case 是什么？

**回答要点：**

设计一个稳定的工具调用系统，核心要解决四层问题：

1. **工具设计层**：Schema 写清楚、描述无歧义、工具边界明确、参数约束充分。这是整个系统质量的地基，做不好后面修起来成本极高。
2. **调用决策层**：通过 Prompt 明确调用规则（什么时候调、什么时候不调、调哪个）、tool_choice 控制调用行为、必要时强制或禁止调用。
3. **执行与兜底层**：超时控制、重试策略、失败兜底路径、工具结果的裁剪和整理。
4. **观测与评测层**：工具调用必须有 trace，把"用户输入 → 工具选择 → 参数 → 工具结果 → 最终回答"全链路记录下来，否则出问题根本没法排查。

**评测指标要分层看：**
- **工具选择正确率**：模型有没有选对工具。
- **参数正确率**：参数有没有填对。
- **任务完成率**：端到端任务有没有完成。

三个指标不能混在一起看。任务完成率高不代表工具选择正确，可能只是凑巧蒙对了；工具选择正确也不代表参数填对了。分层看才能定位问题在哪一层。

**线上最难处理的 bad case：**

- **模型该调工具但直接硬答**：模型认为自己知道答案，绕过了工具，但答案实际上是幻觉。这类问题最隐蔽，因为答案看起来合理，但不可信。
- **工具参数部分正确但关键字段错误**：工具被调用了，参数格式也对，但某个关键字段填错，工具返回了错误结果，模型还基于错误结果给出了"自信"的回答。
- **多工具链路中间某步失败，后续步骤基于错误信息继续执行**：这类问题在最终回答里很难直接看出来，必须靠链路回放才能发现。

**深入追问：** 什么时候更适合用 Function Calling，什么时候更适合手写规则编排？

**回答要点：**

- **Function Calling 更适合**：工具选择逻辑复杂、难以穷举规则、需要根据上下文动态决策的场景。它的优势是灵活，代价是不确定性。
- **手写规则编排更适合**：工具调用路径固定、业务逻辑明确、对稳定性要求极高的场景。它的优势是可预期、可控、易调试，代价是缺乏灵活性。

很多生产系统最终是两者结合：主链路用规则编排保证稳定性，在需要动态决策的局部节点引入 Function Calling。不要因为 Function Calling 更"智能"就全部用它，稳定性永远比灵活性优先级更高。
