# AI Agent / 大模型面试 Hot Top 100 速查卡

> 用途：面试前 1~2 小时快速过完，每题只记**一句核心答**，详细展开见各专题文档。
> 标注：⭐ = 高频必考，🔥 = 近期热门，💡 = 常被忽视的细节

## Part 1：Transformer 与模型基础（10 题）

**Q1. ⭐ Transformer 最大的贡献是什么？**
自注意力机制实现了 O(1) 的全局依赖建模，同时支持并行计算，解决了 RNN 的长距离依赖和串行计算两大痛点。

**Q2. ⭐ 为什么注意力计算要除以 √d_k？**
Q·K 点积的方差是 d_k，d_k 大时 softmax 趋近 one-hot，梯度消失；除以 √d_k 让方差稳定在 1，保持梯度可训练。

**Q3. ⭐ 为什么用 LayerNorm 而不是 BatchNorm？**
NLP 序列变长且有 padding，BatchNorm 依赖 batch 统计会被 padding 干扰；LayerNorm 在单个样本内部归一化，不受 batch 和序列长度影响。

**Q4. ⭐ Pre-LN 和 Post-LN 的区别？**
Post-LN 是 `LayerNorm(x + Sublayer(x))`，训练初期梯度不稳定需要 warm-up；Pre-LN 是 `x + Sublayer(LayerNorm(x))`，训练更稳定，GPT-2/GPT-3 都用 Pre-LN。

**Q5. ⭐ Decoder 里 Masked Attention 的作用？**
将未来位置的注意力分数设为极大负数（-1e9），softmax 后权重趋近 0，防止模型在训练时"偷看"未来 token，保证自回归的因果性。

**Q6. ⭐ 位置编码为什么用 sin/cos？**
对任意固定偏移量 k，PE(pos+k) 可以表示为 PE(pos) 的线性变换，模型能学到相对位置关系。也可以用可学习位置编码（BERT），或 RoPE（Llama 系列）。

**Q7. 🔥 Encoder-Only / Decoder-Only / Encoder-Decoder 分别适合什么任务？**
Encoder-Only（BERT）：分类、NER、理解类；Decoder-Only（GPT）：生成、对话；Encoder-Decoder（T5）：翻译、摘要、seq2seq 任务。

**Q8. 🔥 为什么现在大模型主流是 Decoder-Only，而不是 Encoder-Decoder？**
训练目标统一（Next Token Prediction），无需设计复杂的 encoder-decoder 对齐；In-context Learning 和 few-shot 能力更强；规模效应更好。

**Q9. ⭐ ViT 和 CNN 的核心区别？**
CNN 有归纳偏置（局部性+平移不变性），少量数据也能学好；ViT 归纳偏置弱，需要海量数据预训练，但全局建模能力强。

**Q10. 💡 Scaling Laws 说的是什么？**
模型性能与参数量 N、数据量 D、计算量 C 之间存在幂律关系，性能提升可以预测；Chinchilla 指出最优训练应该同步扩大模型和数据，而不是只扩大模型。

## Part 2：KV Cache 与推理优化（10 题）

**Q11. ⭐ KV Cache 解决什么问题？**
Decode 阶段每步需要所有历史 token 的 K/V，KV Cache 把已计算的 K/V 缓存在显存，避免重复计算，把 decode 从 O(n²) 降到 O(n)。

**Q12. ⭐ MHA / MQA / GQA / MLA 的区别？**
MHA：h 个独立 KV head；MQA：所有 Q head 共享 1 个 KV head（激进但掉质量）；GQA：g 组 KV head，折中（Llama 3 用）；MLA（DeepSeek）：低秩压缩 KV，压缩比~5.7x 且质量接近 MHA。

**Q13. ⭐ FlashAttention 快在哪里？**
不是减少计算量，而是减少 HBM 读写：用 tiling 把 Q/K/V 分块在 SRAM 里算完再写回，避免 N×N attention 矩阵的 HBM 往返，IO 从 O(N²) 降到 O(N)。

**Q14. ⭐ vLLM / PagedAttention 解决什么问题？**
传统推理预分配最大长度显存，碎片严重；PagedAttention 借鉴虚拟内存分页，KV Cache 按 block 按需分配，显存利用率大幅提升，支持高并发。

**Q15. ⭐ Prefill 和 Decode 的性能瓶颈分别是什么？**
Prefill：计算密集型（compute-bound），瓶颈是 FLOPs，决定 TTFT（首 token 延迟）；Decode：内存带宽密集型（memory-bound），瓶颈是显存带宽，决定 ITL（token 间延迟）。

**Q16. 🔥 Speculative Decoding 为什么能加速？**
用小 draft model 猜多个 token，大 target model 并行验证；猜对 k 个 token 只需 1 次 forward，decode 阶段 GPU 利用率低的问题被解决。高预测性任务（代码、翻译）收益明显，高并发时收益被稀释。

**Q17. ⭐ Prefix Cache 是什么？**
多个请求共享相同 system prompt 前缀时，把这段前缀的 KV 结果缓存复用，跳过 prefill 重复计算。System prompt 越长、前缀越稳定，收益越大（可降低 TTFT 50%+）。

**Q18. ⭐ INT4/INT8 量化的 trade-off？**
显存减半/四分之一，decode 速度提升（带宽瓶颈缓解）；代价是精度损失。INT8 通常可接受，INT4 在推理任务精度损失明显；AWQ 比 GPTQ 在 INT4 下质量更好（保护重要权重）。

**Q19. 💡 连续批处理（Continuous Batching）和静态批处理的区别？**
静态批处理等一整批完成才进新请求，GPU 空闲多；Continuous Batching 每个 decode step 后立刻插入新请求，GPU 利用率接近满载，vLLM 的调度器基于此实现。

**Q20. 💡 Token 级限流（TPM）为什么比 QPS 限流更准确？**
同 QPS 下，生成 10 token 和 1000 token 资源消耗差 100 倍；TPM（tokens per minute）直接衡量资源消耗，OpenAI 的限流就用 TPM。

## Part 3：RAG（10 题）

**Q21. ⭐ RAG 的完整链路是什么？**
文档解析 → 分块 → Embedding → 向量存储 → 检索（向量+BM25） → Rerank → 生成。关键：每步都可能是质量瓶颈。

**Q22. ⭐ 为什么用混合检索（Hybrid Search）而不是纯向量检索？**
纯向量检索对精确关键词（产品型号、人名）效果差；BM25 补足关键词匹配；用 RRF（Reciprocal Rank Fusion）合并两路排名，1/(k+rank) 加权，k=60 平滑。

**Q23. ⭐ Rerank 的作用是什么？**
用 Cross-Encoder 对（query, doc）做精确打分，比双塔 Embedding 精度高但慢；通常检索 top-20~50，Rerank 后取 top-3~5 送生成。常用：bge-reranker-v2-m3、Cohere Rerank。

**Q24. ⭐ 分块策略（Chunking）怎么设计？**
经验值：512~1024 tokens/块，10~15% 滑动窗口重叠；更好的是语义分块（按段落/章节边界），不要固定字符截断。块太大语义噪声多，块太小上下文丢失。

**Q25. ⭐ RAG 评测用什么指标？RAGAS 框架怎么理解？**
四维：Faithfulness（答案是否基于检索内容，防幻觉）、Answer Relevancy（答案是否回答了问题）、Context Precision（检索的文档有多少真正有用）、Context Recall（回答所需内容有没有被检索到）。

**Q26. 🔥 Lost in the Middle 是什么？对工程有什么影响？**
模型对长上下文中间部分利用能力明显弱于首尾；关键证据要前置或后置，不要埋在中间；原始文档顺序≠最优送入顺序。

**Q27. ⭐ 多租户权限隔离如何在 RAG 里实现？**
在向量库每个 chunk 打 tenant_id 和权限标签，检索时先用 metadata filter 过滤权限再做语义检索；不能等生成阶段再过滤（模型已看到无权限内容 = 信息泄露）。

**Q28. ⭐ RAG vs 微调，如何选择？**
RAG 解决知识问题（私域数据、最新信息）；微调解决能力/行为问题（格式不稳定、领域风格）；两者不冲突，先 RAG 补知识，再微调改行为。

**Q29. 💡 文档更新时如何做增量索引？**
维护 document_id → [chunk_id] 映射，更新时先按 document_id 批量删除旧 chunk，再写入新 chunk；否则向量库新旧版本并存，可能召回过期内容。

**Q30. 💡 GraphRAG 和传统 RAG 的核心区别？**
传统 RAG 基于语义相似度检索文档块；GraphRAG 把知识库构建成实体-关系图，检索时可以沿图结构多跳推理，适合需要跨文档关联的复杂问答（如"A 和 B 的关系是什么"）。

## Part 4：Agent 与工具调用（15 题）

**Q31. ⭐ Agent 和 ChatBot 的本质区别是什么？**
ChatBot 回答问题，Agent 完成任务。Agent 的核心是：围绕目标持续决策 + 执行动作 + 根据反馈推进，是闭环系统；ChatBot 通常单轮结束。

**Q32. ⭐ ReAct 模式是什么？**
Reasoning + Acting 交替：Thought（推理当前状态）→ Action（工具调用）→ Observation（工具结果）→ 下一轮 Thought，循环直到任务完成。和直接回答的本质区别：边做边想，每步基于实际反馈调整。

**Q33. ⭐ Function Calling 的完整链路？**
用户输入 + 工具列表（含 schema） → 模型决定调哪个工具 + 生成参数 → 外部系统执行 → 结果返回模型 → 模型基于结果生成回答 → 如未完成继续循环。

**Q34. ⭐ 工具 schema 写得好不好为什么直接影响调用准确率？**
模型选工具和填参数完全依赖 schema 描述；工具名语义清晰（get_weather > tool_001）、description 无歧义、参数名语义化、required 字段明确、enum 约束枚举值，80% 的工具调用问题在 schema 层能解决。

**Q35. ⭐ 模型误调/漏调/重复调工具的根因分别是什么？**
误调：description 太宽或工具描述有重叠；漏调：description 没覆盖当前场景，模型认为可以直接回答；重复调：缺少状态追踪；排查顺序：schema → prompt → 编排逻辑 → 模型能力。

**Q36. 🔥 MCP 和 Function Calling 的区别？**
Function Calling 是模型侧能力（如何生成结构化工具调用请求）；MCP 是协议层标准（工具如何被描述、发现、调用）。MCP 内置能力发现机制，Agent 运行时动态查询可用工具，不需要提前硬编码。

**Q37. ⭐ Tool 和 Skill 的区别？**
Tool：原子能力，单一操作（查天气、写数据库）；Skill：业务 SOP 的 AI 化封装，多步工具组合，模型只需决定"用不用这个 Skill"，内部稳定编排，比让模型自己组合 Tool 更可控。

**Q38. ⭐ Plan-and-Execute 和 ReAct 的适用场景区别？**
ReAct：边想边做，适合信息不完整、路径不确定的在线探索；Plan-and-Execute：先想后做，适合任务结构清晰、可提前拆解子目标的长链路任务，全局视角更清晰但规划出错整条链路走偏。

**Q39. ⭐ Prompt Injection 攻击是什么？为什么 Agent 风险更高？**
攻击者通过用户输入/检索内容注入恶意指令，覆盖 System Prompt 约束。Agent 风险更高因为它主动拉取外部内容（网页/文档/工具返回）且有执行能力，被注入成功可能造成真实危害（发邮件、删数据）。

**Q40. 💡 间接注入（Indirect Prompt Injection）为什么最难防？**
通过污染知识库文档/网页等系统主动拉取的内容来攻击，这些内容通常被系统信任，过滤比用户输入更宽松；防护要对所有外部内容一视同仁做隔离清洗。

**Q41. ⭐ Multi-Agent 系统中，Supervisor 模式和 Peer-to-Peer 的区别？**
Supervisor：中心协调者分配任务、汇总结果，便于管控和调试，但中心节点是单点；P2P：Agent 间直接通信协作，更灵活但难以追踪和调试。生产中 Supervisor 更常用。

**Q42. 🔥 A2A 协议（Agent-to-Agent）是什么？**
Google 提出的 Agent 间通信标准，每个 Agent 有 Agent Card（能力描述），通过标准化接口发现和调用其他 Agent；类比 MCP 是 Agent-Tool 协议，A2A 是 Agent-Agent 协议。

**Q43. ⭐ Agent 系统中 stop condition 为什么必须显式设计？**
没有明确终止条件，Agent 会循环执行或过早结束；显式设置：最大步数上限、成本上限、时间上限、可验证完成条件（工具返回预期结果/用户确认），任一触发就终止。

**Q44. 💡 Reflection（反思机制）和 Self-Refine 分别是什么？**
Reflection（Reflexion）：执行后把失败原因存入记忆，下次执行参考；Self-Refine：单次任务内生成初稿 → 自我评估 → 修改 → 再评估，迭代优化。适合有明确验证标准的任务。

**Q45. 💡 通用 Agent 最容易出现的"目标漂移"怎么防？**
每轮 ReAct 循环开始时把原始 Goal 重新注入 Prompt 作为锚点；在关键节点加"当前方向和原始目标是否一致"的显式验证步骤。

## Part 5：意图识别（5 题）

**Q46. ⭐ 意图识别在 Agent 系统里的核心作用？**
入口分流器：把请求路由到 FAQ/RAG/工具调用/多步 Agent/拒答等不同处理路径，用低成本方案处理简单请求，只把复杂请求路由到昂贵流程。

**Q47. ⭐ 意图识别的实现方案怎么选？**
生产首选：小分类模型（BERT 微调，<50ms）处理大多数，低置信度时升级到 LLM 兜底。不要全部用 LLM（慢且贵），不要全部用规则（泛化差）。

**Q48. ⭐ 意图识别 vs 意图对齐的区别？**
识别：把输入映射到意图类别（分类问题）；对齐：确保回答满足用户真实需求，不只是字面请求（"帮我改代码"≠"只修bug"，可能需要追问重构/优化/解释）。

**Q49. ⭐ OOS（Out-of-Scope）类为什么必须单独处理？**
没有 OOS 类，模型会强行把超范围请求分配到最相似的意图，产生错误路由；OOS 类要显式训练，触发时优雅拒答并引导用户，不能乱猜。

**Q50. 💡 槽位填充（Slot Filling）是什么？**
意图识别后检查必需参数是否齐全（"查天气"需要地点+时间），缺失时主动追问，而不是乱猜默认值；和意图识别合称 NLU（自然语言理解）。

## Part 6：微调体系（15 题）

**Q51. ⭐ SFT、RAG、Prompt Engineering、RLHF 分别解决什么问题？**
Prompt：模型有能力但需引导；RAG：模型没有知识；SFT：模型不会按要求做；RLHF/DPO：输出风格和人类偏好不对齐。先诊断根因再选方案。

**Q52. ⭐ LoRA 的核心公式和直觉？**
W' = W + (α/r)·BA，B∈R^{d×r}，A∈R^{r×k}，r<<min(d,k)。原理：微调的"有效变化"内在维度远小于参数空间，低秩矩阵就能表达。r=4~8 可达全量微调 90%+ 效果，参数量 0.1%~1%。

**Q53. ⭐ LoRA 的 α 参数是什么？**
有效缩放因子 = α/r，控制 LoRA 更新的实际强度。常用设置 α=2r；改变 r 时通过调 α 保持有效学习率稳定，不需要重调学习率。

**Q54. ⭐ QLoRA 的三个关键创新？**
1. NF4 4-bit 量化（针对正态分布优化的格式）；2. Double Quantization（对量化常数再量化）；3. Paged Optimizers（优化器状态 offload 到 CPU）。单张 24GB GPU 可微调 70B 模型。

**Q55. ⭐ SFT 的损失函数为什么只对 response 部分计算？**
SFT 目标是教模型"怎么回答"，不是"怎么复述问题"；对 instruction 也计算 loss 会让模型学会在推理时先重写问题，无益于指令遵循。

**Q56. ⭐ 什么是灾难性遗忘？怎么缓解？**
微调后在非目标任务上性能退化。缓解：数据混合（加2%~10%通用语料）；LoRA（原始权重冻结天然防遗忘）；降低学习率；评测时加 MMLU/C-Eval 通用基准对比微调前后。

**Q57. ⭐ DPO 和 PPO 的本质区别？**
PPO：显式训练 Reward Model + RL 优化，四个模型，系统复杂；DPO：数学上证明 RLHF 最优解可用分类损失直接表达，消去 RM，两个模型，更简单稳定。开源首选 DPO。

**Q58. ⭐ DPO 的损失函数是什么？**
L_DPO = -E[log σ(β·log(π_θ(y_w|x)/π_ref(y_w|x)) - β·log(π_θ(y_l|x)/π_ref(y_l|x)))]。让 chosen 回答相对 reference 概率提升，rejected 相对降低。

**Q59. 🔥 GRPO 和 PPO 的核心区别？**
GRPO 去掉了 value model（critic），用同一问题采样 G 个回答的组内相对 reward 估计优势函数：A_i = (r_i - mean) / std。从四个模型降为三个，DeepSeek-R1 使用。

**Q60. 🔥 DeepSeek-R1 的核心发现是什么？**
纯 RL 训练（无 SFT 冷启动）可以产生"顿悟时刻"——模型自发学会反思、验证、纠错，涌现类 o1 的长链推理能力。证明推理能力可以通过 RL 自发涌现，不必依赖人工构造 CoT 数据。

**Q61. ⭐ 为什么微调数据质量比数量重要？**
LIMA 实验（Zhou et al. 2023）：1000 条高质量样本效果超过 50000 条低质量样本。SFT 的目标是激活和引导能力，不是灌输知识；错误示例会覆盖预训练学到的正确行为。

**Q62. ⭐ 继续预训练（CPT）和 SFT 的区别？**
CPT：无监督语料上做 Next Token Prediction，训练"知识吸收"（让模型读懂领域词汇/概念）；SFT：（指令, 回答）对上做监督学习，训练"行为对齐"。通常先 CPT 再 SFT。

**Q63. 💡 可验证 Reward 和 Reward Model 怎么选？**
可验证 Reward（数学题对错、代码通过测试）：准确、免标注、可自动化，首选；Reward Model：主观任务（写作质量、安全性）无客观标准时用，但会 reward hacking，需持续维护。

**Q64. 💡 LoRA 部署：合并 vs 动态加载？**
合并：W' = W + (α/r)·BA，推理零延迟，适合单任务固定部署；动态加载：共享基础模型权重，按任务切换 LoRA，适合多任务/多租户（LoRA MoE 的基础）。

**Q65. 💡 RLHF 的 KL 散度约束是什么意思？**
RLHF 目标 max E[r(x,y)] - β·KL[π||π_ref]，KL 项防止 policy 偏离 reference 模型太远（避免 reward hacking）；β 越大，偏离惩罚越重，越保守。

## Part 7：记忆与上下文（8 题）

**Q66. ⭐ 上下文工程和记忆系统的区别？**
上下文工程：当前这一轮把哪些信息送进模型（请求级）；记忆系统：跨轮/跨会话保留什么、什么时候取出（资产管理级）。记忆系统决定存什么，上下文工程决定当前轮用什么。

**Q67. ⭐ Token 预算怎么分配？**
系统约束 > 用户当前输入 > 相关历史 > 检索结果。预算不够时先裁低相关历史和冗余工具结果，不能裁系统约束。系统 prompt 要稳定不变（行为稳定 + prefix cache 命中）。

**Q68. ⭐ 长期记忆写入的门槛怎么设计？**
只写稳定信息（用户偏好、长期事实）；临时任务状态、一次性工具结果不写；门槛太低记忆库变脏，误召回率上升，系统因记忆变更差。

**Q69. ⭐ 记忆命中率高但效果差是什么原因？**
四类：误召回（命中不相关）；记忆过期（旧信息）；上下文污染（和其他内容形成噪声）；没被利用（信息进去了但模型没用上，关键信息埋在中间）。

**Q70. 💡 KV Cache 和应用层 Memory 的本质区别？**
KV Cache：推理层计算优化，生命周期在单次请求内，节省 GPU 算力；应用层 Memory：跨轮/跨会话信息管理，需要显式设计写入/召回/淘汰。名字像但完全是两回事。

**Q71. 💡 多轮对话为什么容易出现约束丢失？**
任务状态、约束、历史混在一起；约束没被显式持久化，只靠"还没被截掉"来维持有效性，历史越长越容易被冲淡；解决：约束单独存储并每轮显式注入，不要混在消息历史里。

**Q72. 💡 记忆系统 vs RAG 的边界？**
记忆：用户私域信息（偏好、历史任务），用户私有，生命周期随用户；RAG：公共/业务知识（FAQ、规章制度），多用户共享。两者权限属性不同，必须隔离，不能混在同一个检索池里。

**Q73. 💡 Prefix Cache 最适合什么内容？**
System Prompt、固定工具 schema、固定输出格式——稳定不变的前缀。动态内容（用户输入、实时检索结果）破坏前缀稳定性，无法形成可复用缓存；应用层上下文设计要有意识地把静态前缀和动态尾部分开。

## Part 8：评测与幻觉（7 题）

**Q74. ⭐ 幻觉（Hallucination）的主要类型？**
事实错误（Factual）：生成与事实不符的内容；忠实度问题（Faithfulness）：答案与检索上下文矛盾；归因错误（Attribution）：引用了不存在的来源。

**Q75. ⭐ RAG 系统中幻觉来自哪几层？**
生成层幻觉（检索对但生成错）；检索层幻觉（召回的文档本身是错的）；归因层幻觉（有来源支持但被误引）。排查要按层分开，不能全部归因到"模型"。

**Q76. ⭐ LLM-as-Judge 是什么？有什么问题？**
用大模型对其他模型的输出打分，解决了传统指标（BLEU/ROUGE）无法评估语义质量的问题。问题：位置偏见（偏向先出现的答案）、自我偏见（偏向自己生成的风格）、长度偏见（偏向更长答案），需要设计多角度评测和对抗样本。

**Q77. ⭐ 怎么防止 RAG 输出幻觉？**
四层：检索层（确保召回内容正确）；Prompt 层（要求模型"如果上下文中没有相关信息，明确说不知道"）；输出层（Faithfulness 分数过滤）；业务层（高风险内容人工审核）。

**Q78. 💡 BLEU/ROUGE 和 LLM-as-Judge 各适合什么场景？**
BLEU/ROUGE：翻译、摘要等有参考答案的任务，快速、客观、可重复；LLM-as-Judge：开放式生成、创意任务、对话质量，能评估语义和逻辑但成本高且有偏见。

**Q79. 💡 微调后怎么验证没有遗忘通用能力？**
三层评测：目标任务离线集（验证微调有效）、hold-out 测试集（验证泛化）、通用基准（MMLU/C-Eval/HumanEval，验证无遗忘）。只看第一层容易把过拟合误判为提升。

**Q80. 💡 RAG 准确率低时怎么排查？**
按链路分层：数据层（知识库有没有答案）→ 分块层（关键信息有没有被截断）→ 检索层（召回的文档是否相关）→ Rerank 层（排序后 top-k 里有没有正确答案）→ 生成层（给了正确文档是否能正确回答）。不要一上来就调 Prompt。

## Part 9：系统设计（10 题）

**Q81. ⭐ 企业知识库的多租户权限隔离怎么设计？**
向量库每个 chunk 打 tenant_id + 权限标签，检索时先 metadata filter 过权限再做语义检索；不能等生成前再过滤（模型已看到=信息泄露）；向量库选型要支持灵活 filter（Qdrant/Weaviate）。

**Q82. ⭐ 客服 Agent 高风险操作（退款）的安全设计？**
三层：参数从系统查（不由模型算）→ 执行前二次确认（展示金额让用户确认）→ 超阈值转人工审核。模型只负责理解意图和准备参数，执行要有明确授权机制。

**Q83. ⭐ AI 系统的效果/成本/时延三角怎么权衡？**
按业务优先级定底线：实时客服时延优先，知识库效果优先，批处理成本优先；手段：模型分级路由（简单→小模型）、语义缓存、流式输出、降级策略。

**Q84. ⭐ DeepResearch 类系统的核心架构？**
问题分解（子问题列表）→ 并行多源搜索（每个子问题独立 ReAct）→ Map-Reduce 分层摘要（段落→文档→全文）→ 结构化报告生成+引用标注。

**Q85. ⭐ AI 系统上线后的 bad case 闭环怎么做？**
收集（用户差评/人工抽检）→ 分类归因（检索/生成/工具层）→ 修复 → 回归测试 → 上线验证 → **加入 Golden Set**（不加就会反复复发）。

**Q86. 💡 向量库怎么选型？**
Qdrant（高性能 filter，生产首选）；Weaviate（内置 BM25 混合搜索）；Milvus（亿级向量，分布式）；Chroma（本地开发）；Pinecone（托管，filter 性能弱）。

**Q87. 💡 系统设计题怎么开场？**
先问约束（用户规模、数据量、准确率要求、时延要求、成本预算、安全合规），不要一上来就画架构图。约束决定架构，不同假设导出完全不同的方案。

**Q88. 💡 为什么 AI 服务的限流要用 TPM 而不是 QPS？**
同 QPS 下，生成 10 token 和 1000 token 资源消耗差 100 倍；TPM 直接衡量 GPU 资源消耗，比 QPS 更准确；结合 QPS 做双层限流。

**Q89. 💡 系统设计结尾应该说什么？**
主动说方案的局限性（做了什么 trade-off）→ 规模扩展路径（10x 用户/数据时哪些调整）→ 监控和迭代计划（如何数据驱动优化）。比说"这个方案完美"更能体现工程成熟度。

**Q90. 💡 100 万文档知识库需要哪些架构调整？**
向量库单机→分布式（Milvus 集群）；Embedding 计算走 GPU 批处理异步管道；热数据高性能存储/冷数据低成本存储分层；增量更新走 Kafka 消息队列削峰；高频查询检索结果 Redis 缓存。

## Part 10：工程与多模态（10 题）

**Q91. ⭐ SSE 和 WebSocket 在流式输出中怎么选？**
SSE（Server-Sent Events）：单向服务端推流，HTTP 长连接，实现简单，适合大多数 AI 流式输出场景；WebSocket：双向全双工，适合需要实时双向交互（语音对话、实时协作）的场景。

**Q92. ⭐ Trace 和 Observability 在 AI 系统里要记录什么？**
全链路：用户输入 → 检索文档 → 完整 Prompt → 模型输出 → 工具调用参数和结果 → 最终响应 → 各阶段耗时和 Token 消耗。缺任何一层，出问题无法定位。

**Q93. ⭐ AI 应用的接口幂等性为什么重要？**
客户端重试时同一请求可能多次到达；生成类接口（非查询）如果不幂等会重复执行（重复发邮件、重复写数据库）；用 request_id 做幂等键，服务端检查已处理则返回缓存结果。

**Q94. 🔥 CLIP 是什么？多模态 RAG 怎么实现？**
CLIP：对比学习，把图像和文本映射到同一向量空间，实现图文跨模态相似度计算。多模态 RAG：图片提前 Embedding（CLIP 等）存向量库，查询时图文混合检索；或用多模态 LLM（GPT-4V）直接理解图片，伪多模态（图片→文字描述→文字 RAG）vs 原生多模态（图片直接送入 VLM）。

**Q95. 💡 LangGraph 和 LangChain 的区别？**
LangChain：工具/链路封装，偏线性流程；LangGraph：把 Agent 流程建模为有向图（节点=处理步骤，边=状态转移），支持循环、条件分支、并行，适合复杂多步 Agent；Graph 结构让流程可视化和调试更容易。

**Q96. 💡 多模态模型的幻觉比文本模型更严重吗？**
通常是的：图文对齐难度更高，模型容易"看图说文"，生成与图片不符但文本上合理的描述；多模态幻觉的评测更难（需要同时验证文本质量和图文一致性），主流评测：HallusionBench、POPE。

**Q97. 💡 什么是 Structured Output？和 Function Calling 的关系？**
Structured Output：强制模型按固定 JSON Schema 输出；Function Calling：模型决定调哪个工具+生成参数。底层机制相似，区别在用途：不需要模型做工具选择时，Structured Output 更简单稳定（如数据抽取、分类输出）。

**Q98. 💡 大模型应用里怎么做 A/B 测试？**
流量分桶（用户/请求 hash 分组）→ 对照组/实验组各自的 Prompt/模型/检索策略 → 收集同维度指标（任务完成率/用户满意度/时延）→ 统计显著性检验（T-test/Mann-Whitney）。样本量要足够（通常千级以上），不能靠感觉判断。

**Q99. 💡 为什么不能只靠 LLM 自己说"完成了"来判断任务结束？**
LLM 容易过度自信，即使任务实际失败也可能输出"已完成"；要有可验证的完成条件（工具返回了预期结果、输出通过格式校验、用户确认），或者用另一个 LLM 做独立验证（Judge）。

**Q100. ⭐ 如果面试最后问"你有什么问题"，怎么加分？**
问技术方向："你们现在 Agent 系统的主要技术瓶颈是什么？是稳定性、成本还是效果？"或者问迭代节奏："你们的评测/迭代周期是多久？有没有 golden set？"这类问题展示你是从工程实战角度思考，而不是只会背题。
