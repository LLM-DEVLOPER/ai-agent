# 微调体系面试知识点

这份文档专注微调体系，从"什么时候该微调"一路讲到"LoRA 数学原理、DPO 推导、数据质量、项目落地"，每个环节都有具体判断依据和工程经验可以展开。

---

## 1. 微调到底在解决什么问题

**面试官可能会问：** 什么时候该用微调？什么时候用 RAG？什么时候用 Prompt Engineering 就够了？

**回答要点：**

这三种方案解决的是不同层次的问题，不是互相替代的关系，要先诊断问题根因再选方案。

**Prompt Engineering** 解决的是"模型需要引导"的问题。模型已经有能力，只是需要通过 few-shot 示例、角色设定、输出格式约束来激活正确的行为。成本最低，但能力上限受限于模型本身，无法解决模型根本不会做某件事的问题。

**RAG** 解决的是"模型没有知识"的问题。模型的训练数据有截止日期，私有数据更不可能出现在预训练里。RAG 通过在推理时检索相关文档并注入上下文，让模型能回答它"不知道"的问题。知识更新成本低，不需要重新训练，但无法改变模型的行为模式。

**SFT（监督微调）** 解决的是"模型不会按要求做"的问题。比如模型不会按特定格式输出、不会遵循特定领域的写作风格、不会执行某类特定任务。SFT 通过在高质量样本上继续训练，改变模型的行为模式，而不只是补充知识。

**DPO/RLHF** 解决的是"模型输出风格和人类偏好不对齐"的问题，比如输出太啰嗦、有害内容过滤不够、回答风格不符合产品要求。

**深入追问：** 如果系统已经能查到知识，但输出格式不稳定，该用哪种方案？

**回答要点：**

这是典型的"能力问题"而不是"知识问题"，应该用 SFT 而不是 RAG。RAG 补充的是知识，改变不了模型的输出行为。如果模型不会稳定输出特定 JSON 格式，靠 Prompt 约束有时候够，但如果格式要求复杂或者模型本身指令遵循能力弱，就需要 SFT 来强化这个行为。

## 2. LoRA 与参数高效微调

**面试官可能会问：** LoRA 的核心思想是什么？为什么低秩分解有效？QLoRA 又解决了什么问题？

**回答要点：**

LoRA（Low-Rank Adaptation，Hu et al. 2021）的核心思想是：不直接修改预训练权重 W，而是在旁边并联两个低秩矩阵，只训练这两个矩阵：

$$W' = W + \Delta W = W + BA$$

其中 $B \in \mathbb{R}^{d \times r}$，$A \in \mathbb{R}^{r \times k}$，秩 $r \ll \min(d, k)$。原始权重 W 完全冻结，推理时把 $\Delta W = BA$ 加回 W，不增加推理延迟。

低秩分解有效的理论依据来自 Aghajanyan et al. 2020 的研究：模型在微调时的参数更新，其**内在维度**（intrinsic dimensionality）远小于参数空间的维度。换句话说，微调所需的"有效变化"可以用一个低秩矩阵来近似表达，不需要更新全部参数。实验验证了这一点——用 r=4 或 r=8 的矩阵，往往就能达到全量微调 90% 以上的效果，但训练参数量只有原来的 0.1%~1%。

**QLoRA**（Dettmers et al. 2023）在 LoRA 基础上进一步降低显存需求：把基础模型量化到 4-bit（NF4 格式），只有 LoRA 的 A、B 矩阵保持 bf16 精度参与训练。同时引入了 Double Quantization（对量化常数再量化）和 Paged Optimizers（用 CPU 内存缓解 GPU 显存峰值）。这让在单张消费级 GPU 上微调 70B 级别的模型成为可能。

**深入追问：** LoRA 应该加在哪些层？秩怎么选？效果不好时先查什么？

**回答要点：**

通常加在 Transformer 的 attention 层，具体是 Q、K、V、O 投影矩阵，有时也加在 FFN 层。秩的选择取决于任务复杂度——简单任务用 r=4 或 r=8 通常够用，复杂任务可以用 r=16 或 r=32。秩越大，表达能力越强，但训练参数量也越多。

效果不好时的排查顺序：先看数据质量（数据质量是微调效果的天花板，数据差再大的秩也救不了）；再看学习率（LoRA 的学习率通常比全量微调大一个数量级，1e-4 到 1e-3 是常见范围）；再看秩是否够用；最后才考虑是否需要扩大 LoRA 的覆盖范围（加更多层）。

## 3. 对齐与偏好优化：RLHF、PPO 与 DPO

**面试官可能会问：** RLHF 是什么？PPO 和 DPO 有什么本质区别？

**回答要点：**

RLHF（Reinforcement Learning from Human Feedback）是让模型输出更符合人类偏好的训练范式。核心流程是：收集人类对模型输出的偏好标注（chosen/rejected 对）→ 训练 Reward Model 预测人类偏好分数 → 用 RL 优化语言模型，让它生成能获得高 reward 的输出，同时用 KL 散度约束不要偏离原始模型太远：

$$\max_\pi \mathbb{E}_{x \sim D, y \sim \pi}[r(x, y)] - \beta \cdot \text{KL}[\pi \| \pi_{\text{ref}}]$$

PPO 是 RLHF 中最常用的 RL 算法，需要同时维护四个模型：policy model、reference model、reward model、value model（critic）。系统复杂，训练不稳定，超参数敏感，工程成本高。

**DPO（Direct Preference Optimization，Rafailov et al. 2023）** 的突破在于：它从数学上证明了 RLHF 的最优解可以用一个简单的分类损失来直接表达，不需要显式的 reward model 和 RL 训练循环：

$$\mathcal{L}_{\text{DPO}} = -\mathbb{E}\left[\log \sigma\left(\beta \log \frac{\pi_\theta(y_w|x)}{\pi_{\text{ref}}(y_w|x)} - \beta \log \frac{\pi_\theta(y_l|x)}{\pi_{\text{ref}}(y_l|x)}\right)\right]$$

其中 $y_w$ 是 chosen 回答，$y_l$ 是 rejected 回答。DPO 直接优化策略，让 chosen 回答的概率相对于 reference model 提升，rejected 回答的概率相对降低。更简单、更稳定，是目前开源社区最常用的对齐方案。

**深入追问：** DPO 有什么局限性？什么时候还是要用 PPO？

**回答要点：**

DPO 的主要局限是依赖静态的偏好数据集，无法在线更新——模型改变后，原来的偏好标注可能不再准确（distribution shift 问题）。PPO 可以在线采样、在线更新 reward，理论上能持续优化。对于需要复杂推理、长链路任务（数学、代码）的对齐，PPO 或其变体（如 GRPO，DeepSeek-R1 使用）往往效果更好，因为这类任务的 reward 信号更稀疏，需要在线探索来发现正确的推理路径。

## 4. 训练数据与样本质量

**面试官可能会问：** 为什么微调数据的质量比数量更重要？CoT 数据有什么特殊价值？

**回答要点：**

微调阶段，模型已经有了强大的预训练基础，SFT 的目标是"激活"和"引导"能力，而不是"灌输"知识。这个阶段，低质量数据的危害远大于数量不足——错误的格式示例、前后矛盾的标注、低质量的推理链，都会让模型学到错误的行为模式，甚至覆盖掉预训练阶段学到的正确知识。LIMA（Zhou et al. 2023）的实验表明，1000 条高质量样本的 SFT 效果可以超过 50000 条低质量样本。

CoT 数据的特殊价值在于它不只教模型"给出答案"，而是教模型"怎么推理"。包含推理步骤的训练样本，能让模型在推理时也倾向于展开中间步骤，从而在复杂任务上表现更好。这也是为什么 o1、DeepSeek-R1 等推理模型大量使用 long-form reasoning 数据进行训练——推理能力的提升不是靠更多知识，而是靠更多推理轨迹。

**深入追问：** 从用户行为日志抽训练数据时要注意什么？

**回答要点：**

用户日志里的数据有几个典型问题：一是**选择偏差**，用户更倾向于对好的回答点赞，差的回答可能直接被忽略，导致负样本不足；二是**分布偏移**，线上用户的 query 分布和目标任务分布可能不一致；三是**噪声**，用户行为信号（点赞、收藏）是弱监督，不等于回答质量高。实践中通常需要对日志数据做过滤（去掉低质量回答）、去重（避免模板化样本过多）、再平衡（补充边界情况和困难样本），而不是直接拿来训练。

## 5. 项目深挖

**面试官可能会问：** 介绍一个你做过的微调项目，为什么选择微调而不是 RAG 或 Prompt？如果重做，最值得优先改哪一层？

**回答要点：**

微调项目的讲法核心是讲清楚**为什么选微调而不是 RAG 或 Prompt**。这是面试官最想听的判断依据，而不是你用了什么框架。通常的讲法是：先说业务问题（模型在什么任务上表现不够好）→ 分析根因（是知识问题还是能力问题）→ 说明为什么选微调（能力问题，RAG 解决不了）→ 讲数据构造（怎么收集、怎么过滤、质量怎么保证）→ 讲训练方案（LoRA 还是全量，秩怎么选，学习率怎么调）→ 讲评测（怎么证明微调有效，有没有通用能力遗忘）→ 讲上线效果和 bad case。

**典型 bad case** 有几类：第一，**通用能力遗忘**——微调后模型在目标任务上变好了，但在其他任务上退化了，通常是训练数据多样性不够或学习率太高；第二，**过拟合训练格式**——模型学会了训练数据的格式，但泛化到新 query 时格式崩掉；第三，**数据污染**——训练数据里有低质量样本，模型学到了错误的行为模式。

**如果重做，最值得优先改哪一层**：几乎所有微调项目的瓶颈都在数据，而不是训练方案。数据质量差，换更大的秩、更长的训练、更复杂的方案都救不了。先把数据治理好（过滤、去重、补充边界样本、加 CoT 推理链），再考虑训练方案的优化。

**深入追问：** 怎么判断微调是否真的有效，而不是只是在训练集上过拟合？

**回答要点：**

评测要分三层：第一层是目标任务的离线评测集（和训练数据同分布，验证基本能力）；第二层是 hold-out 测试集（和训练数据不同分布，验证泛化能力）；第三层是通用能力基准（如 MMLU、C-Eval，验证有没有遗忘）。只看第一层是不够的，很容易把过拟合误判为真正的提升。线上 A/B 测试是最终验证，但要配合足够的样本量和统计显著性检验，不能靠感觉判断。
