# 模型训练与微调体系面试知识点（完全版）

这份文档覆盖大模型训练的完整生命周期：从预训练、继续预训练、到微调（SFT、RLHF/DPO），每个环节都有具体判断依据、数学原理和工程经验可以展开。

---

## 1. 预训练、继续预训练与微调的本质区别

**面试官可能会问：** 预训练、继续预训练、微调分别在解决什么问题？什么时候用哪个？

**回答要点：**

这三种训练方式解决的是模型能力构建的不同阶段，不是互相替代的关系。

| 训练方式 | 训练目标 | 数据类型 | 解决的问题 | 成本 |
|---------|---------|---------|-----------|------|
| **预训练（Pretraining）** | Next Token Prediction | 海量无标注通用数据（TB 级） | 构建通用语言能力和世界知识 | 极高（千卡月） |
| **继续预训练（CPT）** | Next Token Prediction | 领域无标注数据（GB~TB 级） | 让模型"读懂"专业领域词汇和概念 | 高（百卡天） |
| **监督微调（SFT）** | 监督学习 | 高质量（指令, 回答）对（千~万条） | 让模型"听话"，按指令行事 | 低（单卡小时~天） |
| **偏好对齐（RLHF/DPO）** | 强化学习/偏好优化 | 人类偏好标注对（千~万对） | 让模型输出符合人类偏好 | 中（十卡天） |

**核心区别：**

- **预训练**：在海量通用数据（Common Crawl、Wikipedia、GitHub、Books）上做无监督语言建模，让模型学习语言规律、常识、推理能力。这是模型能力的"地基"，决定了能力上限。训练数据通常是 TB 级，需要千卡月的算力。

- **继续预训练（CPT/Domain-Adaptive Pretraining）**：在特定领域的无监督语料上继续做语言建模，目的是让模型"读懂"该领域的专业词汇、概念和表达方式。适合场景：金融、医疗、法律等高度专业化领域，模型对领域知识完全陌生时。训练的是"知识吸收"。

- **监督微调（SFT）**：在（指令, 回答）对上做监督学习，目的是让模型"按要求行事"。训练的是"行为对齐"。预训练模型虽然知识渊博，但不知道如何以对话形式响应用户，SFT 教会它这个能力。

- **偏好对齐（RLHF/DPO）**：通过人类偏好反馈，让模型输出更有用、更安全、更符合人类价值观。SFT 教会模型"怎么回答"，RLHF 教会模型"怎么回答得更好"。

**实践中的典型路径**：预训练（通用能力）→ 继续预训练（领域知识，可选）→ SFT（指令遵循）→ RLHF/DPO（偏好对齐）。

**深入追问：** 什么时候该用微调？什么时候用 RAG？什么时候用 Prompt Engineering 就够了？

**回答要点：**

这四种方案解决的是不同层次的问题，要先诊断问题根因再选方案。

| 方案 | 解决的根本问题 | 适用信号 | 局限 |
|------|--------------|---------|------|
| **Prompt Engineering** | 模型有能力但需要引导 | 改 Prompt 就能解决 | 能力上限受限于模型 |
| **RAG** | 模型没有（私有/最新）知识 | 知识截止日期、私域数据 | 改变不了行为模式 |
| **SFT（监督微调）** | 模型不会按要求做 | 格式不稳定、风格要求、领域行为 | 需要高质量标注数据 |
| **RLHF/DPO** | 输出风格与人类偏好不对齐 | 有害内容、冗长、风格偏差 | 需要偏好标注对，成本高 |

**深入追问：** 如果系统已经能查到知识，但输出格式不稳定，该用哪种方案？

**回答要点：**

这是典型的"能力问题"而不是"知识问题"，应该用 SFT 而不是 RAG。RAG 补充的是知识，改变不了模型的输出行为。如果模型不会稳定输出特定 JSON 格式，靠 Prompt 约束有时候够，但如果格式要求复杂或者模型本身指令遵循能力弱，就需要 SFT 来强化这个行为。

## 2. 预训练基础：构建通用能力

**面试官可能会问：** 预训练的训练目标是什么？为什么 Next Token Prediction 能让模型学到这么多能力？

**回答要点：**

预训练的核心目标是**自监督的语言建模（Next Token Prediction）**：给定前文 $x_1, x_2, ..., x_{t-1}$，预测下一个 token $x_t$。

损失函数：

$$\mathcal{L}_{\text{pretrain}} = -\sum_{t=1}^{T} \log P_\theta(x_t \mid x_{1:t-1})$$

**为什么这个简单目标能学到复杂能力？**

1. **知识压缩**：要准确预测下一个词，模型必须理解上下文的语义、逻辑、常识。这迫使模型把海量文本中的知识压缩到参数里。

2. **涌现能力**：当模型规模（参数量、数据量、计算量）达到临界点后，会涌现出预训练目标没有显式训练的能力，如推理、代码生成、多步规划。

3. **自监督的优势**：不需要人工标注，可以利用互联网上的所有文本数据（TB 级），数据规模是监督学习无法比拟的。

**深入追问：** 预训练数据的"配方"（Data Recipe）是什么？为什么重要？

**回答要点：**

数据配方指不同数据源的混合比例，直接决定模型的能力分布。

典型配方（参考 LLaMA、Qwen）：

- **网页数据（Common Crawl）**：60%~70% - 提供通用知识和语言多样性
- **代码（GitHub）**：10%~15% - 提升逻辑推理和代码能力
- **书籍（Books）**：5%~10% - 提升长文本理解和叙事能力
- **学术论文（ArXiv）**：3%~5% - 提升科学推理能力
- **百科（Wikipedia）**：2%~5% - 提供结构化知识

**为什么重要**：配方不同，模型能力偏向不同。代码比例高的模型（如 CodeLlama）代码能力强但通用对话能力可能弱；书籍比例高的模型长文本理解更好。

## 3. SFT 训练机制：损失函数与训练数据格式

**面试官可能会问：** SFT 的损失函数是什么？和预训练的目标有什么区别？

**回答要点：**

SFT 使用的是**有监督的语言建模损失**，但有一个关键细节：**只对 response 部分计算 loss，instruction 部分 mask 掉**。

给定（指令 $x$, 回答 $y$），损失函数为：

$$\mathcal{L}_{\text{SFT}} = -\sum_{t=1}^{|y|} \log P_\theta(y_t \mid x, y_{1:t-1})$$

和预训练的 Next Token Prediction 的区别：预训练对整个序列计算 loss，SFT 只对 response token 计算 loss。这是有意义的设计——我们希望模型学会"怎么回答"，而不是学会"把问题重新复述一遍"。

**训练数据格式（Chat Template）**：

现代 SFT 通常使用结构化的对话模板，例如：

```
<|system|>你是一个专业的助手</s>
<|user|>请解释量子纠缠</s>
<|assistant|>量子纠缠是指两个或多个粒子...</s>
```

不同模型有不同的 template（Llama 3、ChatML、Mistral 等），训练时一定要和推理时的 template 一致，否则会出现指令遵循失效的问题。

**深入追问：** SFT 时 instruction 部分应该 mask 吗？不 mask 会有什么影响？

**回答要点：**

业界主流是 mask instruction 部分（即 instruction 不参与 loss 计算）。理由是：如果对 instruction 也计算 loss，模型会学习"复述问题"，这对指令遵循没有帮助，甚至可能让模型在推理时倾向于先把问题重写一遍。

但也有研究（如 Guo et al.）发现，在某些场景下不 mask instruction 能让模型更好地理解指令语境，提升长指令下的遵循能力。实践中大多数场景 mask 效果更好，争议存在但影响通常不大。

## 4. 参数高效微调（PEFT）体系

**面试官可能会问：** 有哪些参数高效微调方法？LoRA 的核心思想是什么？为什么低秩分解有效？

**回答要点：**

PEFT（Parameter-Efficient Fine-Tuning）的核心思路是：冻结大部分预训练权重，只训练少量新增的或特定的参数，在保留通用能力的同时适应目标任务。主流方法：

| 方法 | 核心思想 | 参数量 | 推理延迟 |
|------|---------|--------|---------|
| **LoRA** | 低秩矩阵分解 Δ W=BA | 极少（0.1%~1%） | 无（可合并） |
| **Adapter** | 在 FFN 后插入小型 bottleneck 层 | 少 | 有（串行增加计算） |
| **Prefix Tuning** | 在 KV 中插入可学习 prefix | 少 | 有（增加序列长度） |
| **Prompt Tuning** | 在输入嵌入层加 soft token | 极少 | 几乎无 |
| **IA³** | 对 K、V、FFN 做可学习的逐元素缩放 | 极少 | 几乎无 |

LoRA 是目前最主流的方法，原因是它**可以无损合并到原模型权重中**，推理时零延迟。

### 3.1 LoRA 数学原理

LoRA（Low-Rank Adaptation，Hu et al. 2021）的核心：不直接修改预训练权重 $W$，而是并联两个低秩矩阵，只训练这两个矩阵：

$$W' = W + \Delta W = W + \frac{\alpha}{r} \cdot BA$$

其中：
- $B \in \mathbb{R}^{d \times r}$，$A \in \mathbb{R}^{r \times k}$，秩 $r \ll \min(d, k)$
- $\alpha$ 是缩放超参数（通常设为 r 的两倍，即 $\alpha = 2r$）
- $\frac{\alpha}{r}$ 是有效缩放因子，控制 LoRA 更新的实际强度
- 初始化：$A$ 使用正态初始化，$B$ 初始化为全零，保证训练初期 $\Delta W = 0$

**为什么低秩分解有效？** 理论依据来自 Aghajanyan et al. 2020 的研究：模型在微调时的参数更新，其**内在维度**（intrinsic dimensionality）远小于参数空间的维度。换句话说，微调所需的"有效变化"可以用一个低秩矩阵来近似表达。实验表明，用 r=4 或 r=8 的矩阵，往往就能达到全量微调 90% 以上的效果，但训练参数量只有原来的 0.1%~1%。

**直觉理解**：预训练权重 $W$ 已经捕获了通用语言能力，微调只需要做一个"方向调整"——这个调整所在的子空间维度远小于整个参数空间。LoRA 就是在找那个低维子空间。

### 3.2 α 参数的意义

面试高频考点：**LoRA 里 α 参数是干什么的？**

答：`lora_alpha` 是一个缩放因子，有效更新强度为 $\frac{\alpha}{r}$。这个设计的意义：当我们改变秩 r 时，不需要同步重调学习率——通过调整 α 来保持有效学习率稳定。

实践原则：
- `lora_alpha = 2 * lora_r` 是常见设置，相当于把 BA 的输出乘以 2
- 学习率和 α 可以相互替代调节，但 α 更直观
- 如果发现 LoRA 效果弱，可以尝试增大 `lora_alpha`（等效于增大更新强度）

### 3.3 QLoRA：极限节省显存

**QLoRA**（Dettmers et al. 2023）在 LoRA 基础上进一步降低显存需求，三项关键创新：

1. **NF4 量化（4-bit NormalFloat）**：把基础模型压缩到 4-bit，专门针对正态分布参数优化的数据格式，比普通 int4 精度损失更小
2. **Double Quantization（双重量化）**：对量化常数本身再做一次量化，进一步节省约 0.5 bits/参数
3. **Paged Optimizers**：在 GPU 显存峰值时把优化器状态 offload 到 CPU 内存，避免 OOM

只有 LoRA 的 A、B 矩阵保持 bf16 精度参与训练，基础模型保持 4-bit 冻结。这让在单张 24GB 消费级 GPU 上微调 70B 级别的模型成为可能。

**深入追问：** LoRA 应该加在哪些层？秩怎么选？效果不好时先查什么？

**回答要点：**

通常加在 Transformer 的 attention 层：Q、K、V、O 投影矩阵。有时也加在 FFN 层（`gate_proj`、`up_proj`、`down_proj`）。覆盖越全，表达能力越强，但训练参数量也越多。

秩的选择（经验值）：
- 简单任务（格式对齐、风格迁移）：r=4 或 r=8
- 中等任务（领域问答、特定技能）：r=16 或 r=32
- 复杂任务（代码生成、长链路推理）：r=64 甚至全量

效果不好时的排查顺序：
1. **先看数据质量**——数据质量是天花板，数据差再大的秩也救不了
2. **再看学习率**——LoRA 的学习率通常比全量微调大一个数量级（1e-4 到 1e-3 是常见范围）
3. **再看秩是否够用**——任务复杂度超过 r 的表达能力
4. **最后考虑扩大覆盖范围**——加更多层的 LoRA

**深入追问：** 训练/微调时 OOM 了怎么办？

**回答要点：**

训练 OOM 和推理 OOM 的根因不同。训练显存消耗 = 模型权重 + 梯度 + 优化器状态（Adam 需要 2 份额外状态）+ 激活值（前向传播中间结果，反向传播时需要）。

**排查与解法：**

1. **减小 batch size**：最直接有效，从 8 降到 4 或 2，用 gradient accumulation 补偿（多步累积梯度再更新）
2. **用 LoRA/QLoRA 替代全量微调**：QLoRA 把基础模型量化到 4-bit，只训练 LoRA 参数，单卡 24GB 可训 70B
3. **Gradient Checkpointing**：不保存所有激活值，反向传播时重算，显存换时间（训练慢 20-30%）
4. **优化器状态 offload**：把 Adam 状态放 CPU（DeepSpeed ZeRO、QLoRA 的 Paged Optimizers）
5. **混合精度训练**：用 bf16/fp16 替代 fp32，显存减半
6. **减小序列长度**：`max_seq_length` 从 2048 降到 1024，激活值显存消耗是序列长度的平方

**面试一句话总结**：训练 OOM 先降 batch size + gradient accumulation，再考虑 LoRA/QLoRA，最后上 gradient checkpointing 和 DeepSpeed ZeRO。

## 4.5 训练监控与 Loss 异常排查

**面试官可能会问：** 训练时 Loss 出现异常（NaN、不收敛、震荡）怎么排查？

**回答要点：**

Loss 异常是微调中最常见的问题，需要系统性排查，不能靠猜。标准排查流程：

**1. 先查数据（90% 的问题在这里）**

- **检查 batch 内容**：从 dataloader 取一个 batch，打印 input_ids、labels、attention_mask
- **常见问题**：labels 里有 -100 以外的负数、input 长度超过 max_length、tokenization 错误、数据预处理引入 NaN

**2. 再查学习率**

- **Loss = NaN**：学习率过大导致梯度爆炸 → 降低学习率（从 1e-4 降到 1e-5）
- **Loss 不下降**：学习率过小或 warmup 不够 → 增大学习率或延长 warmup
- **Loss 震荡**：学习率衰减策略不合适 → 检查 scheduler（cosine/linear）

**3. 检查梯度**

- **监控梯度范数（Gradient Norm）**：正常范围通常在 0.1~10 之间
- **梯度爆炸**：grad_norm > 100 → 启用 gradient clipping（max_grad_norm=1.0）
- **梯度消失**：grad_norm < 0.001 → 检查模型初始化或网络结构

**4. 检查混合精度训练**

- **FP16 下溢**：梯度太小被截断为 0 → 调整 loss_scale 或改用 BF16
- **BF16 vs FP16**：BF16 动态范围更大，数值更稳定，是 H100/A100 的推荐选择

**5. 检查分布式训练**

- **梯度同步问题**：多卡训练时梯度没有正确 all-reduce → 检查 DDP/FSDP 配置
- **Batch size 等效性**：`global_batch_size = per_device_batch_size × num_gpus × gradient_accumulation_steps`

**深入追问：** 训练时应该监控哪些关键指标？

**回答要点：**

除了 Loss，还要监控：

- **学习率曲线**：确认 warmup 和衰减是否按预期执行
- **梯度范数**：检测梯度爆炸/消失
- **训练吞吐（tokens/s）**：发现性能瓶颈
- **GPU 利用率和显存占用**：优化资源使用
- **验证集指标**：每 N 步在 hold-out 集上评估，防止过拟合

**面试一句话总结**：Loss 异常排查遵循"数据 → 学习率 → 梯度 → 精度 → 分布式"的顺序，90% 的问题在数据层，不要跳过直接调模型。

## 5. 灾难性遗忘与通用能力保留

**面试官可能会问：** 什么是灾难性遗忘？微调时如何避免？

**回答要点：**

**灾难性遗忘（Catastrophic Forgetting）** 是指：在目标任务上微调后，模型在之前掌握的通用能力上显著退化。表现为：微调后的模型在目标任务 benchmark 上提升，但在 MMLU、C-Eval 等通用基准上明显下降。

**根本原因**：梯度更新优先服务当前任务分布，会覆盖掉预训练阶段学到的权重方向。数据越单一、学习率越高、训练越长，遗忘越严重。

**缓解策略**：

1. **数据混合（Data Mixing）**：在微调数据中混入一定比例的通用预训练数据（如 2%~10% 的通用语料）。这是最直接有效的方法，让模型在学新技能的同时保持"记性"。

2. **参数高效微调（PEFT）**：LoRA、Adapter 等方法天然有遗忘缓解效果——冻结大部分权重，只更新少量参数，原始模型能力被"封装"保留。

3. **低学习率 + 更短训练**：减小更新步幅，降低权重偏移幅度。

4. **EWC（Elastic Weight Consolidation）**：为重要权重添加正则约束，防止关键参数被大幅修改。工程上较少直接用，原理值得了解。

5. **LoRA Merge 策略（DARE/TIES-Merging）**：先分别训练通用 LoRA 和任务 LoRA，再通过稀疏化合并，兼顾两种能力。

**深入追问：** 怎么检测是否发生了灾难性遗忘？

**回答要点：**

评测策略要覆盖"目标任务 + 通用能力"双维度：

- **目标任务**：在 hold-out 测试集上评，验证微调有效性
- **通用能力**：MMLU（英文）、C-Eval（中文）、HumanEval（代码）——用这些基准对比微调前后的分数差异
- **典型安全基准**：如果产品有安全要求，还要验证安全拒答能力有没有退化

一旦发现通用能力明显下降（超过 2~3 个百分点），就应该考虑加通用数据混合或降低训练强度。

## 6. 对齐与偏好优化：RLHF、PPO 与 DPO

**面试官可能会问：** RLHF 是什么？PPO 和 DPO 有什么本质区别？

**回答要点：**

RLHF（Reinforcement Learning from Human Feedback）是让模型输出更符合人类偏好的训练范式，整体分三阶段：

1. **SFT 阶段**：在高质量示范数据上做监督微调，得到初始 policy 模型
2. **Reward Model 训练**：收集人类对模型输出的偏好标注（chosen/rejected 对），训练 RM 预测人类偏好分数
3. **RL 优化**：用 RL 优化 policy 模型，同时用 KL 散度约束不要偏离参考模型太远：

$$\max_\pi \mathbb{E}_{x \sim D, y \sim \pi}[r(x, y)] - \beta \cdot \text{KL}[\pi \| \pi_{\text{ref}}]$$

**Reward Model 架构**：通常以 SFT 模型为基础，在最后一层接一个线性头（scalar head），对（prompt, response）输出一个标量分数。训练目标是 Bradley-Terry 排名损失：

$$\mathcal{L}_{\text{RM}} = -\mathbb{E}\left[\log \sigma(r(x, y_w) - r(x, y_l))\right]$$

其中 $y_w$ 是人类偏好的回答，$y_l$ 是被拒绝的回答。

**PPO 的工程复杂度**：需要同时维护四个模型（policy、reference、reward、value/critic），系统复杂，训练不稳定，超参数敏感，工程成本极高。

### 5.1 DPO：绕开 RL 的优雅推导

**DPO（Direct Preference Optimization，Rafailov et al. 2023）** 的突破在于：它从数学上证明了 RLHF 的最优解可以用一个简单的分类损失直接表达，不需要显式的 RM 和 RL 循环。

推导核心：RLHF 目标的最优 policy 满足：

$$\pi^*(y|x) \propto \pi_{\text{ref}}(y|x) \cdot \exp\left(\frac{1}{\beta} r(x,y)\right)$$

对上式取对数可以反解出 reward：

$$r(x,y) = \beta \log \frac{\pi^*(y|x)}{\pi_{\text{ref}}(y|x)} + \beta \log Z(x)$$

将这个 reward 代入 Bradley-Terry 模型并消去 $Z(x)$，得到 DPO 损失：

$$\mathcal{L}_{\text{DPO}} = -\mathbb{E}\left[\log \sigma\left(\beta \log \frac{\pi_\theta(y_w|x)}{\pi_{\text{ref}}(y_w|x)} - \beta \log \frac{\pi_\theta(y_l|x)}{\pi_{\text{ref}}(y_l|x)}\right)\right]$$

其中 $y_w$ 是 chosen 回答，$y_l$ 是 rejected 回答。DPO 直接优化 policy：让 chosen 回答相对 reference 的概率提升，rejected 回答相对降低。更简单、更稳定，是目前开源社区最常用的对齐方案。

**深入追问：** DPO 有什么局限性？什么时候还是要用 PPO？

**回答要点：**

DPO 的主要局限：

1. **分布偏移（Distribution Shift）**：DPO 依赖静态偏好数据集，数据是用参考模型生成的。训练一段时间后，policy 模型偏离了 reference，原来的偏好标注可能对新分布不再准确。PPO 可以在线采样、实时更新 reward，避免这个问题。

2. **稀疏 reward 任务效果有限**：对于数学推理、代码生成这类任务，正确答案的 reward 信号极为稀疏（只有最终答案对错）。DPO 需要人工构造偏好对，质量和覆盖度有限。PPO（或 GRPO）可以在线探索多条推理路径，发现正确轨迹。

3. **无法处理复杂奖励信号**：当 reward 需要实时环境反馈（如代码执行结果、工具调用结果）时，DPO 的静态数据集无法覆盖。

### 5.2 GRPO：DeepSeek-R1 的推理对齐方案

**GRPO（Group Relative Policy Optimization）** 是 DeepSeek 团队在 DeepSeek-R1 中使用的 RL 方案，是 PPO 的简化变体：

核心改动：**去掉了 value model（critic）**，改用组内相对奖励来估计优势函数（advantage）：

$$A_i = \frac{r_i - \text{mean}(\{r_j\}_{j=1}^G)}{\text{std}(\{r_j\}_{j=1}^G)}$$

对同一个问题采样 $G$ 个回答，计算每个回答相对组内平均 reward 的相对优势，用于更新 policy：

$$\mathcal{L}_{\text{GRPO}} = -\mathbb{E}\left[\min\left(\frac{\pi_\theta(y|x)}{\pi_{\text{old}}(y|x)} A, \text{clip}\left(\frac{\pi_\theta(y|x)}{\pi_{\text{old}}(y|x)}, 1-\epsilon, 1+\epsilon\right) A\right)\right] + \beta \cdot \text{KL}[\pi_\theta \| \pi_{\text{ref}}]$$

**GRPO 的优势**：
- 从四个模型减少到三个（去掉 critic），显著降低显存和工程复杂度
- 通过组内相对比较，reward 信号更稳定，不需要精心设计 value function
- 对推理任务（数学、代码）效果好，因为这些任务有明确的对错 reward（可验证性）

DeepSeek-R1 的一个重要发现：**纯 RL 训练（无 SFT 冷启动）会产生"顿悟时刻"**——模型自己学会了反思、验证、纠错，涌现出类似 o1 的长链推理能力。这证明推理能力可以通过 RL 自发涌现，不一定需要人工构造 CoT 数据。

## 7. 训练数据与样本质量

**面试官可能会问：** 为什么微调数据的质量比数量更重要？CoT 数据有什么特殊价值？

**回答要点：**

微调阶段，模型已经有强大的预训练基础，SFT 的目标是"激活"和"引导"能力，而不是"灌输"知识。这个阶段，低质量数据的危害远大于数量不足——错误的格式示例、前后矛盾的标注、低质量的推理链，都会让模型学到错误的行为模式，甚至覆盖掉预训练阶段学到的正确知识。

**LIMA（Zhou et al. 2023）** 的实验表明：**1000 条高质量样本的 SFT 效果可以超过 50000 条低质量样本**。这个结论背后的理论解释是"Superficial Alignment Hypothesis"——模型的能力来自预训练，SFT 只是在教模型用什么格式和风格来表达这些能力，因此数据质量（示范什么行为）远比数量更重要。

**CoT 数据的特殊价值**：

CoT 数据不只教模型"给出答案"，而是教模型"怎么推理"。包含推理步骤的训练样本，能让模型在推理时也倾向于展开中间步骤，从而在复杂任务上表现更好。

这也是为什么 o1、DeepSeek-R1 等推理模型大量使用 long-form reasoning 数据进行训练——推理能力的提升不是靠更多知识，而是靠更多推理轨迹。数学证明、代码生成、多步规划任务，都受益于 CoT 风格的训练数据。

**深入追问：** 高质量数据集如何构造？有哪些常见方法？

**回答要点：**

主要有三种途径：

1. **人工标注（Gold Standard）**：质量最高，但成本极高。适合高价值核心任务，数据量不需要很大（参考 LIMA 的 1000 条规模）。重点在覆盖典型场景、边界 case 和困难样本。

2. **模型蒸馏（Distillation）**：用更强的 teacher 模型（如 GPT-4）生成高质量回答，用这些回答训练 student 模型。成本低，质量高，是目前最流行的数据构造方式。Alpaca（用 text-davinci-003 生成）、Vicuna（用 ChatGPT 生成）都是这个路线。局限：student 无法超越 teacher 的能力天花板。

3. **Self-Play / 合成数据**：让模型自己生成数据并过滤。过滤标准可以是可验证的正确性（数学题有标准答案）或另一个模型的打分。DeepSeek-R1 和 Gemini Flash 都大量使用合成数据。

**从用户行为日志抽训练数据时要注意什么？**

用户日志里的数据有几个典型问题：

- **选择偏差**：用户更倾向于对好的回答点赞，差的回答可能直接被忽略，导致负样本不足
- **分布偏移**：线上用户的 query 分布和目标任务分布可能不一致
- **噪声**：用户行为信号（点赞、收藏）是弱监督，不等于回答质量高

实践中通常需要：过滤（去掉低质量回答）、去重（避免模板化样本过多）、再平衡（补充边界情况和困难样本），而不是直接拿来训练。

## 7.5 超参数调优实战

**面试官可能会问：** 微调时超参数怎么选？学习率、batch size、warmup 这些参数的经验值是什么？

**回答要点：**

超参数调优是微调成功的关键，但不是盲目试错，而是有经验规律可循。

**1. 学习率（Learning Rate）**

LoRA 微调和全量微调的学习率范围完全不同：

- **全量微调**：1e-5 到 5e-5（比预训练小 10 倍）
- **LoRA 微调**：1e-4 到 1e-3（比全量微调大 10 倍）

原因：LoRA 只训练少量参数，需要更大的学习率才能有效更新。

**调优策略**：
- 先用中间值（全量 3e-5，LoRA 5e-4）跑一个 epoch
- Loss 下降太慢 → 增大学习率
- Loss 震荡或 NaN → 减小学习率

**2. Warmup Steps**

Warmup 是让学习率从 0 线性增长到目标值的过程，防止训练初期梯度过大。

经验值：
- **Warmup steps = 总训练步数的 3%~10%**
- 小数据集（< 1000 样本）：100~500 steps
- 大数据集（> 10000 样本）：1000~3000 steps

**3. Batch Size 与 Gradient Accumulation**

实际 batch size = `per_device_batch_size × num_gpus × gradient_accumulation_steps`

经验值：
- **全量微调**：总 batch size 32~128
- **LoRA 微调**：总 batch size 8~64（更小也能收敛）

显存不够时的策略：
- 降低 `per_device_batch_size` 到 1 或 2
- 增大 `gradient_accumulation_steps` 来补偿（如 16 或 32）

**4. Epoch 数量**

- **全量微调**：1~3 epochs（过多容易过拟合）
- **LoRA 微调**：3~10 epochs（PEFT 不容易过拟合）

判断标准：验证集 loss 不再下降或开始上升时停止。

**深入追问：** 为什么 LoRA 的学习率要比全量微调大 10 倍？

**回答要点：**

LoRA 的有效学习率 = `learning_rate × (lora_alpha / lora_r)`。

假设 `lora_alpha=16, lora_r=8`，有效缩放因子是 2。如果学习率设为 5e-4，实际更新强度是 `5e-4 × 2 = 1e-3`。

但这只是 LoRA 参数的更新强度，由于 LoRA 参数量只有全量的 0.1%~1%，为了让这少量参数产生足够的影响，需要更激进的学习率。

另一个原因：全量微调会破坏预训练权重，需要小心翼翼；LoRA 只更新新增参数，原始权重被冻结保护，可以更大胆。

## 8. 模型选型框架

**面试官可能会问：** 在做一个 AI 应用时，你怎么选择基础模型？有哪些关键维度？

**回答要点：**

模型选型没有标准答案，但有一套系统的决策框架，沿三个维度分析：

### 能力维度

- **参数量 vs 任务复杂度**：简单任务（分类、抽取、简单问答）用 7B~13B 通常够用；复杂推理（数学、代码、多步分析）才需要 70B+；多模态任务额外考虑视觉编码器
- **上下文窗口**：长文档处理、多轮对话、大量工具 schema，需要 32K+ 窗口；实际可用窗口和名义窗口不一样，Lost in the Middle 的影响需要测试
- **特定能力**：代码能力（HumanEval）、中文能力（CMMLU）、工具调用能力（Berkeley Function Calling Leaderboard）要分别评测

### 工程维度

- **推理成本**：参数量直接决定显存需求和推理速度。生产环境中，70B 模型单路 throughput 比 7B 低一个数量级，成本高出 10x
- **量化友好性**：是否支持 AWQ/GPTQ/GGUF 量化，量化后精度损失有多少
- **生态和框架支持**：是否有 vLLM、TGI 支持，是否有活跃的社区和微调案例

### 合规维度

- **License**：Llama 3（限制商用规模）、Mistral（Apache 2.0）、Qwen（可商用）、GPT-4（闭源 API）——不同场景需求不同
- **数据合规**：私有部署 vs 云 API 调用，数据是否能出境

**决策流程**：先确定任务类型和复杂度（决定基础能力要求）→ 确定部署形式和成本预算（决定参数量上限）→ 在候选模型上跑领域 benchmark（实测优先于榜单排名）→ 确认 License 和合规要求。

**深入追问：** MoE（Mixture of Experts）到底在解决什么问题？为什么参数变大了，算力反而可能更省？

**回答要点：**

MoE 的核心是**参数稀疏激活**：总参数量可以很大，但每个 token 只激活少量专家（Top-K）。例如有 $E$ 个专家，每个 token 只走 $k$ 个专家，活跃参数比例是：

$$\text{Active Ratio} = \frac{k}{E}$$

这意味着：模型总容量（参数量）随 $E$ 增长，但单 token 计算主要随 $k$ 增长。直觉上就是"参数很多，但每次只用一小部分"。

MoE 的关键组件是 Router（门控网络），它根据 token 表示输出专家分数并选择 Top-K 专家。工程上最大的挑战不是"能不能跑起来"，而是**负载均衡**：如果 Router 总把 token 路由到少数几个专家，会出现热点专家过载、其他专家闲置，训练不稳定。Switch Transformer 常用辅助负载均衡损失来约束路由分布：

$$L_{aux} = E \sum_{i=1}^{E} f_i \cdot p_i$$

其中 $f_i$ 是第 $i$ 个专家实际接收 token 的比例，$p_i$ 是 Router 给该专家的平均概率。

另一个关键超参是 Capacity Factor（容量系数），单专家可接收 token 上限通常设为：

$$\text{capacity} = \left\lceil \frac{T}{E} \cdot CF \right\rceil$$

其中 $T$ 是 batch 内 token 数，$CF$ 一般大于 1。CF 太小会丢 token，CF 太大又会降低稀疏计算收益。

**面试一句话总结**：MoE 不是"白嫖参数"，而是用路由和稀疏激活把"容量"和"计算"解耦；收益来自 Top-K 稀疏计算，难点在路由均衡和分布式通信。

## 9. 项目深挖

**面试官可能会问：** 介绍一个你做过的微调项目，为什么选择微调而不是 RAG 或 Prompt？如果重做，最值得优先改哪一层？

**回答要点：**

微调项目的讲法核心是讲清楚**为什么选微调而不是 RAG 或 Prompt**。这是面试官最想听的判断依据，而不是你用了什么框架。

标准叙事结构：
1. **业务问题**：模型在什么任务上表现不够好（具体说问题表现）
2. **根因分析**：是知识问题还是能力问题？是格式问题还是推理问题？
3. **方案选择**：为什么选微调（能力问题，RAG 解决不了；知识问题则考虑 RAG 优先）
4. **数据构造**：怎么收集、怎么过滤、质量怎么保证（这里要有细节）
5. **训练方案**：LoRA 还是全量，秩怎么选，学习率怎么调，混入多少通用数据防遗忘
6. **评测体系**：目标任务 + hold-out + 通用基准三层评测
7. **上线效果和 bad case**：线上 A/B 测试，发现了哪些问题，怎么迭代

**典型 bad case** 有几类：

- **通用能力遗忘**：微调后模型在目标任务上变好了，但在其他任务上退化了。根因通常是训练数据多样性不够或学习率太高。修复：加通用数据混合，降低学习率。
- **过拟合训练格式**：模型学会了训练数据的格式，但泛化到新 query 时格式崩掉。根因：训练样本多样性不足，边界 case 覆盖差。修复：扩充训练集多样性，加难例挖掘。
- **数据污染**：训练数据里有低质量样本，模型学到了错误的行为模式。修复：严格数据过滤 pipeline。
- **LoRA 秩不够**：复杂任务用了太小的 r，模型表达能力不足。修复：增大秩或增加 LoRA 覆盖层。

**如果重做，最值得优先改哪一层**：几乎所有微调项目的瓶颈都在数据，而不是训练方案。数据质量差，换更大的秩、更长的训练、更复杂的方案都救不了。先把数据治理好（过滤、去重、补充边界样本、加 CoT 推理链），再考虑训练方案的优化。

**深入追问：** 怎么判断微调是否真的有效，而不是只是在训练集上过拟合？

**回答要点：**

评测要分三层：

- **第一层**：目标任务的离线评测集（和训练数据同分布），验证基本能力
- **第二层**：hold-out 测试集（和训练数据不同分布），验证泛化能力
- **第三层**：通用能力基准（MMLU、C-Eval、HumanEval），验证有没有遗忘

只看第一层是不够的，很容易把过拟合误判为真正的提升。线上 A/B 测试是最终验证，但要配合足够的样本量和统计显著性检验，不能靠感觉判断。

## 10. 面试终极追问 Q&A

### Q1: 全量微调（Full Fine-tuning）vs LoRA，什么时候选哪个？

**A:**

| 维度 | 全量微调 | LoRA |
|------|---------|------|
| 参数更新 | 所有参数 | 0.1%~1% 新增参数 |
| 显存需求 | 极高（需存所有梯度和优化器状态） | 显著降低 |
| 效果上限 | 理论最优 | 接近全量（90%+） |
| 遗忘风险 | 高（容易覆盖预训练知识） | 低（原始权重冻结） |
| 推理延迟 | 无额外延迟 | 可合并，无额外延迟 |
| 适用场景 | 大规模领域适应、有充足计算资源 | 资源受限、需要多任务并行、快速迭代 |

实践中，**99% 的业务微调场景用 LoRA 都够**。全量微调通常只在做大规模 domain adaptation、或者要训练 RM（Reward Model）时才考虑。

---

### Q2: 为什么 Instruction Tuning 能让模型"听话"？

**A:**

Instruction Tuning 本质上是在教模型一种元模式：**"给定指令，生成符合指令意图的回答"**。预训练数据以 web text 为主，模型学的是"续写"；Instruction Tuning 的数据是（指令, 回答）对，教的是"遵循指令"。

关键机制：模型在大量多样化的（指令, 回答）对上训练后，会形成泛化的指令遵循能力——即使遇到训练时没见过的指令格式，也能基于已学到的模式来响应。这种泛化能力是 GPT-3→InstructGPT、LLaMA→Alpaca 质变的核心原因。

---

### Q3: DPO 为什么被称为"去掉 RM 的 RLHF"？

**A:**

DPO 从数学上消去了 RLHF 中 Reward Model 的显式存在。推导关键：RLHF 目标的最优 policy 可以用参考模型的对数比来表示 reward，代入 Bradley-Terry 排名模型后，得到一个只需要 policy 模型和参考模型、不需要单独训练 RM 的损失函数。

本质上，DPO 把 reward 函数"隐式地"编码进了 policy 模型的参数里，而不是用一个独立的 RM 模型来参数化它。

---

### Q4: 什么是 GRPO？和 PPO 的核心区别是什么？

**A:**

GRPO（Group Relative Policy Optimization）是 PPO 的简化版，核心简化是**去掉了 value model（critic）**。PPO 需要 critic 来估计每个状态的价值函数，GRPO 改用组内相对 reward 来估计优势：对同一问题采样 G 个回答，计算每个回答 reward 相对于组内均值的归一化偏差。

工程价值：从四个模型减为三个模型，显存需求和工程复杂度都大幅降低。DeepSeek-R1 使用 GRPO，验证了这种方法在推理任务上的有效性。

---

### Q5: 微调后的模型怎么部署？LoRA 能和基础模型合并吗？

**A:**

LoRA 支持两种部署方式：

1. **合并部署（推荐）**：将 LoRA 权重合并回基础模型（$W' = W + \frac{\alpha}{r} BA$），得到一个完整的全量模型，推理时零额外延迟。适合单任务固定部署。

2. **动态加载**：基础模型保持不变，根据不同任务动态加载不同的 LoRA 权重。适合多任务、多租户场景，可以大幅节省显存（多个任务共享同一份基础模型权重）。这是 **LoRA MoE（Mixture of LoRA Experts）** 的基础。

### Q6: 什么情况下需要 Reward Model，什么情况下可以用可验证 Reward？

**A:**

这是推理模型训练的核心设计决策：

- **Reward Model 适合**：主观评价任务（写作质量、安全性、有用性），没有客观标准答案，需要捕获人类偏好。代价是 RM 会过拟合（Reward Hacking），需要持续维护。

- **可验证 Reward 适合**：有客观标准的任务（数学题答案对错、代码执行是否通过测试、逻辑推理是否自洽）。这类 reward 信号准确、免标注、可自动化，是 DeepSeek-R1、AlphaCode 等模型的核心。

趋势：优先使用可验证 reward，在主观任务上才使用 RM，且尽量用 AI 作为打分模型（Constitutional AI、Reward Model as AI Judge）来降低人工标注成本。
