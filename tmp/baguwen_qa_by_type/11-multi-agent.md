# 多 Agent 系统面试知识点

这份文档以真实面试追问链路为主线整理，目标是让你从"为什么需要多 Agent"一路讲到"A2A 协议、失控风险、什么时候不值得做、项目落地"，每个环节都有具体技术细节和工程判断可以展开。

---

## 1. 单 Agent vs 多 Agent：什么时候需要多 Agent

**面试官可能会问：** 什么时候单 Agent 不够用？多 Agent 解决的核心问题是什么？

**回答要点：**

默认先单 Agent，只有当以下边界被明显触碰时，才值得考虑多 Agent：

- **上下文过载：** 复杂任务的中间状态、工具调用结果、历史记录会快速撑满上下文窗口，单 Agent 无法在一个窗口内处理超长任务链
- **能力边界差异大：** 不同子任务需要截然不同的专业能力（代码生成 vs 数据分析 vs 文档撰写），让一个通用 Agent 全部承担，效果往往不如专门配置的专家 Agent
- **并行效率瓶颈：** 单 Agent 是串行执行的，多个独立子任务必须排队，而多 Agent 可以并行处理，大幅缩短整体耗时

多 Agent 的核心价值是**职责解耦 + 上下文隔离 + 并行执行**。但一旦拆成多 Agent，通信、协调、同步和失控风险都会明显上升。这不是免费的收益，是用治理成本换任务弹性。

**深入追问：** 一句话给出选择原则。

**回答要点：**

能用单 Agent 解决，就别急着上多 Agent。多 Agent 应该是为了解决明确的边界问题，而不是为了显得架构更高级。很多项目上了多 Agent 之后，最难处理的问题反而是"怎么让它稳定"，而不是"怎么让它更强"。

## 2. 多 Agent 协作模式：Supervisor、Planner-Worker、Reviewer/Critic

**面试官可能会问：** 多 Agent 系统常见的协作模式有哪些？各自适合什么场景？

**回答要点：**

**Supervisor 模式（中心化调度）：** 一个 Supervisor Agent 负责理解总任务、分解子任务、分发给各个子 Agent，并汇总结果。子 Agent 只负责执行，不需要感知全局状态。优点是可控性强，任务流向清晰，容易调试；缺点是 Supervisor 成为单点瓶颈，如果 Supervisor 判断失误，整个任务链都会跑偏。

**Planner-Worker 模式：** Supervisor 的变体，Planner 专门负责任务规划和分解，Worker 专门负责执行。两者职责更清晰，Planner 可以用更强的模型，Worker 可以用更轻量的模型，成本更可控。适合任务结构相对清晰、可以一次性规划的场景。

**Reviewer/Critic 模式：** 在执行 Agent 之外加一个专门做质量审核的 Critic Agent，对输出做评估和反馈，驱动执行 Agent 修正。适合对输出质量要求高、允许多轮迭代的任务（代码审查、文档润色）。

**Router + Specialist 模式：** Router 根据用户输入判断应该交给哪个专家 Agent 处理，各 Specialist 只处理自己擅长的领域。适合意图分发场景（客服、问答系统），路由准确率是核心指标。

**深入追问：** 为什么很多多 Agent 系统最后都会回到"中心化调度"？

**回答要点：**

去中心化的 Agent 网络看起来更灵活，但实际上很难控制。Agent 之间的直接通信容易产生循环依赖、状态不一致和责任不清。中心化调度虽然有单点瓶颈，但任务流向清晰、状态集中管理、出问题时容易定位。生产系统更在意可控性下限，而不是理论灵活性上限。

## 3. A2A 协议与 Agent Card：定义、定位与能力发现

**面试官可能会问：** A2A 是什么？它和 MCP、Function Calling 有什么区别？Agent Card 又是什么？

**回答要点：**

A2A（Agent-to-Agent）是 Google 提出的 Agent 间通信协议，解决的核心问题是：不同 Agent 之间如何发现彼此的能力、如何传递任务和结果。在没有标准协议之前，多 Agent 系统的 Agent 间通信都是自定义实现，不同框架之间无法互通。

区分这三个概念的关键是看通信的两端是什么：
- **Function Calling：** 模型与函数之间的接口，模型决定调用哪个函数、传什么参数，函数返回结果
- **MCP（Model Context Protocol，Anthropic 提出）：** Agent 与工具/资源之间的协议，标准化了 Agent 如何发现和调用外部工具
- **A2A：** Agent 与 Agent 之间的协议，标准化了一个 Agent 如何发现另一个 Agent 的能力、如何委托任务、如何接收结果

三者的层次关系：Function Calling 是最底层的模型能力，MCP 标准化了工具接入，A2A 在更高层标准化了 Agent 间的协作。

**Agent Card** 是 A2A 协议里的能力描述文件，类似于 REST API 的 OpenAPI spec。它描述了这个 Agent 能做什么、接受什么类型的输入、返回什么格式的输出、支持哪些任务类型。Agent Card 的核心价值是让 Supervisor 或 Planner 在任务分解时动态查询和选择子 Agent，而不是硬编码依赖关系。新增一个 Agent 只需要注册它的 Agent Card，现有调度逻辑不需要修改。

**深入追问：** A2A 和 MCP 的本质区别是什么？

**回答要点：**

MCP 解决的是"Agent 怎么用工具"，工具是被动的、无状态的执行单元。A2A 解决的是"Agent 怎么和另一个 Agent 协作"，对端是有自主决策能力的 Agent，可以有自己的状态、记忆和工具。MCP 更像 RPC，A2A 更像 Agent 间的任务委托协议。两者不互斥，一个 Agent 可以同时通过 MCP 调用工具、通过 A2A 委托子 Agent。

## 4. 通信方式、状态共享与记忆协同

**面试官可能会问：** 多 Agent 系统里，Agent 之间怎么通信？同步 vs 异步怎么选？共享状态有什么风险？

**回答要点：**

**通信方式：**
- **同步通信：** 调用方等待被调用方返回结果再继续。适合强依赖场景（下一步必须用上一步的结果），但会阻塞整个链路，一个 Agent 慢会拖慢所有人
- **异步通信：** 调用方发出任务后继续执行，通过回调或轮询获取结果。适合并行子任务，但需要处理结果汇聚和超时逻辑，复杂度更高

**状态共享的三种方式：**
- **消息传递：** Agent 之间通过结构化消息传递中间结果，每个 Agent 只持有自己的局部状态，最干净，但需要设计好消息格式
- **共享黑板（Blackboard）：** 所有 Agent 读写同一个共享状态存储，适合频繁协调的场景，但并发写入容易产生冲突
- **Supervisor 中转：** 子 Agent 不直接通信，所有状态更新都经过 Supervisor，可控性最强但吞吐量受限

**共享状态的核心风险：** 状态污染——一个 Agent 写入了错误的中间结果，其他 Agent 读取后会把错误当作事实继续推理，错误在系统内放大而不是被纠正。多个 Agent 并发修改同一状态时，如果没有锁机制，最终状态可能是不确定的。

**深入追问：** 多 Agent 系统里的记忆怎么设计？

**回答要点：**

通常分三层：短期记忆是每个 Agent 自己的上下文窗口，只在当前任务内有效；工作记忆是任务级别的共享状态，任务结束后清除；长期记忆是跨任务持久化的知识，通常存在向量数据库或结构化存储里。多 Agent 场景下，长期记忆的写入权限需要谨慎控制，避免一个 Agent 的错误推断污染全局知识库。

## 5. 多 Agent 为什么更容易失控

**面试官可能会问：** 多 Agent 系统有哪些典型的失控场景？怎么防范？

**回答要点：**

多 Agent 系统的失控风险比单 Agent 高出不止一个量级，主要有以下几类典型场景。

死循环是最常见的问题：Agent A 调用 Agent B 处理某个子任务，Agent B 判断需要 Agent A 的能力来辅助，于是调回 Agent A，形成循环调用。如果没有调用深度限制或循环检测机制，系统会一直运行直到超时或耗尽资源。

成本失控是多 Agent 特有的风险：每个 Agent 都可能调用大模型，任务分解越细，模型调用次数越多，成本呈指数级上升。一个看似简单的用户请求，经过多层分解后可能触发几十次 LLM 调用。

错误放大是最难察觉的问题：上游 Agent 的推理错误或工具调用失败，会被下游 Agent 当作可信的中间结果继续处理。单 Agent 的错误通常在一次对话内暴露，多 Agent 的错误可能在多个 Agent 的处理后才显现，此时已经很难追溯根因。

责任不清也是工程上的痛点：任务失败时，很难判断是哪个 Agent 出了问题，日志分散在多个 Agent 的执行记录里，调试成本极高。

**深入追问：** 有哪些工程手段可以降低多 Agent 失控的风险？

**回答要点：**

几个实用的防护措施：设置最大调用深度和最大 token 预算，超出后强制终止；在 Agent 间通信中加入调用链追踪（类似分布式 tracing），每个任务有唯一 trace ID；对关键决策节点加入人工审核（Human-in-the-loop）；对 Agent 的工具调用权限做最小化授权，避免子 Agent 有超出其职责范围的操作权限。

## 6. 什么时候不值得做多 Agent

**面试官可能会问：** 什么场景下根本不值得上多 Agent？怎么判断是过度设计？

**回答要点：**

这是一个很重要的反向判断题，能答好这个问题说明你真的做过系统，而不是只会堆架构。

**不值得上多 Agent 的典型场景：**

- **任务链路固定、步骤可以预先枚举：** 如果你能在设计阶段把所有步骤画成流程图，用 Workflow（DAG 编排）就够了。多 Agent 的动态分工在这里是纯开销，不是收益
- **只是为了"看起来高级"：** 把一个单 Agent 机械地拆成多个 Agent，但它们没有真实的职责边界，还共享大量上下文，这种拆法只会增加 token 消耗和协调成本，不会提升效果
- **团队没有多 Agent 治理能力：** 多 Agent 系统的调试、可观测性、成本控制都比单 Agent 复杂得多。如果团队连单 Agent 的 trace 和评测都没做好，上多 Agent 只会让问题更难排查
- **任务对延迟敏感：** 多 Agent 的协调通信会增加整体延迟，如果业务对响应时间要求严格，多 Agent 的并行收益可能被通信开销抵消

**判断标准：** 先问"如果不上多 Agent，最大的损失是什么"。如果答案是"没有损失，只是架构看起来更复杂"，那就不值得上。

**深入追问：** 有些项目明明上了多 Agent，最后却很难证明这是对的架构选择，为什么？

**回答要点：**

因为多 Agent 的收益很难量化，但成本很容易感受到。并行效率提升了多少、职责解耦带来了多少维护收益，这些很难用数字说清楚。但 token 成本增加了多少、调试时间增加了多少、系统稳定性下降了多少，这些是实实在在的。很多团队在没有明确 baseline 对比的情况下上了多 Agent，事后很难说清楚到底值不值。

## 7. 多 Agent 项目深挖

**面试官可能会问：** 介绍一个你做过的多 Agent 项目，为什么选择多 Agent？遇到了什么问题？

**回答要点：**

最稳的讲法按这条链路展开：

> 业务背景（为什么需要多 Agent，单 Agent 的瓶颈在哪）→ 角色划分（怎么分工，边界怎么定）→ 通信设计（同步还是异步，状态怎么共享）→ 调度与裁决（谁来分发任务，冲突怎么处理）→ 失控防护（怎么防死循环和成本失控）→ 上线效果（遇到了什么 bad case）

**典型 bad case 类型：**

- **格式不对齐：** 上游 Agent 输出的格式和下游 Agent 期望的输入格式不一致，是最常见的集成问题。根因通常是 Agent Card 里的 schema 定义不够严格，或者 Supervisor 层没有做格式校验
- **死循环：** Agent A 调用 Agent B，Agent B 又调回 Agent A，系统一直运行直到超时。根因是没有调用深度限制和循环检测机制
- **成本爆炸：** 一个用户请求经过多层分解后触发了几十次 LLM 调用，成本远超预期。根因是任务分解粒度太细，没有设置 token 预算上限
- **错误放大：** 上游 Agent 的一个小错误，被下游多个 Agent 当作事实继续处理，最终输出完全错误但每一步看起来都合理。这类 bad case 最难排查，必须靠完整的调用链追踪才能定位

**深入追问：** 如果重做一次多 Agent 系统，你最想先改哪一层？

**回答要点：**

通常优先改可观测性层，而不是先改 Agent 的能力或协作逻辑。多 Agent 系统最大的工程痛点不是"Agent 不够聪明"，而是"出了问题不知道在哪"。从一开始就建立结构化日志、调用链追踪（每个任务有唯一 trace ID）和 Agent 状态快照，后续所有优化都会更有依据。

其次是任务边界设计——很多多 Agent 问题的根因是角色边界模糊，导致 Agent 之间职责重叠或互相依赖。重新梳理哪些子任务真的需要独立 Agent，哪些可以合并或用 Workflow 固定，往往比调整模型或 prompt 更有效。
