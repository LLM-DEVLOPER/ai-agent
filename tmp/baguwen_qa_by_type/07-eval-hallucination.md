# 07 评测与幻觉治理

## 1. 为什么 AI 应用一定要做评测

**面试官可能会问：** 为什么 RAG 或 Agent 系统上线前后都要做评测？靠人工试用不够吗？

**回答要点：**

AI 应用的核心问题是输出不确定性——同一个问题在不同上下文、不同模型版本下可能给出截然不同的答案，而且错误往往是"看起来正确"的。靠人工试用只能覆盖少量 case，无法系统性地发现边界问题，也无法量化"这次改动让效果变好了还是变差了"。评测的本质是把主观感受转化为可追踪的指标，让每次迭代都有数据支撑。

另一个现实原因是回归保障。每次更换 Prompt、调整检索策略、升级模型版本，都可能引入退化。没有评测集就没有回归基线，团队只能靠感觉判断，这在生产环境里是不可接受的。

**深入追问：** 评测集应该怎么构建？从哪里来？

**回答要点：**

评测集的来源通常有三条路：第一是从线上真实流量中采样，这类数据最贴近真实分布，但需要人工标注答案；第二是让领域专家手工编写问题和参考答案，覆盖高风险场景；第三是用 LLM 自动生成问题，再人工审核。实践中三者结合效果最好——用线上数据保证分布真实性，用专家数据覆盖边界 case，用自动生成快速扩充数量。评测集要定期更新，因为用户行为会漂移，旧的评测集会逐渐失去代表性。

## 2. 评测到底在评什么——分层评测体系

**面试官可能会问：** RAG 系统的评测应该怎么分层？每层评什么？

**回答要点：**

RAG 系统的评测天然对应它的三层架构：召回层、排序层、生成层，每层的评测目标和指标都不一样。

召回层评测的是检索系统能不能把相关文档找出来。核心指标是 Recall@K（前 K 个结果里有多少相关文档）、MRR（Mean Reciprocal Rank，第一个相关文档排在第几位的倒数均值）和 NDCG（Normalized Discounted Cumulative Gain，考虑位置权重的排序质量指标）。这层的问题通常是向量化质量差、分块策略不合理或者查询改写没做好。

排序层评测的是重排序模型有没有把最相关的文档排到前面。这层通常用 Precision@K 和 NDCG 来衡量，关注的是排序的精准度而不只是召回率。

生成层评测最复杂，需要同时评估答案的多个维度。Faithfulness 检查答案里的每个声明是否都能在检索到的上下文中找到依据，防止模型"编造"内容；Answer Relevance 检查答案是否真正回答了用户的问题，而不是答非所问；Context Utilization（也叫 Context Recall）检查模型有没有充分利用检索到的上下文，而不是忽略关键信息。

**深入追问：** 这三层里哪层最难评？

**回答要点：**

生成层最难评，因为它没有唯一正确答案。召回层和排序层都有明确的相关性标注可以对比，但生成层的答案可以有很多种正确表达方式。Faithfulness 相对好评，因为可以逐句核查；但 Answer Relevance 就很主观——什么叫"真正回答了问题"，不同人的判断可能不一样。这也是为什么 LLM-as-a-judge 在生成层评测里越来越流行。

## 3. RAGAS 框架与自动化评测

**面试官可能会问：** RAGAS 是什么？它解决了什么问题？

**回答要点：**

RAGAS 是一个专门为 RAG 系统设计的自动化评测框架，核心思路是用 LLM 来充当评委，对检索和生成的各个维度打分，从而实现无需人工标注的自动化评测。

它的四个核心指标各有侧重。Faithfulness 评估答案中每个陈述是否有上下文支撑，做法是把答案拆成若干原子声明，逐一判断能否从检索文档中推导出来，最后算比例。Answer Relevance 评估答案和问题的相关性，做法是让 LLM 根据答案反向生成几个问题，再计算这些问题和原始问题的语义相似度。Context Precision 评估检索到的文档里有多少是真正有用的，衡量检索的精准度。Context Recall 评估参考答案中的信息有多少能在检索文档中找到，衡量检索的覆盖度。

RAGAS 的价值在于它把原本需要大量人工标注的评测工作自动化了，可以在 CI/CD 流程里集成，每次代码变更都自动跑评测。TruLens 是另一个类似的框架，提供了更细粒度的链路追踪和可视化能力。

**深入追问：** RAGAS 有什么局限性？

**回答要点：**

RAGAS 最大的局限是它本身依赖 LLM 来评分，所以评分质量受底层模型能力限制。如果用一个较弱的模型做评委，评分结果可能不可靠。另外 RAGAS 的 Faithfulness 指标在处理长文档或复杂推理链时容易出现误判，因为它是逐句核查而不是整体理解。Context Precision 和 Recall 需要参考答案作为 ground truth，在没有标注数据的场景下无法使用。实践中 RAGAS 适合做快速的自动化回归，但对于高风险场景还是需要人工复核。

## 4. LLM-as-a-judge 的优势与风险

**面试官可能会问：** LLM-as-a-judge 是什么思路？为什么越来越流行？

**回答要点：**

LLM-as-a-judge 是用大语言模型来替代人工评委，对生成结果进行打分或排序。它流行的核心原因是规模化——人工评测成本高、速度慢，而 LLM 可以在几分钟内处理数千条样本，成本只有人工的几十分之一。对于需要语义理解的评测任务（比如判断答案是否忠实于上下文），LLM 的表现往往比基于规则的方法好得多。

实践中常见的做法是给 LLM 一个评分 Prompt，包含评分标准、待评内容和参考答案，让它输出分数和理由。有时候会用两个 LLM 互相评分（Pairwise 比较），判断哪个答案更好，这比绝对打分更稳定。

**深入追问：** LLM-as-a-judge 有哪些已知的偏见和风险？

**回答要点：**

LLM-as-a-judge 有几个系统性偏见需要注意。第一是位置偏见，当做 Pairwise 比较时，模型倾向于认为排在前面的答案更好，需要通过交换顺序来消除。第二是长度偏见，模型倾向于给更长、更详细的答案打高分，即使内容质量并不更好。第三是自我偏好，如果用 GPT-4 做评委，它可能倾向于给 GPT-4 生成的答案打高分。第四是评委本身的知识局限，如果评委模型对某个领域不熟悉，它的评分就不可靠。缓解这些问题的方法包括：设计更细致的评分标准、多次采样取平均、用多个不同模型做评委然后投票。

## 5. 什么是幻觉，幻觉从哪里来

**面试官可能会问：** 幻觉到底是什么？所有"错误答案"都算幻觉吗？

**回答要点：**

幻觉在 NLP 领域有严格定义：模型生成了看起来合理但实际上与事实或给定上下文不符的内容。关键词是"看起来合理"——如果模型输出明显的乱码，那不叫幻觉，叫错误。幻觉的危险在于它的欺骗性，用户很难察觉。

不是所有错误答案都算幻觉。如果模型说"我不知道"但实际上知识库里有答案，这是召回失败，不是幻觉。如果模型理解错了问题意图，给出了一个对另一个问题的正确答案，这是指令误解，也不完全是幻觉。幻觉特指模型"编造"了不存在的信息，或者把上下文中没有的内容当作事实陈述出来。

**深入追问：** 幻觉通常来自哪几层？怎么区分？

**回答要点：**

幻觉的来源可以按链路层来分类，这对定位和修复都很重要。

知识缺失型幻觉发生在模型参数知识不足时，模型用"合理猜测"填补空白，比如编造一个不存在的论文引用。这类幻觉在 RAG 场景下可以通过强制要求"只基于检索内容回答"来缓解。

上下文利用失败型幻觉是检索到了正确文档，但模型没有正确利用，反而用了自己的参数知识覆盖了上下文信息。这类问题通常出在 Prompt 设计上，需要更明确地指示模型优先使用上下文。

推理错误型幻觉是模型在多步推理中出现逻辑跳跃，中间步骤的错误累积导致最终答案偏离事实。这类问题在 Agent 场景下更常见，需要通过 Chain-of-Thought 和中间步骤验证来缓解。

指令误解型幻觉是模型对问题的理解和用户意图不一致，给出了一个"对错误问题的正确答案"。这类问题需要通过查询改写和意图识别来处理。

## 6. Bad Case 管理与评测闭环

**面试官可能会问：** 线上出现 bad case 之后，你们是怎么管理和处理的？

**回答要点：**

Bad case 管理的核心是分类和闭环，而不只是修复单个问题。首先要按链路层对 bad case 分类：检索层问题（相关文档没召回、召回了不相关文档）、排序层问题（相关文档排名靠后）、生成层问题（有正确上下文但答案仍然错误）。这个分类决定了修复方向——检索问题改向量化或分块策略，排序问题调重排序模型，生成问题改 Prompt 或换模型。

分类之后要建立回流机制。线上的 bad case 经过人工标注后应该进入评测集，成为下一轮迭代的回归用例。这样评测集会随着时间积累越来越多的真实边界 case，评测的覆盖度也会越来越高。

还有一个容易忽视的点是 bad case 的优先级排序。不是所有 bad case 都值得立刻修复，要按照影响频率和严重程度来排序。高频出现的幻觉问题优先于低频的边界 case，涉及事实错误的问题优先于答案不够完整的问题。

**深入追问：** 如果一个 bad case 在检索层和生成层都有问题，怎么判断根因？

**回答要点：**

这种情况需要分步骤隔离。先固定检索结果，手动把正确文档注入上下文，看模型能不能给出正确答案。如果能，说明根因在检索层；如果不能，说明生成层本身有问题。这个"oracle 检索"测试是定位根因最直接的方法。实践中可以在评测框架里把这个步骤自动化，对每个 bad case 都跑一遍，自动判断问题出在哪一层。

## 7. 线上效果退化与监控

**面试官可能会问：** 系统上线后效果退化了，怎么发现、怎么定位？

**回答要点：**

线上效果退化的发现依赖两类信号：一是指标监控，二是用户反馈。指标监控需要在线上持续跑一个"金标准"评测集，定期（比如每天）自动评测并记录分数变化，一旦某个指标下降超过阈值就触发告警。用户反馈包括显式的踩/赞操作和隐式的行为信号（比如用户对答案追问"你确定吗"往往意味着答案可信度低）。

定位退化原因需要排查几个常见来源：模型版本变更（API 提供商悄悄更新了模型）、知识库内容变化（文档更新导致检索结果变化）、流量分布漂移（用户开始问以前没见过的问题类型）、基础设施变化（向量数据库索引重建导致召回结果变化）。

实践中最有效的方法是在每次变更时做 A/B 对比，把新旧版本的输出都记录下来，用 LLM-as-a-judge 做 Pairwise 比较，快速判断变更方向是否正确。

**深入追问：** 如果没有 ground truth 标注，线上能做哪些评测？

**回答要点：**

没有 ground truth 时可以用几种无监督的方法。第一是 Faithfulness 检测，不需要参考答案，只需要检查答案是否能从检索文档中推导出来。第二是答案一致性检测，对同一个问题多次采样，如果答案差异很大说明模型不稳定。第三是置信度校准，让模型输出置信度，观察置信度和实际准确率的相关性。第四是用户行为作为隐式标注，把用户的追问、纠正、放弃等行为作为负向信号。这些方法都不如有 ground truth 的评测精确，但在冷启动阶段或标注资源有限时很有价值。

## 8. 幻觉治理与产品侧缓解

**面试官可能会问：** 除了技术手段，产品层面有哪些方式可以缓解幻觉的危害？

**回答要点：**

技术上无法完全消除幻觉，所以产品侧的缓解策略同样重要。最常见的做法是引用溯源——在答案旁边展示来源文档和具体段落，让用户可以自行核实。这不能防止幻觉发生，但能让用户有能力发现幻觉，同时也给模型一个"不要编造"的隐性约束。

另一个有效策略是不确定性表达，让模型在置信度低时主动说"我不确定"或"根据我的知识库，但建议核实"，而不是用同样的语气陈述确定和不确定的内容。这需要在 Prompt 里明确指示，并在评测时把"该说不确定时说了不确定"作为一个正向指标。

对于高风险场景（医疗、法律、金融），还可以加一层人工审核或规则过滤，对涉及关键事实的陈述做二次验证。拒绝回答也是一种合理的产品策略——当检索结果置信度低于阈值时，直接告诉用户"我没有找到可靠信息"比给出一个可能错误的答案要好。

**深入追问：** 项目里遇到过最难处理的幻觉 bad case 是什么类型？如果重做会先改哪层？

**回答要点：**

最难处理的通常是"上下文存在但模型选择性忽略"的幻觉。检索层已经把正确文档召回了，排在前几位，但模型在生成时用了自己的参数知识覆盖了上下文，给出了一个"听起来更权威"但实际上错误的答案。这类问题很难通过改检索来解决，因为检索本身没问题，根因在模型对上下文的权重分配。

如果重做，会优先改生成层的 Prompt 设计，明确加入"如果上下文和你的训练知识冲突，以上下文为准"的指令，并在评测集里专门构造这类冲突 case 来验证效果。其次会在召回层加入 Faithfulness 过滤，对生成结果做事后验证，把 Faithfulness 分数低于阈值的答案标记出来走人工审核流程，而不是直接返回给用户。
