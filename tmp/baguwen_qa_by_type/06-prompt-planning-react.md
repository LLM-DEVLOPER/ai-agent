# Prompt、规划与 ReAct 面试知识点

这份文档按面试真实追问链路整理，从"Prompt 在数学上是什么"一路讲到"ReAct 怎么落地、Prompt Injection 怎么防、项目里怎么做取舍"，每个环节都有具体技术细节和工程判断可以展开。

---

## 1. Prompt 的本质与数学定义

**面试官可能会问：** Prompt Engineering 到底是什么？Prompt 在数学上是在做什么？

**回答要点：**

从数学角度看，LLM 本质上是在建模条件概率分布 P(y | x)，其中 x 是输入，y 是输出。Prompt Engineering 做的事情是：在不改变模型参数 θ 的前提下，通过设计输入 x = (p, q)（p 是 Prompt，q 是用户 query），来改变模型的输出分布 P(y | p, q; θ)。换句话说，Prompt 是在**条件概率层面**控制模型行为，而不是在参数层面。

这个视角解释了很多工程现象：为什么 Prompt 的措辞、顺序、结构会显著影响输出——因为它们直接改变了模型的条件分布。为什么 Few-shot 有效——因为示例在上下文里提供了额外的条件信息，把模型的输出分布"拉"向示例所展示的模式。为什么 System Prompt 优先级更高——因为它在上下文里位置靠前，对后续 token 的条件概率影响更大。

在 Agent 系统里，Prompt 不只是一次性的输入，而是整条链路的控制信号——规划怎么拆、工具怎么选、什么时候停，很多时候都靠 Prompt 来约束。

**深入追问：** 为什么很多线上问题最后都会追到 Prompt 层？

**回答要点：**

因为 Prompt 是模型行为的直接输入，任何一层的问题——检索结果没被利用、工具被误调、输出格式不稳定——最终都会在 Prompt 层有所体现。但这不代表所有问题都是 Prompt 的问题，很多时候 Prompt 只是放大了上游的问题，比如检索质量差、工具 schema 不清晰。排障时要先分清"Prompt 本身写错了"还是"Prompt 在放大其他层的问题"。

## 2. Prompt 的分层与组织方式

**面试官可能会问：** System Prompt、Developer Prompt、User Prompt 的边界是什么？为什么 Prompt 要做分层？

**回答要点：**

三层 Prompt 的职责分工是：**System Prompt** 定义模型的角色、能力边界、全局约束和安全规则，是最高优先级的控制层，通常由平台或产品方写死；**Developer Prompt**（也叫 Instruction Prompt）注入任务上下文、工具描述、检索结果、业务规则，是运行时动态组装的部分；**User Prompt** 是用户的实际输入，是最不可控的一层。

分层的价值在于：职责清晰之后，调试和迭代才能落层。如果所有内容都揉在一个大 Prompt 里，改一处就可能影响全局，而且很难判断问题出在哪一层。模块化 Prompt 还能让不同部分独立版本管理，比如 System Prompt 改了不影响工具描述，工具描述改了不影响用户侧约束。

一个典型的 Agent Prompt 结构通常是：角色定义 + 能力边界 + 工具列表（含 schema）+ 检索上下文 + 历史轨迹 + 当前任务 + 输出格式约束 + 拒答规则。这些部分的顺序和边界都会影响模型的行为——越靠前的内容对后续 token 的条件概率影响越大，这也是为什么很多系统都会尽量保持 System Prompt 稳定不变。

**深入追问：** 为什么很多系统都要把 Prompt 做成模块化配置？

**回答要点：**

模板是静态的、可复用的部分，比如角色定义、格式约束、拒答规则；运行时上下文是动态注入的，比如检索结果、工具返回、当前用户输入。两者混在一起最常见的问题是：模板里夹带了运行时数据，导致模板无法复用；或者运行时数据没有清晰的注入位置，导致上下文污染。好的做法是把模板做成带占位符的结构，运行时只填充动态部分，边界清楚，也方便回放和调试。模块化还有一个工程收益：稳定的前缀部分可以充分利用 Prefix Cache，降低推理成本。

## 3. Zero-shot、Few-shot 与示例策略

**面试官可能会问：** Zero-shot 和 Few-shot 的区别是什么？什么时候值得引入 Few-shot？示例怎么选？

**回答要点：**

**Zero-shot** 是直接给任务描述，不提供示例，依赖模型的泛化能力；**Few-shot** 是在 Prompt 里提供若干输入-输出示例，帮助模型理解任务格式、推理风格和边界条件。

Few-shot 的核心价值不是"教模型新知识"，而是**校准模型的输出分布**——通过示例把 P(y | p, q) 拉向期望的模式。对于格式要求严格（如结构化 JSON 输出）、任务定义模糊（如主观分类）、或者模型容易走偏的场景，Few-shot 通常能显著提升稳定性。

示例选择的原则：**质量优先于数量**。3-5 个高质量示例通常比 20 个随手挑的示例效果更好。示例应该覆盖典型场景、边界情况和容易出错的 case，而不是只挑最简单的正例。反例（展示错误做法并纠正）在某些场景下也很有价值，尤其是模型容易犯的系统性错误。示例的位置也有讲究：通常放在任务描述之后、用户输入之前，让模型先看到示例再处理当前输入。

**深入追问：** 示例选择为什么会直接影响效果稳定性？

**回答要点：**

因为模型会从示例里归纳模式，如果示例本身有偏差（比如全是简单 case、或者示例风格不一致），模型归纳出来的模式也会有偏差。最常见的问题是示例覆盖面不够，导致模型在边界情况下行为不稳定。好的示例集应该像一个小型评测集，覆盖典型、边界和容易出错的三类情况。

## 4. CoT 与 Self-Consistency

**面试官可能会问：** 什么是 CoT？它为什么对复杂推理有效？Self-Consistency 又是什么？

**回答要点：**

**CoT（Chain-of-Thought，思维链）** 是 Wei et al. 2022 年提出的技术，核心思想是在 Prompt 里引导模型逐步展示推理过程，而不是直接给出答案。形式上，CoT 把输出从直接生成答案 y，变成先生成一系列中间推理步骤 z₁, z₂, ..., zₙ，再生成最终答案 y：

P(y | x) → P(y | zₙ, ..., z₁, x) × P(zₙ | zₙ₋₁, ..., z₁, x) × ... × P(z₁ | x)

CoT 为什么有效？根本原因在于 LLM 的 token 生成是**自回归**的——每个 token 的生成都以前面所有 token 为条件。显式的推理步骤相当于给模型提供了"工作记忆"，让它能在生成过程中积累中间结论，而不是强迫它在一次 forward pass 里完成所有推理。这对数学推理、多步逻辑、代码分析等任务效果尤其明显——这类任务的答案依赖于中间步骤，没有中间步骤直接跳结论，模型很容易出错。

CoT 有两种形式：**Zero-shot CoT**（在 Prompt 末尾加"Let's think step by step"这类引导语）和 **Few-shot CoT**（提供带推理过程的示例）。后者通常效果更稳定，因为示例里的推理风格会直接影响模型的推理路径。

**Self-Consistency** 是 Wang et al. 2022 年提出的 CoT 增强版：对同一个问题用 CoT 采样多次（temperature > 0），得到多条推理路径和答案，再通过多数投票（majority voting）选出最终答案。它解决的问题是：单次 CoT 推理可能走偏，多次采样再聚合能显著提升准确率，尤其对有唯一正确答案的推理任务（数学、逻辑）效果很好。代价是推理成本乘以采样次数，所以通常只在高价值、高准确率要求的场景下使用。

**深入追问：** CoT 为什么有效存在哪些争议？两种解释路径在数学上有什么区别？

**回答要点：**

学术界对 CoT 的有效性机制有两种解释路径的争议，可以用概率分解来区分：

**路径 1：因果推理（Causal Reasoning）**

标准的链式分解，推理步骤 z 是答案 y 的因：

$$P(y, z | x) = P(y | z, x) \times P(z | x)$$

模型先生成推理步骤 z，再基于 z 生成答案 y。关键假设：**z 真的影响 y 的生成**，中间步骤提供了"工作记忆"，让模型能逐步积累信息、修正错误。

**路径 2：选择偏差（Selection Bias / Post-hoc Rationalization）**

模型先通过模式匹配"确定"答案 y，然后**根据答案倒推生成推理步骤 z**：

$$P(y, z | x) = P(z | y, x) \times P(y | x)$$

生成顺序是：模型先隐式地采样答案 $y \sim P(y | x)$，再生成"看起来能推导出 y"的推理步骤 $z \sim P(z | y, x)$。推理步骤是**事后合理化（post-hoc rationalization）**，不是真正的推理过程。

这解释了为什么有时推理步骤有逻辑错误，但答案仍然正确——因为答案不是从推理步骤推导出来的，而是推理步骤是为了"解释"答案而生成的。

CoT 的作用不是帮助推理，而是改变**采样分布**，让高质量答案更容易被采样到：

$$P_{\text{CoT}}(y | x) \neq P_{\text{direct}}(y | x)$$

**如何区分两种机制：**

1. **干预实验**：人为修改或删除中间步骤，观察答案质量变化
   - 因果路径：答案质量显著下降（因为 y 依赖 z）
   - 选择偏差：答案质量影响较小（因为 y 不真正依赖 z）

2. **推理步骤质量分析**：检查推理步骤的逻辑正确性
   - 因果路径：推理步骤错误 → 答案错误（强相关）
   - 选择偏差：推理步骤错误但答案正确（弱相关）

**当前研究共识**：两种机制都存在，具体哪种占主导取决于任务复杂度和模型能力。简单任务（模式匹配足够）更可能是选择偏差，复杂推理任务（需要多步依赖）更可能是真正的因果推理。

**深入追问：** 什么任务加了 CoT 收益反而不大？

**回答要点：**

简单的事实查询、格式转换、分类任务通常不需要 CoT——这类任务本来就不需要多步推理，强行加 CoT 只会增加 token 消耗和时延，有时还会引入不必要的"过度思考"。CoT 的价值在于任务本身有推理链路，如果任务是"直接查答案"而不是"推导答案"，CoT 的收益就很有限。另外，对于非常简单的任务，CoT 有时反而会让模型绕弯路，降低准确率。

## 5. ReAct 模式

**面试官可能会问：** 什么是 ReAct？它为什么成为 Agent 里的经典模式？和单次直接回答有什么本质区别？

**回答要点：**

**ReAct（Reasoning + Acting）** 是 Yao et al. 2022 年提出的 Agent 推理模式，核心思想是把推理（Thought）和行动（Action）交替进行，每次行动后观察结果（Observation），再基于观察继续推理和行动，形成 Thought → Action → Observation 的循环：

```
Thought_t: 当前状态分析，下一步应该做什么
Action_t: 调用工具或执行操作
Observation_t: 工具返回结果
→ Thought_{t+1}: 基于 Observation 继续推理
```

和单次直接回答的本质区别在于：单次回答是"一次生成结束"，ReAct 是"边做边想的闭环"。ReAct 让模型能在执行过程中根据实际反馈调整策略，而不是在没有任何外部信息的情况下一次性生成答案。Thought 步骤是关键——它让模型在行动前先显式推理"我现在知道什么、还缺什么、下一步应该做什么"，这个显式推理过程既提升了决策质量，也让整条轨迹可解释、可调试。

ReAct 特别适合的场景：信息不完整需要边查边做、任务路径不确定需要动态调整、需要多工具协作的任务。不适合的场景：任务简单可以直接回答、工具调用成本高但收益低、对时延要求极高的场景——这些情况下 ReAct 的额外推理步骤只是成本，不是收益。

**深入追问：** ReAct 在实际落地时最容易出什么问题？

**回答要点：**

最常见的问题有三类：第一，**Thought 质量不稳定**——模型的推理步骤看起来合理但实际走偏，导致后续 Action 选错；第二，**循环不收敛**——没有明确的 stop condition，模型会一直循环下去；第三，**Observation 处理不好**——工具返回的结果没有被正确理解或整合，导致下一步决策失真。解决方案分别是：优化 Thought 的 Prompt 约束、显式设置 stop rule 和最大步数、对工具返回结果做结构化处理再喂给模型。

## 6. Plan-and-Execute 与 Reflection

**面试官可能会问：** Plan-and-Execute 和 ReAct 的区别是什么？什么是 Reflection？什么时候值得加？

**回答要点：**

**Plan-and-Execute** 的核心思想是先用 Planner 把任务分解成子目标列表，再逐步执行每个子目标。和 ReAct 的区别在于：ReAct 是**边想边做**，规划和执行交织在一起，适合信息不完整、路径不确定的在线探索；Plan-and-Execute 是**先想后做**，适合任务结构相对清晰、可以提前拆解子目标的长链路任务。

Plan-and-Execute 的优势是全局视角更清晰，不容易在局部细节里迷失；劣势是规划一旦出错，整条链路都会走偏，而且规划阶段无法利用执行过程中的实时反馈。所以它更适合任务结构稳定的场景，不适合高度动态、需要频繁根据反馈调整的场景。

**Reflection（反思机制）** 是在任务执行完成后（或某个阶段后），让模型对自己的输出进行自我评估和修正。典型实现如 **Reflexion**（Shinn et al. 2023）：把执行结果和失败原因存入记忆，下次执行时参考这些反思来改进策略。**Self-Refine** 则是在单次任务内迭代优化：生成初稿 → 自我评估 → 修改 → 再评估，直到满足质量标准。

什么时候值得加 Reflection：任务结果可以被客观验证（有明确的成功/失败标准）、允许多轮迭代、错误模式有规律可循。什么时候不值得加：任务本来就简单、验证成本高于修正收益、对时延敏感的场景——这些情况下 Reflection 只是额外的 token 消耗。

**深入追问：** 这三种模式（ReAct、Plan-and-Execute、Reflection）在实际项目里怎么组合？

**回答要点：**

实际落地很少是纯粹的单一模式。常见的组合是：用 Plan-and-Execute 做顶层任务分解，每个子任务内部用 ReAct 执行，执行完成后用 Reflection 做质量检查。但这种组合的复杂度和成本也会叠加，所以不是默认配置，而是在主链路跑稳之后，针对具体瓶颈按需引入。

## 7. Prompt Injection 与防护

**面试官可能会问：** 什么是 Prompt Injection？为什么 Agent 场景下风险更高？应该从哪几层防？

**回答要点：**

**Prompt Injection** 是攻击者通过在用户输入、检索内容、工具返回结果等渠道注入恶意指令，试图覆盖或绕过系统的 System Prompt 约束，让模型执行未授权的行为。本质上是在攻击**模型的指令优先级机制**——利用模型无法严格区分"系统指令"和"数据内容"的弱点。

为什么 Agent 场景下风险更高？因为 Agent 会主动从外部获取内容（检索文档、网页、工具返回），这些内容都可能成为注入源。而且 Agent 有执行能力——一旦被注入成功，攻击者不只是能让模型说错话，还能让它执行危险操作（发邮件、删数据、调用高权限 API）。单轮问答被注入最多是输出错误内容，Agent 被注入可能造成真实的系统损害。

Prompt Injection 的防护需要多层配合，没有单一银弹：**指令隔离**——在 Prompt 结构上明确区分"系统指令区"和"数据内容区"，用结构化标记（如 XML 标签）把用户输入和检索内容包裹起来，减少模型把数据当指令执行的概率；**输入清洗**——对用户输入和外部内容做预处理，过滤或转义可能的注入模式；**工具权限控制**——高风险工具（写操作、外部调用）加显式确认或权限校验，不让模型直接执行；**输出约束与二次校验**——对模型输出做格式校验和语义校验，检测是否有异常行为；**最小权限原则**——Agent 只暴露完成任务必需的工具，减少被注入后的攻击面。

"告诉模型不要被注入"通常不够，因为这本质上还是在依赖模型的指令遵循能力，而注入攻击恰恰是在绕过这个能力。防护必须在系统层面做，而不是只靠 Prompt 里的一句话。

**深入追问：** 间接注入（Indirect Prompt Injection）为什么特别难防？

**回答要点：**

间接注入是通过污染模型会读取的外部内容（知识库文档、网页、工具返回）来攻击，而不是直接在用户输入里注入。因为检索内容通常被系统信任——它是"系统主动拉取的"，所以很多系统对检索内容的过滤比对用户输入更宽松。但如果知识库里有被污染的文档，或者检索的是外部网页，这些内容完全可能包含注入指令。间接注入是目前 Agent 安全里最难防的攻击面之一，防护重点在于对所有外部内容一视同仁地做清洗和隔离，不能因为是"系统拉取的"就降低警惕。

## 8. 项目深挖

**面试官可能会问：** 如果让你介绍一个做过的 Prompt / Planning 项目，链路通常应该怎么设计？遇到过哪些 bad case？如果重做，最值得优先改哪一层？

**回答要点：**

**链路设计**的核心是把 Prompt 的职责分清楚，不要把所有东西都塞进一个大 Prompt。一个典型的 Agent Prompt 链路通常包含：意图识别层（判断用户想做什么）→ 规划层（决定用哪些工具、走什么路径）→ 执行层（ReAct 循环，每步 Thought + Action + Observation）→ 生成层（基于执行结果组织最终回答）。每一层的 Prompt 职责不同，应该分开设计和评测，而不是用一个 Prompt 包打天下。

**典型 bad case** 通常有几类：第一，**规划漂移**——模型在 Thought 步骤里推理看起来合理，但实际上偏离了原始任务目标，越走越远；第二，**工具误调**——工具描述不够清晰，模型选错了工具或者用错了参数；第三，**上下文污染**——检索结果或工具返回的内容里有噪声，模型把噪声当成了有效信息；第四，**格式不稳定**——输出格式在某些边界情况下崩掉，导致下游解析失败。

定位 bad case 的方法：先看完整轨迹（Thought + Action + Observation 的完整链路），找到第一个出错的步骤，再判断是 Prompt 约束不够、工具 schema 有问题、还是检索质量差。不要一上来就改 Prompt，很多时候问题根源在上游。

**如果重做，最值得优先改哪一层**：通常是工具 schema 和 Prompt 的结构化程度，而不是先换模型。工具描述写得不清晰、参数定义模糊，是工具误调最常见的根因；Prompt 结构混乱、职责不清，是输出不稳定最常见的根因。这两层改好之后，很多"模型不够聪明"的问题会自然消失。

**深入追问：** 线上最难处理的 Prompt / Planning bad case 通常是什么？

**回答要点：**

最难的是"每一步 Thought 都看起来合理，但最终结果是错的，而且不稳定复现"。这类 case 的特点是：单看任何一步都找不到明显错误，但整条链路的方向就是偏了。根因通常是任务目标在 Prompt 里定义得不够精确，导致模型在多步推理中逐渐漂移。解决方法是在 Prompt 里加更明确的目标锚点和中间验证点，让模型在关键节点显式确认"我现在的方向还对吗"。
