# 编码题与手撕题

## 1. 为什么 AI 开发岗还会考代码

**面试官可能会问：** 你平时写代码多吗？AI 岗为什么还要考算法？

**回答要点：**

AI 岗考代码，考的不只是算法能力，更是工程判断力。面试官通过代码题在看三件事：你能不能把一个模糊问题转化成清晰的实现（这和写 prompt、设计 Agent 链路是同一种能力）；你写代码时有没有工程意识（边界条件、复杂度、并发安全，这些在 AI 系统里同样重要）；你遇到不会的问题时怎么应对（是卡死，还是能拆解、能沟通、能推进）。

AI 岗的代码题通常不是纯算法竞赛题，更多是"工程感"题：LRU 缓存、并发调度器、字符串解析、任务队列。这些题直接对应 AI 系统里的真实场景——模型缓存、工具并发调用、prompt 模板解析、异步任务管理。只会调 API 的人，遇到性能问题、并发问题、数据一致性问题就会卡住，代码题是在验证你有没有这层工程底子。

**深入追问：** AI 岗的代码题和纯算法岗有什么不同侧重？

**回答要点：**

AI 岗更看重工程实现能力，而不是竞赛算法技巧。面试官更关心你能不能把一个系统组件实现得正确、健壮、可扩展，而不是你能不能在 20 分钟内解出一道 Hard 动态规划。LRU、并发调度器、限流器这类题，考的是你对数据结构的实际运用和对系统设计的理解，这比背 DP 转移方程更贴近真实工作。

## 2. 高频基础题

**面试官可能会问：** 请实现一个 LRU 缓存。（或：反转链表、两数之和、滑动窗口最大值）

**回答要点：**

AI 岗高频基础题集中在几个方向，每类都有对应的考察意图。

LRU 缓存是最高频的题。核心实现是 HashMap + 双向链表，HashMap 保证 O(1) 查找，双向链表保证 O(1) 的节点移动和删除。为什么用双向链表而不是单向链表？因为删除一个节点时需要知道它的前驱节点，单向链表做不到 O(1) 删除任意节点。实现时要注意：get 操作命中时要把节点移到链表头部（表示最近使用）；put 操作时如果 key 已存在要先删旧节点，如果容量满了要删链表尾部节点（最久未使用）。AI 系统里模型推理缓存、KV cache、embedding 缓存都是 LRU 的真实应用场景。

字符串处理类题包括模板解析、括号匹配、表达式求值、字符串解码。对应 prompt 模板处理、工具调用参数解析等场景。这类题考的是对栈、递归、状态机的掌握。括号匹配用栈，遇到左括号入栈，遇到右括号出栈并匹配，最后栈为空则合法。

链表、哈希、数组是基本盘，不会就是硬伤。反转链表、合并有序链表、两数之和、滑动窗口——这些题不是在考你刷题量，而是在验证你有没有基本的数据结构直觉。

树与回溯类题包括二叉树遍历、路径搜索、全排列、括号生成。回溯题要能说清楚剪枝条件，这对应 Agent 的决策树和搜索空间剪枝等概念。

**深入追问：** LRU 的 get 和 put 操作为什么都是 O(1)？

**回答要点：**

get 是 O(1) 因为 HashMap 的查找是 O(1)，找到节点后把它移到链表头部也是 O(1)（双向链表的节点移动只需要修改前后指针）。put 是 O(1) 因为插入到链表头部是 O(1)，删除链表尾部节点也是 O(1)（维护一个 tail 指针）。整个实现的关键是 HashMap 存的是节点的引用，而不是节点的位置，所以可以直接操作节点而不需要遍历链表。

## 3. 并发与调度题

**面试官可能会问：** 请实现一个并发限制为 N 的请求调度器。（或：实现一个令牌桶限流器、实现一个任务队列）

**回答要点：**

并发调度题是 AI 岗的特色题型，直接对应工具调用并发控制、模型推理请求限流、异步任务队列等真实场景。

并发限制调度器的核心设计是：用信号量（Semaphore）或计数器控制并发数，用队列管理等待任务，任务完成时释放一个槽位并触发下一个任务。答题前要先问清楚：任务是同步还是异步的，有没有优先级，失败了要不要重试，超时怎么处理。这些问题本身就是加分项，说明你有工程意识，不是只会写 happy path。

令牌桶限流器的核心是：维护一个令牌计数器，按固定速率补充令牌，每次请求消耗一个令牌，令牌不足时拒绝或等待。和漏桶的区别是令牌桶允许突发流量（桶里有积累的令牌可以一次性消耗），漏桶是严格匀速输出。AI 系统里对外部 API（如 OpenAI）的调用限流通常用令牌桶。

任务队列的关键设计点是：任务的状态管理（pending/running/done/failed）、任务完成后的回调机制、失败重试策略（指数退避）、超时处理（任务运行超过阈值时强制终止）。

**深入追问：** 并发调度器在多线程环境下有哪些需要注意的问题？

**回答要点：**

共享状态的线程安全是第一个问题：计数器、队列这些共享状态在多线程下需要加锁或用原子操作保护，否则会出现竞态条件。任务完成顺序和提交顺序的关系是第二个问题：并发执行时任务完成顺序不确定，如果业务要求按顺序处理结果，需要额外的排序机制。异常传播是第三个问题：子任务的异常不能静默吞掉，要有明确的错误处理和上报机制。资源释放的时机是第四个问题：任务失败时也要释放槽位，否则调度器会逐渐卡死。

## 4. 手撕实现类题

**面试官可能会问：** 请手写全排列。（或：括号生成、编辑距离、最长公共子序列）

**回答要点：**

手撕实现类题考的是你能不能在没有 IDE 的情况下，把一个算法从思路变成可运行的代码。这类题的关键不是背答案，而是能在面试现场推导出来。

全排列是回溯的经典题。思路是：维护一个 used 数组标记哪些元素已经用过，递归地把每个未用过的元素加入当前路径，到达叶子节点（路径长度等于数组长度）时记录结果。剪枝条件是 used[i] 为 true 时跳过。时间复杂度 O(n * n!)，空间复杂度 O(n)。

括号生成也是回溯。维护左括号数和右括号数，剪枝条件是：左括号数不能超过 n，右括号数不能超过左括号数（否则括号不合法）。到达长度 2n 时记录结果。

编辑距离是经典动态规划。dp[i][j] 表示 word1 前 i 个字符变成 word2 前 j 个字符的最少操作数。转移方程：如果 word1[i] == word2[j]，dp[i][j] = dp[i-1][j-1]；否则 dp[i][j] = min(dp[i-1][j], dp[i][j-1], dp[i-1][j-1]) + 1，分别对应删除、插入、替换三种操作。

**深入追问：** 回溯和动态规划的选择依据是什么？

**回答要点：**

回溯适合枚举所有可能的解，问题有明确的剪枝条件，解的数量是指数级的（如全排列、括号生成）。动态规划适合有重叠子问题和最优子结构的问题，目标是求最优解或计数（如编辑距离、最长公共子序列）。一个简单的判断方法是：如果问题要求"所有解"，通常用回溯；如果问题要求"最优解"或"方案数"，通常用动态规划。

## 5. 复杂度、边界条件与工程追问

**面试官可能会问：** 你的解法复杂度是多少？如果数据规模再大十倍怎么办？如果放到生产环境你还会补什么？

**回答要点：**

复杂度分析要养成习惯，不要等面试官问才说。写完代码后主动说：时间复杂度是多少，空间复杂度是多少，最坏情况是什么。如果有多个操作，分别说每个操作的复杂度。

"数据规模再大十倍"类追问的回答框架是：先说当前方案的瓶颈在哪里（内存、CPU、IO），再说规模扩大后哪个瓶颈会先触发，最后说对应的优化方向（分片、流式处理、外部存储、并行化）。不要说"换更好的机器"，这不是工程回答。

"如果放到生产环境你还会补什么"类追问，要说出具体的点：输入校验和异常处理（空输入、超大输入、非法格式）；并发安全（如果有共享状态，用什么机制保护）；超时和重试机制（外部依赖不可用时怎么处理）；监控和日志（怎么知道系统是否正常运行）。这类追问不是在考你能不能立刻写出生产级代码，而是在看你有没有工程意识——知道 demo 和生产之间的差距在哪里。

**深入追问：** 为什么代码能跑不等于代码合格？

**回答要点：**

能跑只说明 happy path 没问题。生产代码还要处理：空输入、超大输入、并发竞争、网络超时、依赖服务不可用。面试里能主动说出这些，说明你有真实的工程经验，而不是只在本地跑过 demo。

## 6. 代码表达与翻车点

**面试官可能会问：** （给出题目后，等你开始）

**回答要点：**

写代码前的表达是面试里最容易被忽视、也最容易拉开差距的环节。正确的流程是：先复述题意确认理解（"我理解这道题是要实现一个……，输入是……，输出是……，对吗"）；再说出思路方向（"我的思路是用哈希表 + 双向链表，时间复杂度 O(1)"）；主动说边界条件（"我需要处理空输入、容量为 0、重复 key 这几种情况"）；边写边说关键逻辑；写完后主动分析复杂度。

翻车是正常的，关键是怎么应对。完全不会这道题时，不要沉默，直接说"我没有直接做过这类题，我先想一下思路"，然后大声拆解：暴力解法是什么，复杂度是多少，有没有优化空间。能说出暴力解法并分析复杂度，比沉默五分钟强得多。写到一半卡住时，停下来说"我在这里卡住了，我想一下"，然后说出你卡在哪里——是边界条件没想清楚，还是数据结构选错了。主动暴露卡点，面试官可能会给提示。写完发现有 bug 时，不要等面试官指出来，自己先说"我来 walk through 一遍，看看有没有边界问题"，然后用一个具体的测试用例跑一遍代码。代码写崩了逻辑乱了时，直接说"我觉得这个方向有问题，我重新想一下"，从头说思路重新写，不要在错误的方向上继续修补。

**深入追问：** 为什么很多人不是不会做，而是不会表达？

**回答要点：**

因为面试不是 OJ 判题，面试官在看的是你的思维过程，不只是最终代码。一个思路清晰但代码有小 bug 的人，比一个闷头写完但说不清楚为什么这么写的人，通常评价更高。代码题里的表达能力，和系统设计题里的表达能力是同一种能力——把你的思考过程外显出来，让面试官能跟上你的逻辑。
