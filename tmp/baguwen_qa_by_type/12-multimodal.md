# 多模态

## 1. 多模态的定义、价值与边界

**面试官可能会问：** 什么是多模态？多模态模型和纯文本模型的本质区别是什么？

**回答要点：**

多模态指系统能同时处理两种或两种以上模态的信息，最常见的组合是图文（Vision-Language），其次是图文音频、图文视频。和纯文本模型的本质区别不只是"多了一个输入通道"，而是表示空间不同。纯文本模型的输入是 token 序列，语义空间是离散符号空间；多模态模型需要把图像、音频等连续信号映射到和文本兼容的语义空间，才能让模型统一推理。这个映射过程通常由视觉编码器（如 ViT）完成，再通过投影层（如 MLP 或 Q-Former）对齐到语言模型的 embedding 空间。

多模态的核心价值在于三点：让系统能理解人类真实交互中大量非文本信息（截图、图表、产品图）；让检索和问答不再局限于文本；让 Agent 能感知视觉环境，执行更复杂的任务。

**深入追问：** 多模态的边界在哪里？什么问题不适合用多模态？

**回答要点：**

多模态不是万能的。信息本身就是纯文本时，强行加图只会增加成本；图像质量差、渲染不稳定时，视觉信号的噪声大于收益；任务对精确数字、表格结构要求极高时，视觉模型的识别误差会直接影响结果，这时 OCR + 结构化解析往往比端到端视觉模型更可靠。判断要不要上多模态，核心问题是：视觉信息是不是任务的关键输入，而不是可选的补充。

## 2. 多模态输入接入与表示

**面试官可能会问：** 图片、截图、图表这类输入通常怎么接进系统？OCR、视觉编码、图像 Embedding 各自在做什么？

**回答要点：**

多模态输入处理通常有三条路径，选哪条取决于任务类型和精度要求。

第一条是 OCR 转文本，也叫伪多模态路径。用 OCR 引擎（如 PaddleOCR、Azure Document Intelligence）把图片中的文字提取出来，后续走纯文本 RAG 或问答链路。优点是成本低、可控性强、文本检索效果好；缺点是丢失了布局、颜色、图形等视觉信息，对图表、流程图、手写内容效果差。

第二条是视觉编码，也就是端到端多模态路径。用视觉编码器（ViT、CLIP 的图像侧）把图片编码成向量，再通过投影层对齐到语言模型空间，让 LLM 直接"看图说话"。GPT-4V、Claude 3、LLaVA 都走这条路。优点是能理解图形语义、布局关系；缺点是成本高、时延大，对小字和精确数字识别不如 OCR。

第三条是图像 Embedding，主要用于检索场景。用 CLIP 类模型把图片编码成向量，存入向量数据库，支持"以文搜图"或"以图搜图"。这条路解决的是检索问题，不是生成问题。

实际工程里三条路经常混用。比如截图先 OCR 提文字用于检索，再把原图送给视觉模型用于理解，两路结果合并后送给 LLM 生成答案。

**深入追问：** 为什么图片输入不只是"把图丢给模型"这么简单？

**回答要点：**

图片在进模型前通常要做预处理：分辨率标准化（不同模型对输入尺寸有要求）、格式转换（PNG/JPEG/WebP 的处理方式不同）、对于长截图或多页文档还要做切图和分块。元数据（来源、时间戳、页码）要和图片内容一起保留，否则后续引用和溯源会出问题。图片 token 消耗远大于文字，GPT-4V 处理一张高分辨率图片可能消耗 1000+ token，所以分辨率和 detail 级别的选择直接影响成本。

## 3. 多模态 RAG 与图文联合检索

**面试官可能会问：** 什么是多模态 RAG？伪多模态和原生多模态的区别是什么？CLIP 适合什么场景？

**回答要点：**

多模态 RAG 是在标准 RAG 流程里，把检索和生成的对象从纯文本扩展到图文混合内容。核心问题是：图片怎么被检索到，检索到之后怎么被利用。

伪多模态 RAG 是目前主流的落地方案。把图片先转成文本（OCR 提字、视觉模型生成 caption、图表转描述），再走标准文本 RAG 流程。优点是工程简单、检索效果可控、成本低；缺点是视觉信息有损失，图形语义、布局关系、颜色信息会丢失。

原生多模态 RAG 则是用 CLIP 类模型（如 OpenAI CLIP、BLIP-2、SigLIP）把图片和文本都编码到同一个向量空间，支持跨模态检索——用文字查图、用图查文。CLIP 的核心设计是图文对齐：用大量图文对做对比学习，让"一只猫的图片"和"一只猫"这段文字在向量空间里距离很近。检索到图片后，直接把原图送给多模态 LLM 生成答案，保留了完整视觉信息。缺点是 CLIP 的向量空间对细粒度文字内容（如图表里的具体数字）表达能力弱，且整体链路成本更高。

图文联合检索是同时对文本块和图片块建索引，检索时两路并行，再用 Rerank 或规则融合。适合文档里图文混排、图片和文字互相补充的场景，比如技术手册、产品说明书。

**深入追问：** 什么时候用 CLIP，什么时候用 OCR + 文本检索？

**回答要点：**

看任务类型。如果用户的查询是"找一张包含红色按钮的截图"或"找和这张图风格相似的产品图"，CLIP 更合适，因为它能做语义级图文匹配。如果用户的查询是"找包含'退款政策'这几个字的文档页"，OCR + 文本检索更准，因为 CLIP 对精确关键词不敏感。生产系统里通常两路都建，按 Query 类型路由。

## 4. 截图、图表与界面理解

**面试官可能会问：** 截图理解为什么不是单纯的视觉识别问题？图表内容为什么比普通图片更难处理？

**回答要点：**

截图理解是实际业务里最高频的多模态场景之一，难点不在"看到了什么"，而在"理解了什么"。可以分三层来看。

第一层是文字识别层，OCR 提取文字，这层相对成熟，但对小字、特殊字体、背景复杂的截图仍有误识别。第二层是结构理解层，需要理解界面布局——哪里是导航栏、哪里是内容区、哪里是按钮、哪里是表单。这需要结合视觉位置信息和语义信息，纯 OCR 做不到。第三层是功能语义层，需要理解"这个按钮点了会发生什么"、"这个错误提示意味着什么"，这层需要结合领域知识，通常要配合 RAG 或 Fine-tuning。

图表（柱状图、折线图、饼图）的难点在于视觉模型需要同时理解坐标轴含义、数值精度、图例对应关系、趋势方向。端到端视觉模型在精确数值提取上容易出错，更可靠的方案是先用专门的图表解析工具（如 ChartQA 专用模型，或把图表转成结构化数据）再送给 LLM 分析。

Playwright 这类工具负责自动化渲染网页并截图，是截图链路的数据采集层。渲染失败（JS 未加载、动态内容缺失）是截图理解最常见的上游故障，必须在这层做好重试和质量检测。

**深入追问：** 前端界面截图为什么还需要保留布局和结构信息？

**回答要点：**

因为很多界面理解任务不只是"读文字"，而是"理解空间关系"。比如"这个错误提示在哪个输入框旁边"、"这个按钮在页面的什么位置"，这些信息在纯文本里是丢失的。保留布局信息的方案包括：保留 HTML/DOM 结构；用带坐标的 OCR（如 Azure Form Recognizer）；用支持空间理解的视觉模型（如 GPT-4V 的 high-detail 模式）。三种方案各有成本，通常按任务精度要求选择。

## 5. 多模态系统的评测

**面试官可能会问：** 多模态系统怎么评测？为什么比纯文本系统更难评？

**回答要点：**

多模态系统的评测比纯文本更难自动化，原因在于视觉理解的"对不对"很难用字符串匹配来判断。一张图表的解读是否准确、一个界面的布局理解是否正确，都需要更复杂的评判标准。

评测通常分几层来做。第一层是组件级评测，分别评 OCR 准确率、图像 embedding 的检索召回率、视觉模型的 caption 质量，这层可以用标注数据做自动化评测。第二层是端到端任务评测，评整条链路的任务完成率，比如"给定一张截图，能否正确回答关于界面内容的问题"，这层通常需要 LLM-as-Judge 或人工抽检。第三层是对比评测，把多模态方案和纯文本方案（OCR + 文本 RAG）做 A/B 对比，证明多模态带来了真实收益。

常见的评测数据集包括 DocVQA（文档视觉问答）、ChartQA（图表理解）、TextVQA（图片中的文字理解），可以用来做基准测试。

**深入追问：** 如果多模态链路线上效果不稳定，通常先排查哪一层？

**回答要点：**

按链路分层排查：先看采集层（截图质量、OCR 准确率）；再看表示层（图像 embedding 是否准确、caption 质量）；再看检索层（图文联合检索的召回率）；最后才看生成层（视觉模型的理解能力）。很多多模态问题根因在采集层，不要一上来就归因给视觉模型不够强。

## 6. 多模态工程落地

**面试官可能会问：** 多模态系统在工程上最常见的难点有哪些？时延、成本、存储分别怎么控？

**回答要点：**

多模态系统的工程难点通常比纯文本系统多一个数量级，主要集中在以下几层。

渲染和采集层的稳定性是第一道关。截图失败、OCR 失败、图片格式不支持是最常见的上游故障，必须做重试机制、质量检测（空白截图、截图不完整的检测）和降级策略（截图失败时回退到文本链路）。

时延压力是第二个核心问题。图片预处理（OCR、视觉编码）加上多模态模型推理的时延通常是纯文本链路的 3-10 倍。常见优化手段包括：异步预处理（上传时就做 OCR 和 embedding，不等到查询时再做）、图片压缩（降低分辨率减少 token 消耗）、缓存（相同图片的 embedding 和 OCR 结果缓存复用）。

成本控制方面，图片 token 消耗远大于文字，成本控制策略包括：按需选择 detail 级别（low/high）、对不需要视觉理解的图片只做 OCR、对重复图片做去重和缓存。

存储和索引方面，图片原文件、OCR 文本、图像 embedding、视觉模型生成的 caption 需要分层存储并保持关联关系，向量数据库需要同时支持文本向量和图像向量的混合检索。

**深入追问：** 多模态 Agent 相比纯文本 Agent 的额外难点是什么？

**回答要点：**

在 Web Agent、GUI Agent 这类场景里，Agent 执行动作后需要通过截图来感知当前界面状态，截图是 Observation 的载体。额外难点主要有三个：每一步都要截图加视觉模型推理，链路时延显著高于纯文本 Agent；视觉模型的识别错误会被 Agent 当作事实，在多步链路里被放大；图片 token 消耗远大于文字，长链路里上下文窗口压力更大，需要做图片压缩或选择性保留。视觉理解的输出最好结构化（比如"按钮坐标"、"错误类型"），而不是直接把自然语言输出喂给工具层，否则工具调用会很不稳定。

## 7. 多模态项目深挖

**面试官可能会问：** 如果让你介绍一个做过的多模态项目，通常应该怎么讲？什么情况下值得做多模态，而不是只做文本？

**回答要点：**

什么时候值得做多模态，判断标准有三个：业务信息本身就是图文混排，纯文本丢失了关键信息（如技术文档里的架构图、产品手册里的操作截图）；用户的查询天然包含视觉意图（"找一张和这个类似的图"、"帮我看看这个截图里的错误"）；纯文本方案已经跑通，但效果卡在"文字描述不够准确"这个瓶颈上。

项目讲法的最稳链路是：业务背景（为什么需要多模态）→ 数据来源和预处理（OCR/视觉编码选择原因）→ 检索方案（伪多模态 vs 原生多模态，CLIP 还是文本检索）→ 生成方案（视觉模型选型）→ 工程难点（渲染稳定性、时延、成本）→ 评测方案（怎么证明多模态比纯文本好）→ 上线效果和 Bad Case。

面试官最想听的通常是：你为什么需要多模态而不是只做 OCR + 文本 RAG；图片的表示和检索方案为什么这么选；视觉模型怎么选，精度和成本怎么平衡；工程上遇到了哪些坑（渲染失败、OCR 误识别、图片 token 超限）；怎么证明多模态方案比纯文本方案效果更好。

**深入追问：** 如果重做一次多模态链路，最值得优先改哪一层？

**回答要点：**

通常优先看数据预处理层和检索层。很多多模态系统的问题不是视觉模型不够强，而是图片的表示质量太差——OCR 错误多、caption 不准确、embedding 不匹配。先把知识表示做好，再谈模型升级，这个顺序通常更有效。如果重做，我会先把 OCR 质量和图像 embedding 的评测做扎实，再决定要不要换更强的视觉模型。
