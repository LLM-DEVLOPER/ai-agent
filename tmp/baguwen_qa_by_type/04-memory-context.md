# 记忆与上下文工程面试知识点

这份文档以真实面试追问链路为主线整理，目标是让你从"上下文工程是什么"一路讲到"怎么设计一个真正可用的记忆系统"，每个环节都有具体判断依据和工程经验可以展开。

---

## 1. 核心概念：上下文工程 vs 记忆系统

**面试官可能会问：** 什么是上下文工程？什么是记忆系统？两者有什么区别？

**回答要点：**

- **上下文工程（Context Engineering）** 解决的是"当前这一轮到底把哪些信息送进模型"的问题——包括信息的选择、排序、裁剪、压缩和结构化拼装，是请求级别的输入设计。
- **记忆系统（Memory System）** 解决的是"系统长期保留什么、什么时候取出来用"的问题——是跨轮、跨会话的信息资产管理，有写入、召回、更新、淘汰的完整生命周期。
- **两者的关系**：记忆系统决定"存什么"，上下文工程决定"当前轮用什么"。记住了但喂不进去，或者喂进去了但组织不好，效果都会出问题。很多看起来是"模型记不住"的故障，排查下来其实是上下文构造问题——不是没存住，而是没在当前轮把对的信息正确放进上下文。

**深入追问：** 为什么很多 AI 应用的问题，本质上不是模型能力问题，而是上下文构造问题？

**回答要点：**

三个常见表现：

1. **关键信息没进上下文**：模型再强也没法凭空知道。用户说"我之前告诉过你"，但那段信息在当前轮的上下文里根本不存在。
2. **无关信息太多**：会稀释重点，甚至把正确证据埋掉。检索召回了 20 个文档，但真正有用的那段被埋在第 15 个，模型注意力被前面的噪声分散了。
3. **信息顺序不对**：关键约束被放在了上下文中间，容易被忽略（Lost in the Middle）。

最常见的表现是："历史里明明说过""知识库里明明有"，但模型当前轮就是没用上。这不是模型问题，是上下文组织问题。所以很多优化的第一步不是换更强的模型，而是把上下文整理好。

**深入追问：** 上下文工程和 Prompt Engineering 是一回事吗？

**回答要点：**

不完全是。Prompt Engineering 更侧重"怎么写指令让模型理解任务"，是指令设计；上下文工程的范围更广，包括指令，但更关注"把什么信息、以什么顺序、用什么结构送进模型"，是整个输入的架构设计。

一个好的 Prompt 可能因为上下文组织混乱而失效；一个好的上下文架构能让普通 Prompt 也有不错效果。

## 2. 上下文构造：Token 预算与组织策略

**面试官可能会问：** 上下文窗口有限，怎么分配 Token 预算？什么信息应该优先放进去？

**回答要点：**

Token 预算分配的核心原则是**按重要性分层，硬约束优先**：

```
系统约束（System Prompt）> 用户当前输入 > 相关历史 > 检索结果 > 工具输出
```

1. **系统约束**：角色定义、行为规范、输出格式要求——这是系统行为的基础，不能被裁剪
2. **用户当前输入**：当前轮的问题或指令，必须完整保留
3. **相关历史**：多轮对话的上下文，按相关性和时间衰减选择性保留
4. **检索结果**：RAG 召回的文档，可以做摘要压缩
5. **工具输出**：工具调用的返回结果，冗余部分可以裁剪

预算不够时的裁剪顺序：先裁低相关历史和冗余工具结果，再压缩检索文档，**绝不裁系统约束和当前输入**。

**深入追问：** 为什么 System Prompt 要保持稳定不变？

**回答要点：**

System Prompt 稳定不变有三个关键原因：

1. **行为一致性**：System Prompt 定义了模型的角色和约束，频繁变化会导致行为不稳定，用户体验割裂
2. **Prefix Cache 命中**：现代推理服务（vLLM、Anthropic Prompt Caching）会缓存相同前缀的 KV Cache，System Prompt 稳定意味着每次请求都能命中缓存，跳过 Prefill 阶段，首 token 延迟降低 50%+，成本也大幅下降
3. **调试可控**：System Prompt 是系统行为的"宪法"，稳定后出问题容易定位，频繁改动会让 bad case 难以复现

工程实践：把动态内容（用户信息、检索结果）放在 User Message 里，不要塞进 System Prompt。

**深入追问：** 上下文信息的组织顺序有什么讲究？

**回答要点：**

顺序直接影响模型的注意力分配，核心原则是**关键信息放首尾，避免埋在中间**：

1. **首位效应（Primacy）**：开头的信息模型注意力最集中，适合放核心约束和任务定义
2. **近因效应（Recency）**：结尾的信息模型最容易"记住"，适合放当前轮最相关的证据
3. **Lost in the Middle**：中间部分容易被忽略，不要把关键证据或约束埋在长上下文的中间

实践中的组织顺序：System Prompt（约束） → 检索文档（证据） → 对话历史 → 用户当前输入（问题）。这样关键约束在开头，当前问题在结尾，模型注意力分配最优。

## 3. 多轮对话：历史管理与约束持久化

**面试官可能会问：** 多轮对话场景下，历史消息越来越长，怎么管理？

**回答要点：**

多轮对话的核心挑战是**历史膨胀**和**约束丢失**。常见的历史管理策略：

1. **滑动窗口（Sliding Window）**：只保留最近 N 轮对话，简单但会丢失早期重要信息
2. **摘要压缩（Summarization）**：定期把旧对话压缩成摘要，保留关键信息但丢失细节
3. **相关性过滤（Relevance Filtering）**：根据当前问题，只保留相关的历史轮次，需要额外的相关性判断
4. **分层存储**：近期对话全量保留，远期对话压缩或只保留关键轮次

生产中通常组合使用：最近 5 轮全量保留 + 更早的轮次做摘要 + 超过 20 轮的按相关性过滤。

**深入追问：** 多轮对话中最容易出现的"约束丢失"是什么？怎么防？

**回答要点：**

约束丢失的典型表现：用户在第 1 轮说"用中文回答"，到第 10 轮模型又开始用英文回答了。根因是**约束被当成普通历史消息处理，随着对话推进被截断或稀释**。

防护策略：

1. **约束显式化**：把约束单独存储，不要混在消息历史里，每轮都显式注入到 System Prompt 或 User Message 开头
2. **约束持久化**：用户明确的约束（输出格式、语言、角色设定）要写入持久化的 State，不依赖"还没被截掉"来维持有效性
3. **约束验证**：关键约束在生成后做验证，违反约束时触发重试或修正

工程实践：维护一个 `constraints` 字段，独立于 `messages` 历史，每轮构造上下文时显式拼接。

**深入追问：** 什么时候应该主动"遗忘"历史？

**回答要点：**

并不是所有历史都值得保留，主动遗忘的场景：

1. **任务切换**：用户从"写代码"切换到"查天气"，旧任务的上下文是噪声，应该清空
2. **错误路径**：Agent 尝试了错误的工具调用路径，保留这些失败记录会干扰后续决策，应该裁剪
3. **隐私敏感**：用户提供了临时的敏感信息（密码、API Key），任务完成后应该主动清除

判断标准：这段历史对当前任务是帮助还是干扰？如果是干扰，主动遗忘比盲目保留更好。

## 4. 长上下文：能力边界与 Lost in the Middle

**面试官可能会问：** 现在模型都支持 128K 甚至 1M 上下文了，是不是可以把所有信息都塞进去？

**回答要点：**

长上下文能力≠可以无脑塞满。三个关键限制：

1. **注意力衰减**：上下文越长，模型对每个 token 的注意力越分散，关键信息容易被忽略
2. **Lost in the Middle**：模型对上下文首尾部分的利用率远高于中间部分，关键证据埋在中间等于没给
3. **成本和延迟**：上下文长度和推理成本、首 token 延迟成正比，128K 上下文的成本是 4K 的 32 倍

工程实践：**不要因为能塞就塞，而是精选最相关的信息**。RAG 召回 50 个文档不代表要全部放进上下文，Rerank 后取 top-3~5 效果通常更好。

**深入追问：** Lost in the Middle 具体是什么现象？对工程有什么影响？

**回答要点：**

Lost in the Middle 是指模型对长上下文中间部分的信息利用能力明显弱于首尾。实验表明：

- 关键证据在开头：模型准确率 ~85%
- 关键证据在结尾：模型准确率 ~80%
- 关键证据在中间：模型准确率 ~60%

**工程影响**：

1. **检索结果排序**：不能按原始相关性分数直接放入，要把最相关的文档放在首尾，次相关的放中间
2. **约束位置**：关键约束要放在 System Prompt 开头或 User Message 结尾，不要埋在长文档中间
3. **多轮对话**：重要的历史轮次要前置或后置，不要让关键信息沉到中间

实践中的优化：检索到 10 个文档，按相关性排序后，放入顺序是 [1, 3, 5, 7, 9, 10, 8, 6, 4, 2]——最相关的在首尾，次相关的在中间。

**深入追问：** 长上下文场景下，怎么判断模型是真的"理解"了还是只是"看到"了？

**回答要点：**

"看到"≠"理解"≠"使用"。验证方法：

1. **针刺测试（Needle in a Haystack）**：在长文档中间插入一个关键事实，看模型能否准确提取
2. **多跳推理测试**：答案需要综合上下文中多处信息，测试模型的全局理解能力
3. **干扰测试**：加入相似但错误的信息，看模型能否区分

生产中的实践：不要假设模型能完美利用所有上下文，关键信息要显式标注（"以下是最重要的证据"）或结构化组织（用 XML 标签分隔不同来源）。

## 5. 记忆系统设计：分层架构

**面试官可能会问：** 一个完整的记忆系统应该包含哪些层次？

**回答要点：**

生产级记忆系统通常是**三层架构**：

```
短期记忆（Short-term Memory）
  ├─ 当前会话的对话历史
  ├─ 当前任务的中间状态
  └─ 生命周期：会话结束即清空

长期记忆（Long-term Memory）
  ├─ 用户偏好和画像
  ├─ 历史任务和决策
  └─ 生命周期：跨会话持久化

结构化状态（Structured State）
  ├─ 任务目标和约束
  ├─ 工具调用历史
  └─ 生命周期：任务完成即清空
```

**关键区别**：

- **短期记忆**：临时、易变、全量保留，用于维持当前会话的连贯性
- **长期记忆**：持久、稳定、选择性写入，用于跨会话的个性化
- **结构化状态**：显式、可验证、独立于对话历史，用于任务执行的可控性

**深入追问：** 长期记忆的写入门槛应该怎么设计？

**回答要点：**

长期记忆不是什么都写，**写入门槛是质量的保证**。判断标准：

1. **稳定性**：这个信息在未来多次会话中都有用吗？临时的任务状态不写
2. **可验证性**：这个信息是事实还是推测？只写确认的事实（"用户是 Python 开发者"），不写推测（"用户可能喜欢 React"）
3. **非冗余性**：这个信息能从其他地方推导出来吗？能从代码仓库看出的技术栈不需要单独记忆

**反例**：把每次工具调用结果都写入长期记忆 → 记忆库迅速膨胀，噪声增加，召回精度下降。

**正例**：只写用户明确表达的偏好（"我喜欢用 TypeScript"）、反复出现的模式（连续 5 次任务都涉及数据库优化 → 记录"用户关注数据库性能"）。

**深入追问：** 记忆系统和 RAG 的边界在哪里？

**回答要点：**

两者的核心区别是**数据属性**：

| 维度 | 记忆系统 | RAG 知识库 |
|------|---------|-----------|
| **数据来源** | 用户交互产生 | 外部文档导入 |
| **数据属性** | 用户私有 | 多用户共享 |
| **生命周期** | 随用户存在 | 独立于用户 |
| **更新频率** | 高频（每次对话） | 低频（文档更新） |
| **权限模型** | 用户级隔离 | 可能跨用户共享 |

**工程实践**：记忆和知识库必须物理隔离，不能混在同一个向量库里。原因：

1. **权限泄露风险**：用户 A 的记忆不能被用户 B 检索到
2. **召回策略不同**：记忆召回看时间衰减和相关性，知识库召回看语义相似度
3. **更新策略不同**：记忆可以覆盖更新，知识库通常是版本管理

## 6. 记忆生命周期：写入、召回、更新、淘汰

**面试官可能会问：** 记忆系统的完整生命周期是怎样的？每个环节的关键决策是什么？

**回答要点：**

记忆系统不是"写进去就完事"，而是有完整的生命周期管理：

**1. 写入（Write）**：什么时候写？写什么？
- 触发条件：用户明确表达偏好、任务完成后的总结、反复出现的模式
- 写入内容：事实性信息（"用户是 Python 开发者"）、偏好（"喜欢简洁的代码风格"）、决策历史（"上次选择了方案 A"）
- 门槛控制：只写稳定、可验证、非冗余的信息

**2. 召回（Recall）**：什么时候取？怎么取？
- 触发时机：每轮对话开始时、任务规划阶段、遇到歧义需要上下文时
- 召回策略：语义相似度 + 时间衰减 + 显式关联（用户 ID、任务类型）
- 召回数量：通常 top-3~5，过多会污染上下文

**3. 更新（Update）**：什么时候改？怎么改？
- 覆盖更新：用户偏好变化（"我现在更喜欢 TypeScript"）
- 增量更新：补充细节（"Python 开发者" → "Python 开发者，专注数据分析"）
- 版本管理：关键决策保留历史版本，可追溯

**4. 淘汰（Eviction）**：什么时候删？
- 过期淘汰：时间超过阈值且未被访问（如 90 天未使用）
- 冲突淘汰：新信息和旧信息矛盾时，保留新的
- 容量淘汰：记忆库达到上限，按访问频率和时间衰减淘汰

**深入追问：** 记忆召回命中率高，但效果反而变差了，可能是什么原因？

**回答要点：**

命中率高≠效果好，四类常见问题：

1. **误召回**：召回的记忆和当前任务不相关，语义相似但场景不同（"Python 开发"召回了"Python 爬虫"的记忆，但当前任务是数据分析）
2. **记忆过期**：召回的信息已经过时（"用户喜欢 Vue 2" 但用户已经切换到 Vue 3）
3. **上下文污染**：召回的记忆和其他信息形成噪声，干扰模型判断
4. **没被利用**：记忆进了上下文但模型没用上，可能是信息埋在中间（Lost in the Middle）或表达不够显式

排查方法：分层验证——记忆本身对不对？召回的记忆相关不相关？模型有没有真正使用？

**深入追问：** 记忆更新的冲突怎么处理？

**回答要点：**

冲突场景：旧记忆说"用户喜欢 A"，新信息说"用户喜欢 B"。处理策略：

1. **时间优先**：默认新信息覆盖旧信息，但保留更新时间戳
2. **置信度加权**：明确表达（"我更喜欢 B"）> 推测（"用户可能喜欢 B"），高置信度覆盖低置信度
3. **版本保留**：关键偏好保留历史版本，标注"已废弃"但不删除，便于追溯用户偏好变化
4. **人工确认**：高风险冲突（如支付方式变更）触发二次确认

**深入追问：** 历史记忆数据很大时（百万级），如何优化查询效率？

**回答要点：**

大规模记忆数据的查询优化是工程实践的关键，三个层面：

**1. 索引优化**：
- **向量索引**：用 HNSW、IVF 等近似最近邻算法，查询从 O(n) 降到 O(log n)
- **元数据索引**：user_id、task_type 等高频过滤字段建 B-tree 索引
- **组合索引**：先用元数据过滤（user_id = X），再在结果集内做向量检索，大幅缩小搜索空间

**2. 分层存储**：
- **热数据**（最近 30 天访问）：高性能向量库（Qdrant、Weaviate）
- **温数据**（30-90 天）：标准存储
- **冷数据**（90 天以上）：低成本存储（S3 + 延迟加载）
- 按访问频率自动迁移

**3. 缓存策略**：
- **查询结果缓存**：相同 query 的召回结果缓存 5-10 分钟（Redis）
- **Embedding 缓存**：用户高频查询的 query embedding 预计算
- **负缓存**：记录"查不到"的 query，避免重复无效查询

**工程实践**：字节面试真题——"历史数据很大时如何优化"，答案就是这三层组合。

**深入追问：** 多用户并发场景下，记忆系统如何做隔离和一致性保证？

**回答要点：**

多用户并发是生产环境的核心挑战，两个维度：

**1. 数据隔离**：
- **物理隔离**：每个用户的记忆存在独立的 namespace（如 `memory:user:{user_id}`）
- **权限校验**：每次读写前验证 user_id，防止跨用户访问
- **向量库 filter**：查询时强制加 `user_id` 过滤条件，从数据库层面隔离

**2. 并发一致性**：
- **写入冲突**：用乐观锁（version 字段）或分布式锁（Redis）防止同一用户的并发写覆盖
- **读写一致性**：写入后立刻读可能读到旧数据（缓存未更新），用 write-through 或短暂的 cache invalidation
- **会话隔离**：同一用户多个会话（多标签页）的记忆要共享，但状态（当前任务）要隔离

**典型故障**：
- 用户 A 的记忆被用户 B 召回 → 权限校验缺失
- 并发写入导致记忆丢失 → 缺少锁机制
- 缓存不一致导致"刚写入的记忆召回不到" → 缓存更新策略问题

**工程实践**：蚂蚁面试真题——"LangChain memory 多用户并发隔离"，核心就是 namespace + 锁 + 权限校验三层防护。

## 7. 推理优化：KV Cache 与 Prefix Cache

**面试官可能会问：** KV Cache 和应用层的 Memory 是一回事吗？

**回答要点：**

完全不是。两者容易混淆，但是完全不同层面的概念：

**KV Cache（推理层优化）**：
- 作用：缓存 Transformer 每层的 Key 和 Value 矩阵，避免 decode 阶段重复计算
- 生命周期：单次请求内，请求结束即释放
- 目的：节省 GPU 计算，加速推理
- 管理者：推理引擎（vLLM、TGI）

**应用层 Memory（业务逻辑）**：
- 作用：跨轮、跨会话保存用户信息和对话历史
- 生命周期：跨请求持久化
- 目的：维持对话连贯性和个性化
- 管理者：应用代码

**一句话区别**：KV Cache 是"算得快"，Memory 是"记得住"。

**深入追问：** Prefix Cache 是什么？为什么能大幅降低成本和延迟？

**回答要点：**

Prefix Cache（也叫 Prompt Caching）是 KV Cache 的延伸：**把多个请求共享的前缀部分的 KV Cache 缓存起来，后续请求直接复用，跳过 Prefill 计算**。

**典型场景**：
- System Prompt 固定不变（如"你是一个专业的客服助手..."）
- Few-shot 示例固定（如代码生成任务的示例代码）
- RAG 场景下长文档前缀固定

**收益**：
- **延迟降低**：跳过前缀的 Prefill 阶段，首 token 延迟降低 50%~80%
- **成本降低**：缓存命中的 token 按更低价格计费（Anthropic 是 1/10 价格，OpenAI 是 1/2 价格）
- **吞吐提升**：Prefill 阶段计算量大，跳过后服务端可以处理更多并发

**工程要点**：
1. **前缀要稳定**：动态内容（用户输入、实时检索结果）不能放在前缀里，否则破坏缓存
2. **前缀要足够长**：太短（< 1024 tokens）收益不明显，缓存管理开销反而可能拖慢
3. **应用层设计配合**：把静态内容（System Prompt、工具 schema）和动态内容（用户输入）明确分开

**深入追问：** 什么内容适合放在 Prefix Cache 里？什么不适合？

**回答要点：**

**适合放入 Prefix Cache**：
- System Prompt（角色定义、行为约束）
- 固定的工具列表和 schema
- Few-shot 示例
- 固定的输出格式说明
- 长期不变的知识库文档（如公司规章制度）

**不适合放入 Prefix Cache**：
- 用户当前输入（每次都不同）
- 实时检索结果（每次查询结果不同）
- 动态生成的上下文（如当前时间、用户状态）
- 频繁变化的约束（如"今天是周一"）

**判断标准**：这段内容在接下来的 N 次请求中会保持不变吗？如果是，放前缀；如果不是，放后缀。

## 8. 记忆与 RAG 的协同

**面试官可能会问：** 一个系统同时有记忆和 RAG，两者怎么配合？会不会冲突？

**回答要点：**

记忆和 RAG 不是二选一，而是**互补关系**，各自解决不同问题：

**记忆系统**：
- 解决"这个用户是谁、喜欢什么、历史做过什么"
- 数据来源：用户交互产生
- 典型内容：用户偏好、对话历史、任务状态

**RAG 知识库**：
- 解决"这个领域的知识是什么、规则是什么"
- 数据来源：外部文档导入
- 典型内容：产品文档、FAQ、规章制度

**协同场景**：客服系统回答"我的订单什么时候发货"
1. 记忆系统召回：用户上次咨询过订单 #12345，偏好简洁回答
2. RAG 召回：物流政策文档、发货时效说明
3. 工具调用：查询订单 #12345 的实时物流状态
4. 生成回答：结合记忆（知道是哪个订单）+ 知识（物流政策）+ 实时数据（当前状态）

**深入追问：** 记忆和 RAG 的召回结果都放进上下文，会不会互相干扰？

**回答要点：**

会，如果不做隔离。三个常见问题：

1. **信息冲突**：记忆说"用户上次选择了 A"，RAG 文档说"推荐使用 B"，模型不知道听谁的
2. **优先级不明**：记忆和知识库的信息混在一起，模型无法判断哪个更重要
3. **上下文污染**：两者召回的内容加起来太多，关键信息被稀释

**解决方案**：

1. **结构化分隔**：用 XML 标签或 Markdown 标题明确区分来源
   ```
   ## 用户信息（来自记忆）
   - 上次咨询订单：#12345
   - 偏好：简洁回答

   ## 相关知识（来自知识库）
   - 物流政策：...
   ```

2. **优先级标注**：在 Prompt 里明确"用户偏好优先于通用规则"

3. **召回数量控制**：记忆 top-3 + RAG top-3，总量不超过 Token 预算的 30%

**深入追问：** 什么时候应该只用记忆，什么时候应该只用 RAG？

**回答要点：**

**只用记忆**：
- 纯个性化任务（"继续上次的对话"）
- 用户私域信息查询（"我上次问了什么"）
- 不需要外部知识的任务

**只用 RAG**：
- 首次交互，没有历史记忆
- 纯知识查询（"公司的报销政策是什么"）
- 用户明确要求忽略历史（"重新开始"）

**两者都用**：
- 需要结合用户历史和领域知识的任务（大多数生产场景）

## 9. 项目实践：常见问题排查

**面试官可能会问：** 上下文或记忆系统出问题了，怎么排查？

**回答要点：**

按链路分层排查，**不要一上来就调 Prompt**：

**问题 1：模型"记不住"之前说过的话**

排查路径：
1. **记忆层**：这段信息有没有被写入记忆？查记忆库
2. **召回层**：写入了但有没有被召回？查召回日志，看相似度分数
3. **上下文层**：召回了但有没有放进当前轮的上下文？查完整 Prompt
4. **利用层**：放进去了但模型有没有用上？可能是 Lost in the Middle 或信息被其他内容稀释

**问题 2：记忆召回了错误或过期的信息**

排查路径：
1. **写入质量**：写入时有没有做验证？是事实还是推测？
2. **更新机制**：信息变化时有没有更新记忆？还是新旧版本并存？
3. **淘汰策略**：过期信息有没有被及时清理？
4. **召回策略**：召回时有没有考虑时间衰减？

**问题 3：上下文超长，首 token 延迟很高**

排查路径：
1. **Token 分布**：各部分占比是多少？用 tokenizer 统计
2. **冗余内容**：有没有重复的信息？工具输出是否可以压缩？
3. **Prefix Cache**：System Prompt 是否稳定？缓存命中率如何？
4. **召回数量**：RAG 和记忆召回是否过多？能否减少到 top-3？

**深入追问：** 如果让你从零设计一个记忆系统，你会怎么做？

**回答要点：**

分三步走：

**第一步：明确需求**
- 用户规模多大？单用户还是多租户？
- 记忆类型？只有对话历史还是需要长期偏好？
- 隐私要求？数据能否出域？

**第二步：选型架构**
- 短期记忆：Redis（快速读写，会话结束清空）
- 长期记忆：向量数据库（语义召回）+ 关系数据库（结构化查询）
- 状态管理：独立的 State Store（任务目标、约束、工具历史）

**第三步：设计策略**
- 写入策略：门槛控制（只写稳定信息）+ 去重（避免冗余）
- 召回策略：语义相似度 + 时间衰减 + 显式关联
- 更新策略：冲突检测 + 版本管理
- 淘汰策略：时间 + 访问频率 + 容量上限

**关键设计决策**：
1. 记忆和知识库物理隔离（权限和召回策略不同）
2. 约束独立于消息历史（防止被截断）
3. 全链路 Trace（出问题能回溯）

## 总结

记忆与上下文工程的核心是**在有限的 Token 预算下，把对的信息、以对的顺序、在对的时机送进模型**。技术选型和架构设计都要服务于这个目标。
