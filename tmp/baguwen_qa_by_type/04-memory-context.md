# 记忆与上下文工程面试知识点

这份文档按面试真实追问链路整理，目标不是堆概念，而是让你从"上下文工程是什么"一路讲到"怎么设计一个真正可用的记忆系统"，每个环节都有具体判断依据和工程经验可以展开。

---

## 1. 上下文工程与记忆系统

**面试官可能会问：** 什么是上下文工程？什么是记忆系统？两者有什么区别？

**回答要点：**

- **上下文工程（Context Engineering）** 解决的是"当前这一轮到底把哪些信息送进模型"的问题——包括信息的选择、排序、裁剪、压缩和结构化拼装，是请求级别的输入设计。
- **记忆系统（Memory System）** 解决的是"系统长期保留什么、什么时候取出来用"的问题——是跨轮、跨会话的信息资产管理，有写入、召回、更新、淘汰的完整生命周期。
- 两者的关系是：记忆系统决定"存什么"，上下文工程决定"当前轮用什么"。记住了但喂不进去，或者喂进去了但组织不好，效果都会出问题。很多看起来是"模型记不住"的故障，排查下来其实是上下文构造问题——不是没存住，而是没在当前轮把对的信息正确放进上下文。

**深入追问：** 为什么很多 AI 应用的问题，本质上不是模型能力问题，而是上下文构造问题？

**回答要点：**

- 关键信息没进上下文，模型再强也没法凭空知道。
- 无关信息太多，会稀释重点，甚至把正确证据埋掉。
- 最常见的表现是："历史里明明说过""知识库里明明有"，但模型当前轮就是没用上。这不是模型问题，是上下文组织问题。所以很多优化的第一步不是换更强的模型，而是把上下文整理好。

## 2. Token 预算与上下文构造

**面试官可能会问：** 上下文窗口越大越好吗？Token 预算应该怎么分配？上下文应该按什么顺序组织？

**回答要点：**

- 上下文窗口越大不等于效果越好。窗口更大只是理论上能塞更多内容，但塞得越多，噪声、成本和注意力分散问题也会一起放大。真正决定效果的是"放进去的内容质量"，而不是"放进去的内容数量"。
- Token 预算的分配原则是"按当前任务价值排优先级"，不是平均分：
  - 系统提示词和核心约束要留足稳定空间，这是整个系统的行为边界，不能被历史和工具结果挤掉。
  - 用户当前输入永远要有足够空间，这是本轮任务的核心。
  - 历史、检索结果、工具结果按当前任务相关性动态分配。预算不够时，先裁低相关历史和冗余工具结果，而不是先裁系统约束。
- 上下文的组织顺序通常是：系统约束 → 当前用户输入 → 最相关的历史 → 检索和工具结果。核心原则是先让模型知道边界和目标，再让它看证据。顺序很重要——模型对前面、更清晰、更结构化的内容通常更敏感。

**深入追问：** 为什么很多系统都会尽量保持 system message 稳定不变？

**回答要点：**

两个原因同时成立：行为稳定性和推理性能。

- 行为稳定性：system message 频繁变化，系统边界和行为就会波动，难以预期和调试。
- 推理性能：稳定前缀越固定，prefix cache 的命中空间越大，推理成本越低。

所以很多生产系统会把系统角色、核心约束、工具 schema 这类内容尽量前置并保持不变——这既有利于行为稳定，也有利于推理性能优化。两件事是同一个决策的结果。

## 3. 多轮对话与历史管理

**面试官可能会问：** 多轮对话时历史应该如何管理？全部保留、窗口滑动还是分层存储？

**回答要点：**

- 多轮上下文管理的核心原则是：不是"全保留"，而是"保留当前轮还相关的部分"。把所有历史都塞进去，会把窗口和注意力浪费在低价值内容上，关键约束反而被后续内容冲淡。
- 三种策略各有适用场景，真实系统通常组合使用：
  - **窗口滑动**：保留最近 N 轮，适合近历史强相关、轮次不深的场景。简单但可能丢失关键早期约束。
  - **历史摘要**：对远历史做压缩，保留关键语义。适合长时间跨度对话，但会丢细节，近历史还是要窗口直接保留。
  - **结构化状态提取**：把稳定信息单独抽取存储——用户偏好、任务状态、关键事实不应该永远混在消息历史里，混在一起最终必然串线。

**深入追问：** 多轮对话里为什么特别容易出现话题漂移、约束丢失？

**回答要点：**

根本原因是任务状态、业务状态和对话历史混在了一起。历史越长，噪声和过期信息越多；约束如果没有被显式保留，就很容易被后续内容冲淡。最常见的事故是：系统忘了前面明确说过的约束，或者把上一话题的状态带到了新话题里。

解决方向是把"约束"和"历史"分开管理——约束要显式持久化，不能只靠它在历史消息里"还没被截掉"来维持有效性。

## 4. 长上下文与 Lost in the Middle

**面试官可能会问：** 长上下文能力能解决什么问题？为什么实际效果常常低于想象？

**回答要点：**

- 长上下文解决的是"能放下"的问题，不是"自然就能用好"的问题。很多人对长上下文期待过高，把"能塞进去"误以为"模型一定能有效利用"——这是两回事。
- 实际效果低于想象的原因：内容一多，模型更容易抓不住重点；噪声、顺序和信息稀释问题仍然存在，并不会因为窗口变大而消失。长上下文只是把上限抬高，上下文治理的问题还是要靠治理解决。

**深入追问：** "Lost in the Middle" 是什么？对工程实现意味着什么？

**回答要点：**

- Lost in the Middle 指的是模型对长上下文**中间部分**的信息利用能力明显更差。前面和后面的内容更容易被注意到，中间内容即使很重要也可能被忽略。这是有论文支撑的现象（Liu et al., 2023），在实际系统里非常常见。
- 工程含义很直接：关键证据不能随便埋在中间，要前置、后置或做结构化强调。原始文档顺序是给人看的，不一定是最适合模型利用的顺序。很多"明明把材料都放进去了，但模型还是没用到最关键那段"的问题，根源就在这里。

**深入追问：** 长上下文和 RAG 是什么关系？

**回答要点：**

长上下文不是 RAG 的替代品，很多时候更像 RAG 的补充。两者解决的是不同层次的问题：RAG 解决"从大规模知识库里找准相关内容"，长上下文解决"把找到的内容一次性处理完"。当材料规模大到噪声开始压过收益时，就该先用 RAG 检索再选关键内容送入模型，而不是直接把整个知识库塞进去。

## 5. 记忆系统设计

**面试官可能会问：** Agent 的记忆系统通常应该怎么设计？长期记忆和短期记忆有什么区别？

**回答要点：**

记忆系统的核心设计原则是分层，而不是把所有信息都塞进一份对话历史里。通常分三层：

- **短期记忆（Short-term Memory）**：当前会话和近几轮的上下文连续性。生命周期随会话结束，解决"当前任务的连贯性"。
- **长期记忆（Long-term Memory）**：跨轮、跨会话的稳定信息，如用户偏好、长期事实、稳定背景。需要显式写入和管理，解决"跨时间的信息复用"。
- **结构化状态（Structured State）**：任务推进、流程状态和强约束。不应该和自由文本混在一起，用结构化格式单独存，解决"流程控制和状态追踪"。

对话历史既不适合长期存稳定事实，也不适合承载结构化任务状态，只用一份历史必然越用越乱。

**深入追问：** 哪些信息适合写入长期记忆，哪些不应该写？

**回答要点：**

- **适合写的**：稳定偏好、长期事实、跨轮有价值的用户约束、高置信度背景信息——这些信息"值得长期留"。
- **不适合写的**：临时任务状态、一次性工具结果、未验证信息——这些信息只在当前任务有效，写进长期记忆只会变成噪声。

记忆写入不是"能存就存"，写入门槛太低，记忆库很快会变脏，误召回率会显著上升，最后系统反而因为记忆变得更差。

## 6. 长期记忆的写入、召回与淘汰

**面试官可能会问：** 长期记忆的完整生命周期应该怎么管理？

**回答要点：**

长期记忆要按完整生命周期设计，不能只设计"写进去"，不设计"怎么改、怎么删、怎么过期"：

- **写入**：要有触发条件（稳定事实和长期偏好才写，临时状态不写）和去重逻辑。很多生产系统更偏异步写——记忆写入本身不需要阻塞主回答链路。
- **召回**：要有相关性控制和类型边界。"存过"不等于"该拿出来"。不同记忆类型最好分仓或分路由召回，混在一起容易误召回。
- **更新**：不是直接覆盖，而是要有版本和更新规则。新偏好可能是临时表达，更稳的做法是版本化或带时间衰减，避免系统被一次偶然输入带偏。只有用户主动明确更改时，才适合直接覆盖。
- **淘汰**：靠写入门槛、过期策略、冲突检测和定期清理，而不是无限追加。

很多系统的记忆越用越差，根本原因是只做了写入，没做好更新、冲突处理和淘汰，最后旧信息和噪声一起堆积。

**深入追问：** 召回长期记忆时如何控制误召回？记忆命中率高但回答效果差是什么原因？

**回答要点：**

误召回的根源在写入侧——写入越脏，召回误伤越大，所以治理要从写入侧开始，而不是等召回出问题再补救。

记忆命中率高但回答效果差，是一个反直觉但很常见的现象，常见原因有四类：
1. **误召回**：命中了但不相关，污染了当前上下文。
2. **记忆过期**：召回的是旧信息，已经不准确。
3. **上下文污染**：进去了但和其他内容形成噪声，模型注意力被分散。
4. **没被真正利用**：上下文里信息太多，记忆虽然进去了但没被模型用上。

定位方法是把"记忆检索结果 → 最终上下文 → 最终回答"三层拆开看，而不是只盯着最终输出。

## 7. 记忆 vs RAG 的边界

**面试官可能会问：** 记忆系统和 RAG 有什么区别？为什么很多 Agent 同时需要两套机制？

**回答要点：**

- **记忆**解决的是"这个用户/这个任务过去发生过什么"——用户画像、偏好、历史任务背景，是用户私域信息。
- **RAG**解决的是"外部知识库里有什么资料"——规章制度、产品说明、FAQ、业务知识，是公共或业务知识。

两者都可能走检索，但数据边界、权限属性和用途完全不同。混在一起最容易出串台和安全问题——用户私有记忆和公共知识的权限边界、生命周期和可信度都不一样，隔离要在存储、路由、权限过滤和召回层同时做，不能只在最后一层补。

很多 Agent 同时需要"会话记忆 + 用户记忆 + RAG 知识库"三套，因为它们解决的是三类完全不同的信息问题：少一套就会出现明显短板——只靠 RAG 没法记住用户，只靠记忆又拿不到业务知识。

## 8. KV Cache、Prefix Cache 与上下文复用

**面试官可能会问：** KV Cache 是什么？它为什么能提升推理速度？它和应用层 memory 有什么区别？

**回答要点：**

- **KV Cache** 把前面 Token 的注意力中间结果（Key-Value 矩阵）缓存下来，避免每步生成都从头重算前缀部分。它本质上是**推理层的计算优化**，不是应用层的信息管理。节省的是前缀 Token 在注意力层的重复计算，在 decode 阶段收益尤其明显——前缀越长、后续生成越多，收益越显著。
- **和应用层 memory 的本质区别**：KV Cache 解决的是算力和时延问题，生命周期在单次请求内；应用层 memory 解决的是跨轮、跨会话的信息保存和召回问题，需要显式设计和管理。两者名字里都有"缓存/记忆"的概念，但根本不是一层东西，面试里被一起问的原因是它们最终都影响"当前轮模型能看到什么"。

**深入追问：** Prefix Cache 是什么？为什么生产上大家很关心它？

**回答要点：**

- Prefix Cache 是**跨请求**复用稳定前缀的机制——多个不同请求如果共享相同的前缀（如 system prompt、工具 schema），就可以复用已计算好的 KV 结果，避免每次请求都重新计算这部分。KV Cache 更偏单次请求内部的复用，Prefix Cache 更强调跨请求的复用。
- 生产上关心它的原因很实际：很多请求前面那一大段（系统提示、工具说明、固定 schema）都很稳定，如果每次请求都重新计算这部分，成本非常高。Prefix Cache 能把这部分成本几乎降到零。

**深入追问：** 哪些内容最适合做 Prefix Cache？KV Cache 为什么最终会追到应用层设计？

**回答要点：**

- **最适合的**：system message、固定工具 schema、固定输出格式——这些内容稳定、不随会话频繁变化，前缀稳定性最高，缓存命中空间最大。
- **最不适合的**：高频变化的用户输入、实时工具结果、动态检索证据——它们天然破坏前缀稳定性，几乎不可能形成可复用前缀。

这带来一个很重要的逻辑：**推理优化反过来约束了应用层上下文设计**。推理层想复用前缀，前提是前缀别老变。前缀稳不稳，取决于系统提示、schema、工具说明和上下文组织方式——这些都是应用层的决策。所以一个好的上下文设计，要有意识地区分"固定前缀"和"动态尾部"，把稳定内容尽量前置并保持不变，这既有利于行为稳定，也有利于推理性能。

## 9. 项目深挖

**面试官可能会问：** 如果让你设计一个带长期记忆的 Agent 系统，你会怎么做？

**回答要点：**

核心原则是把"性能复用""语义记忆""上下文治理"分三层做，而不是试图用一个机制同时解决所有问题：

1. **推理层**：用 KV Cache / Prefix Cache 提升速度，应用层尽量保持前缀稳定以最大化命中率。
2. **记忆层**：近历史窗口解决当前会话连续性；长期记忆检索解决跨轮用户信息复用；结构化状态解决流程和强约束。三套机制各司其职，不混在一起。
3. **上下文治理层**：负责去重、排序、裁剪、摘要和污染控制——把真正该进模型的内容选出来，而不是把所有东西都往里塞。

这套设计最容易翻车的是长期记忆的写入和召回。一旦记脏了，后面每轮都会受影响。所以记忆系统的治理要从写入侧开始，而不是等召回出问题再补救。

**深入追问：** 如果线上出现"系统记不住用户"的问题，你会怎么排查？

**回答要点：**

按链路分三层拆开看，而不是一上来就归因给模型：

- **记忆检索结果里没有** → 问题在写入或召回层（没写进去，或者写进去了但召不回来）
- **检索结果里有，但上下文里没进去** → 问题在路由和上下文构造层
- **上下文里有，但回答没用上** → 问题在排序、位置或生成约束层（关键信息被埋在了中间）

最有效的定位方式是做**样本回放**，把"写入 → 召回 → 上下文构造 → 最终回答"四层串起来一起看，哪层断了就从哪层修。
