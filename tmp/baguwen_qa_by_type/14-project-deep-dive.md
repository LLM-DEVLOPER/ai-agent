# 项目深挖

## 1. 项目介绍应该先讲什么

**面试官可能会问：** 请介绍一下你做过的最有代表性的 AI 项目。

**回答要点：**

项目介绍最常见的错误是一上来就堆技术名词——"我们用了 LangChain、向量数据库、GPT-4，做了一个 RAG 系统"。这种开场让面试官完全不知道你在解决什么问题，也无法判断这个项目的难度和价值。

正确的顺序是：业务背景 → 核心问题 → 解决方案 → 结果。先说清楚这个项目在什么业务场景下产生，用户是谁，他们遇到了什么痛点；再说清楚核心技术挑战是什么，为什么这个问题不好解；然后才说你的解决方案；最后用具体数字说明结果。

好答案的典型特征是：上来先说"我们的业务场景是客服知识库问答，核心问题是用户问题和文档表述差异大，导致召回率低。我们针对这个问题做了查询改写和混合检索，最终把召回率从 62% 提升到了 81%"。技术名词是用来支撑叙述的，不是叙述本身。

**深入追问：** 为什么不能一上来就堆技术名词？

**回答要点：**

因为面试官首先要判断你是不是真的理解了业务，而不是只会用工具。技术名词堆得越多，越容易暴露你不知道为什么要用这些技术。先讲清楚问题，再讲方案，面试官才能跟着你的逻辑走，也才能判断你的技术选型是不是合理的。

---

## 2. 业务背景、项目来源与数据来源

**面试官可能会问：** 这个项目是怎么来的？数据从哪里来？

**回答要点：**

项目来源要讲清楚业务动机，不能只说"公司要做 AI"。好的回答要说明这个项目解决了什么业务痛点，为什么现在做，为什么是你们团队做。比如"原来客服团队每天要处理 500 个重复性问题，人工成本很高，而且回答质量不稳定，所以我们做了这个系统"，这比"公司想做智能客服"信息量大得多。

数据来源是面试官经常追问的点，因为数据质量直接决定系统上限。要说清楚：数据从哪里来（内部系统、爬取、人工标注）、数据量级（文档数、标注样本数）、数据质量如何（有没有噪声、格式是否统一）、有没有数据隐私或合规约束。如果数据来源有特殊挑战（比如文档格式混乱、标注成本高），这本身就是项目难点，值得展开讲。

**深入追问：** 如果数据质量很差，你们是怎么处理的？

**回答要点：**

数据质量问题通常要分层处理。第一层是数据清洗：去重、格式统一、过滤明显噪声。第二层是数据增强：对于标注数据不足的情况，可以用 LLM 生成合成数据，但合成数据要做质量过滤，不能直接全量使用。第三层是持续监控：上线后要监控数据漂移，知识库内容更新时要及时同步索引。数据质量问题没有一劳永逸的解法，关键是建立持续的数据治理机制。

## 3. 个人职责与 Ownership

**面试官可能会问：** 你在这个项目里具体负责什么？

**回答要点：**

"我们团队做了 X"不是好答案，因为面试官想评估的是你个人的能力，而不是团队的能力。好的回答要说清楚：你具体负责哪些模块、你做了哪些关键决策、哪些问题是你独立解决的。

区分度在于 ownership 的深度。"我负责了检索模块"是一个起点，但面试官会继续追问：检索模块的方案是你设计的还是别人设计的？你在里面做了哪些技术决策？遇到了什么问题，你是怎么解决的？能回答这些追问，才说明你真的 own 了这个模块。

如果项目是团队协作，要诚实说明分工，但同时要能清楚地说出自己的贡献边界。"我负责了检索模块的设计和实现，包括分块策略、Embedding 模型选型和混合检索方案，最终把召回率从 60% 提升到 82%"，这种回答既诚实又有信息量。真正做过核心部分的人，被追问细节时会越说越具体，而不是越说越模糊。

**深入追问：** 哪些表述最容易暴露自己没有真正做过核心部分？

**回答要点：**

只能讲整体架构、讲不清自己那部分的具体实现细节；被问到"你当时遇到什么问题"时答案是"整体比较顺利"；被问到"为什么这么选型"时答案是"团队讨论决定的"或"业界都这么做"；被问到数据细节、接口细节时答案是"这个我不太清楚，是另一个同学负责的"。这几种回答都是明显的信号，说明你没有真正 own 过这个模块。

## 4. 架构、链路与模块拆解

**面试官可能会问：** 这个系统的整体架构是什么？各个模块是怎么分工的？

**回答要点：**

讲架构不是画一张"模型接工具"的示意图，而是要说清楚数据流和决策流：用户的请求进来之后，经过哪些处理步骤，每一步的输入输出是什么，模块之间怎么交互。

好的架构讲法是按链路走：请求进来 → 意图识别/路由 → 检索/工具调用 → 生成 → 后处理 → 返回。每个节点说清楚这里在做什么、为什么要有这个节点、这个节点的关键设计决策是什么。以 RAG 系统为例：用户输入 → 查询改写（为什么要改写，改写策略是什么）→ 向量检索 + 关键词检索（为什么混合，权重怎么定）→ 重排序（用什么模型，阈值怎么设）→ 拼装 prompt → 模型生成 → 后处理 → 返回用户。

模块拆解要说清楚边界：哪些模块是你们自研的，哪些是用开源框架的，哪些是调用外部服务的。自研的模块要能说清楚为什么不用现成的；用开源框架的要能说清楚做了哪些定制化改造。

**深入追问：** 为什么很多项目深挖最后会落到接口、状态和数据结构细节？

**回答要点：**

因为这些细节是最难伪造的。架构图谁都能画，但接口的字段设计、状态机的转移条件、数据结构的选型理由，只有真正做过的人才能说清楚。面试官追问细节，不是为了刁难，而是在验证你是不是真的做过这件事。

## 5. 技术选型与方案取舍

**面试官可能会问：** 为什么选这个技术方案？为什么不用 X？

**回答要点：**

技术选型问题最差的回答是"因为业界都这么做"或"因为我们团队熟悉这个"。这两个理由都没有说明这个方案和你的业务场景的匹配关系。

好的技术选型回答要有三层：第一层是需求层，说清楚你的业务场景有什么特殊约束（数据量、时延要求、成本预算、团队规模）；第二层是方案对比层，说清楚你评估了哪些方案，每个方案的优缺点是什么；第三层是决策层，说清楚为什么在你的约束下，当前方案是最合适的。

举个例子：为什么选 Milvus 而不是 Pinecone？好的回答是"我们的数据不能出域，必须私有化部署，所以排除了 Pinecone；在自托管方案里，Milvus 的性能和我们的数据量级匹配，而且团队有运维经验，成本可控"。这比"Milvus 性能好"信息量大得多。"为什么不选别的"要从技术维度（性能、精度、功能支持）、工程维度（运维成本、团队熟悉度、生态）和业务维度（当前规模、未来扩展性、时间约束）三个角度来拆。

**深入追问：** 如果当时的选型现在回头看是错的，你会怎么讲？

**回答要点：**

这是加分题，不是减分题。能说清楚"当时为什么做了这个决定，现在看哪里判断有偏差，如果重来会怎么选"，说明你有真实的工程判断力和复盘能力。比"我们的选型完全正确"更可信，也更有价值。

## 6. 效果、评测与结果证明

**面试官可能会问：** 这个系统的效果怎么样？你们怎么评测的？

**回答要点：**

"效果还不错"等于没有回答。面试官想听的是具体数字：召回率从 X% 提升到 Y%、任务完成率 Z%、用户满意度从 A 提升到 B、人工处理量减少了 C%。没有数字的效果描述是没有说服力的。

效果量化的标准结构是：基线是什么（优化前的指标是多少，用什么方法测的）、优化后是什么（具体数字，不是"提升了很多"）、怎么证明是你的优化带来的（有没有 A/B 测试，有没有控制变量）。业务指标和技术指标都要准备：技术指标包括召回率、精度、延迟、吞吐量；业务指标包括用户满意度、任务完成率、人工介入率。

如果效果没达到预期，要能说清楚为什么没达到、你怎么分析的、后来怎么处理的。"没达到预期但我们分析了原因并调整了方向"比"效果很好"更真实，也更能体现工程能力。

**深入追问：** 如果没有 A/B 测试条件，你怎么证明优化有效？

**回答要点：**

可以用离线评测集 + 人工标注对比，或者用前后时间段的业务指标对比，但要主动说明这种对比的局限性——比如时间段对比可能受其他因素影响。能主动说出评测方法的局限性，比只说"我们测了效果很好"更有说服力。

## 7. 问题、事故与 Bad Case

**面试官可能会问：** 项目里遇到过什么印象深刻的问题或事故？

**回答要点：**

这个问题是区分"真做过"和"只做过 demo"的关键。做过真实系统的人，一定踩过坑，而且能说清楚坑在哪里、怎么发现的、怎么解决的。只做过 demo 的人，通常只能说"遇到了一些小问题，后来都解决了"。

好的 bad case 回答要有具体细节，结构是：这个问题的现象是什么（用户反馈了什么、监控报了什么警）→ 定位过程是什么（你是怎么一步步找到根因的）→ 根因是什么（是检索问题、生成问题还是数据问题）→ 怎么修复的 → 修复后有没有做预防机制。

举个例子："我们上线后发现有一类问题的召回率特别低，用户反馈说系统经常答不上来。排查后发现是因为这类问题用的是行业术语，而我们的 Embedding 模型是通用模型，对行业术语的语义理解很差。我们的解法是对这类术语做了同义词扩展，同时在这个领域的文档上做了 Fine-tuning，最终这类问题的召回率从 35% 提升到 71%。"如果一个项目从来没有问题，反而显得不可信——真实的线上系统一定会有问题，没有问题说明要么没上线，要么没有监控。

**深入追问：** 遇到线上问题时，你的定位思路是什么？

**回答要点：**

先缩小范围：是全量用户还是部分用户，是特定时间段还是持续发生，是特定输入类型还是随机出现。然后按链路逐层排查：是输入问题、检索问题、模型问题还是后处理问题。定位过程本身就是能力的体现，不要只说结论，要说你是怎么一步步找到根因的。

## 8. 复盘、Trade-off 与优化方向

**面试官可能会问：** 如果重做这个项目，你会做什么不同的事？这个系统还有哪些优化空间？

**回答要点：**

复盘问题考的是你的反思能力和对系统的深度理解。差答案是"我会把所有地方都优化一遍，把模型换成更好的，把检索也优化，把评测也做得更完善"——没有优先级，没有取舍，说明没有真正思考过。好答案是说出一个最重要的点，说清楚为什么，说清楚当时的约束和现在的判断。比如"我最想改的是评测体系。当时我们上线前的评测集太小，只有 200 条，而且都是内部标注的，和真实用户分布差异很大，导致线上效果和离线评测差距明显。如果重来，我会在上线前先做一轮用户行为数据采样，用真实分布构建评测集"。

Trade-off 是复盘里最有价值的部分。每个技术决策都有代价，好的工程师能说清楚自己的决策换来了什么、牺牲了什么。成本、时延、效果之间的取舍要说清楚当时的约束条件——"我们当时选了效果更好但延迟更高的方案，因为业务对延迟不敏感，但对准确率要求高"，这才是有判断力的回答。

优化方向要具体，要说清楚哪个方向的优化收益最大、为什么现在还没做（资源约束、优先级）、如果做了预期能带来什么提升。这种回答既展示了你对系统的深度理解，也展示了你的工程优先级判断。

**深入追问：** 你们团队在这个项目上最大的教训是什么？

**回答要点：**

好的回答是说出一个真实的教训，而不是把教训包装成"我们学到了很多"。比如"最大的教训是我们太晚才建立评测体系。前三个月我们一直在凭感觉迭代，直到上线后用户反馈差才意识到我们的离线评测完全不够。如果重来，我会在第一个月就建好 golden set 和自动化评测流程"。这种回答比"我们学到了要重视评测"更有说服力。
